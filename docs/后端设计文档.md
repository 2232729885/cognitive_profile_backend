# 后端设计文档
# 课题四认知画像系统 —— 架构总览（给算法组）

> 这份文档讲的是"整个系统怎么设计的"，帮助理解 T1-T6 接口在整个系统里处于什么位置、数据从进来到最后变成画像经过了哪些环节。具体每个接口怎么调，看对应的《T1标注接口规约》《T2抽取接口规约》《T3实体消歧接口规约》，这几份文档只讲全貌和背景，不重复贴接口细节。

## 一、系统在做什么

这是课题四的后端系统：从社交媒体（Reddit等平台）持续接入内容、账号、媒体等原始数据，通过 T1-T4 自动化标注/抽取/融合/索引，构建一个包含人物、组织、事件、地点及其关系的知识图谱，并在此基础上生成人物画像（T5）、识别重点目标（T6）。

系统由课题三负责把各平台数据采集、标准化后通过 Kafka 推送过来，课题四（本系统）负责后续的语义理解、实体抽取、知识图谱构建和画像生成，实际调用 T1-T6 语义能力的是算法组各自实现的服务。

## 三、数据接入层

### 3.1 六种数据类型

课题三通过 Kafka 推送六类数据，每类对应一个独立的 topic 和消费者：

| record_type | Kafka Topic | 消费者 | 是否进 T1-T4 流水线 |
|---|---|---|---|
| `social_content` | `kt3.social_content` | `SocialContentConsumer` | 是 |
| `news_article` | `kt3.news_article` | `NewsArticleConsumer` | 是（处理路径跟social_content相同） |
| `social_account` | `kt3.social_account` | `SocialAccountConsumer` | 否，只做标准化映射 |
| `account_relation` | `kt3.account_relation` | `AccountRelationConsumer` | 否 |
| `media_asset` | `kt3.media_asset` | `MediaAssetConsumer` | 否，T4异步补图像向量 |
| `collection_task` | `kt3.collection_task` | `CollectionTaskConsumer` | 否，仅用于采集覆盖范围核查 |

（对应枚举见 `common.constant.RecordType`）

### 3.2 消息公共信封

所有消息共享同一套外层结构（字段名以实际接收到的真实数据为准，历次真实数据联调都验证过）：

```json
{
  "schema_version": "kt3_to_kt4_v2",
  "record_type": "social_content",
  "raw_record_id": "全局唯一ID，用于去重",
  "raw_payload_hash": "原始载荷哈希，用于去重兜底",
  "platform": "x / reddit / telegram ...",
  "collected_at": "采集时间",
  "data": { "...标准化后的结构化字段..." },
  "raw_payload": { "...平台原始返回，未必每个字段都标准化，酌情兜底取值..." }
}
```

**`schema_version` 目前不做强校验**（早期版本卡死过一次真实数据导入，因为版本号从 v1 升到 v2 后被硬编码校验拒收，这次改成不限制具体版本号）。

### 3.3 摄入公共步骤（`IngestionMessageSupport` 封装了这几步的通用逻辑）

1. 解析消息，取出 `sourceRecordId`（`raw_record_id`，取不到兜底取 `source_record_id`）、`payloadHash`（`raw_payload_hash`，取不到就对整条消息算sha256）
2. **去重检查**（`DeduplicationChecker`）：`sourceRecordId` 或 `payloadHash` 命中已有 `raw_records` 记录，直接跳过。**`collection_task` 类型例外**——这类消息的 `raw_payload` 经常是空对象，哈希没有区分度，只按 `sourceRecordId` 判重（`isDuplicateBySourceRecordId`）
3. 写入 `raw_records`（原始记录审计表，记录 `pipeline_status`：`RECEIVED`→`NORMALIZED`→（如果走流水线）`T1_DONE`→`T2_DONE`→`T4_INDEXED`）
4. 调用对应的 `xxxNormalizer` 把 `data`/`raw_payload` 映射成实体对象
5. 落库；`social_content`/`news_article` 额外创建 `pipeline_tasks` 记录，触发后续 T1→T2→T4 流水线（`PipelineExecutor.submitAfterCommit()`，事务提交后再触发，避免脏读）

### 3.4 账号/内容互动数据的自动补建

- 内容摄入时（`SocialContentConsumer`）会用内容里带的作者信息，给作者账号建一条"最小信息"记录（`upsertAuthorAccount`），即使这个作者没有单独的 `social_account` 快照消息也能有账号节点可关联。`upsertByPlatformAndUserId` 用 `COALESCE(新值, 旧值)` 逐字段合并，不会用最小信息覆盖掉已经有的完整资料。
- 媒体资产的关联：`social_content` 消息里的 `data.media_asset_ids`（字符串ID数组）会存进 `media_contents.source_media_asset_ids`，`media_asset` 消息到达时按这个反查关联（`MediaAssetConsumer`），不管两条消息谁先到都能关联上。

### 3.5 文件批量导入（前端"文件导入"功能）

前端支持直接上传 zip/jsonl/json 文件（不用等课题三通过Kafka推送），上传后文件存入 MinIO，`BatchImportJob`（每分钟轮询一次）解析文件内容、按 `record_type` 转发到对应 Kafka topic，走跟正常消息完全一样的消费者处理逻辑。zip 包里如果带了媒体文件（`media/` 目录），会重新上传到自己的 MinIO，并把 `minio_bucket`/`minio_key` 补进对应 `media_asset` 消息里再转发。

## 四、T1-T4 内容处理流水线

### 4.1 整体顺序

```
社交内容/新闻入库 → T1标注(annotate) → T2抽取(extract_entities) 
                                        │
                                        ├─ 后端候选召回（ES+Milvus）
                                        ├─ T3判断(resolve_batch)
                                        └─ 按判断结果 合并/新建/转审核
                → 关系落图 → T4向量化索引
```

对应代码：`T1AnnotationStep` → `T2ExtractionStep`（内部通过 `EntityResolutionService` 完成候选召回+T3判断+落库，这一步骨子里就是 T2+T3 协同，没有单独的"T3Step"类）→ `T4IndexingStep`。三步之间通过 `PipelineTask` 状态机串联（`pipeline_tasks.t1_status`/`t2_status`/`t4_status`）。

### 4.2 T1：自动标注

- 一个统一接口 `annotate`，支持纯文本/纯图片视频/图文混合，同时判断 15 个内容级标注维度 + AIGC 检测。
- 另外两个独立接口：`annotate_account`（账号类别，账号摄入时同步调用）、`annotate_event_heat`（事件热度，图谱聚合数据后异步批量调用，见4.5节）。
- 详见《T1标注接口规约》《T1_annotation_v0.5_README.md》。
- 落库方式：整段 JSON 存 `media_contents.t1_annotation`（PG）和 `MediaContent.t1Annotation`（Neo4j 属性），不逐字段拆解——这样以后标注维度再调整，不用改一堆列。

### 4.3 T2/T3：实体抽取与消歧

- T2 输出 mention 级的实体和关系（`extract_entities`），不判断这个实体是不是图谱里已经存在的。
- 后端候选召回（`EntityCandidateRetrievalService`，ES名称匹配+Milvus向量召回）为每个 mention 找可能匹配的已有实体。
- T3 判断每个 mention 该 `MERGE`（合并进已有实体）/`REVIEW`（转人工审核）/`CREATE`（新建），阈值默认 0.9/0.6，通过请求体动态传入，不写死在算法侧。
- `EntityResolutionService` 是这套"候选召回+T3判断+落库"逻辑的唯一实现，**内容抽取场景和账号身份识别场景（见4.4节）共用同一份**，不需要为账号场景单独实现。
- 关系落图前会校验 `predicate` 必须在16个关系词表内（`docs/关系词表与头尾实体类型说明.md`），不在词表内的会被过滤掉。
- 每条内容成功解析出来的实体，后端会自动补一条内容指向该实体的关系（`event`类型走`DESCRIBES`，其余走`MENTIONS`），不需要 T2 在 `relations[]` 里额外输出。
- 详见《T2抽取接口规约》《T3实体消歧接口规约》《Kg融合接口规约》。

### 4.4 账号身份识别（独立于内容流水线）

账号摄入之后，`SocialAccountIdentityJob`（定时任务，每15分钟一批，最多200个）按账号类别把账号本身也解析成图谱实体：

- 个人类账号（`ordinary_user`/`political_actor`/`academic_or_expert`/`influencer_kol`）→ 用 `displayName` 当 mention，走候选召回+T3判断，匹配/新建 `Person`，写 `HAS_ACCOUNT` 关系
- 机构类账号（`news_media`/`government_agency`等10类）→ 同上，匹配/新建 `Organization`
- 跳过类（`anonymous_account`/`suspected_bot_or_automated`/`unknown`/`other`）→ 不处理

这条路径跳过 T2 的实体抽取环节（账号资料本身已知是谁，不需要"抽取"），直接从候选召回开始。

### 4.5 事件热度（独立接口，异步批量）

单条内容的热度感知信息量有限，真正的事件热度需要图谱聚合了足够多关联内容才能判断。`EventHeatJob`（定时任务）扫描已去重的 `Event` 实体，通过 Neo4j 一跳查询取出关联的人物/组织/地点/账号和内容（内容数量多时按互动量取Top20采样），连同后端算好的聚合统计（关联内容总数、总互动量、时间跨度等）一起发给 T1 的 `annotate_event_heat` 接口。

### 4.6 T4：向量化与索引

- 生成文本/图像 Embedding 写入 Milvus，全文索引同步到 Elasticsearch；文本内容同时写原文向量 `media_content_embeddings` 和英文 pivot translation 向量 `media_content_pivot_embeddings`。
- 图片/视频资产的向量化和T1标注是分开异步进行的（`ImageEmbeddingJob`），因为媒体经常比文本晚到，架构上专门做了"文本先标注、媒体后补标注"的兜底；OCR/ASR/caption 会额外写英文 pivot translation 向量到 `media_asset_pivot_embeddings`。

## 五、存储设计

| 存储 | 用途 |
|---|---|
| **PostgreSQL** | 主库，所有结构化数据：内容/账号/媒体资产、`persons`/`organizations`/`events`（图谱实体的PG侧存储，`locations`类型只存Neo4j不落PG）、`entity_fusion_records`（融合审核记录）、`pipeline_tasks`/`raw_records`（流水线状态追踪）、`sub_agent_registry`（T1-T6接口配置） |
| **Neo4j** | 知识图谱本体：`Person`/`Organization`/`Event`/`Location`/`SocialAccount`/`MediaContent` 节点 + 16个关系类型 |
| **Milvus** | 向量检索：`entity_embeddings`（实体名称向量）、`media_content_embeddings`（内容原文向量）、`media_content_pivot_embeddings`（内容英文 pivot translation 向量）、`media_asset_embeddings`（图片/视频及OCR/ASR/caption原文向量）、`media_asset_pivot_embeddings`（OCR/ASR/caption英文 pivot translation 向量） |
| **Elasticsearch** | 全文/模糊检索：`entities_index`（实体名称+别名）、内容全文索引 |
| **MinIO** | 媒体文件对象存储 + 批量导入的原始文件暂存 |
| **Redis** | 缓存层 |

**PG 和 Neo4j 的一致性策略**：PG 允许无约束快速插入（新建实体先标 `dedup_status='pending'`），靠 `EntityDeduplicationJob` 定时按 `canonical_name` 归并；Neo4j 用确定性 UUID（`type+canonicalName` 算hash）+ `MERGE` 语义，实时去重。这是两边"最终一致"但节奏不同的设计，不是bug——PG 短期内可能有同名重复行，Neo4j 应该始终只有一个节点。

## 六、定时批处理任务一览

| 任务 | 频率 | 作用 |
|---|---|---|
| `EntityDeduplicationJob` | 60分钟 | PG 里同 `canonical_name` 的 `pending` 实体归并成一个 `canonical` |
| `SocialAccountIdentityJob` | 15分钟/批200 | 账号→Person/Organization 身份识别（见4.4节） |
| `EventHeatJob` | 30分钟 | 事件热度计算（见4.5节） |
| `ImageEmbeddingJob` | 10分钟 | 图片补标注 + 向量化，兼做内容-资产关联的兜底回填 |
| `ContentPropagationBackfillJob` | 10分钟 | 内容间传播关系（回复/转发）回填，应对目标内容晚到的情况 |
| `AccountRelationBackfillJob` | 1分钟 | 账号间关系（关注/成员等）实时同步 Neo4j |
| `BatchImportJob` | 1分钟 | 前端文件导入任务处理（见3.5节） |
| `PersonProfileGenerationJob` | 每日凌晨2点 | 挑选高价值人物生成画像（T5） |

## 七、Mock / 真实切换机制

`sub_agent_registry` 表逐个 agent（T1-T6）配置 `active_url_type`（`mock`/`real`）、`mock_url`、`base_url`，`AgentProxyClient` 统一按这个配置路由请求，请求URL = 对应的url字段 + `/` + action。

现在默认全部是 `mock`（`MockAgentController` 提供固定示例数据，用于验证数据链路是否打通），切到 `real` 之后请求会打到 `LlmAgentController`（真实大模型驱动的实现，需要背后配置好实际可用的模型服务）。**Mock 模式下的实体/标注内容是写死的示例，不反映输入内容本身**，验证的是"数据能不能正确流过整条链路"，不是"算法准不准"。

## 八、T5/T6 与分析层（简述，非本次重点）

- **T5 画像生成**：`PersonProfileGenerationJob` 挑选候选人物（`canonical` 状态、7天内未生成过画像、按重要度排序），调用 T5 接口生成画像，写入 `person_profiles`。
- **T6 目标识别**：`IdentificationTaskService`，基于叙事ID或账号列表输入。
- **对话式分析助手**：`CoordinatorAgentService`，基于 Spring AI 的工具调用（`GenerateProfileTool`/`IdentifyTargetsTool`/`QueryGraphTool`/`SearchContentTool`），支持前端以自然语言提问、SSE流式返回，让用户不用直接写 Cypher/SQL 也能查图谱、查画像、查内容。这一层调用的还是上面这些既有能力，不是新的语义理解环节。

这两块目前不是本次跟算法组对接的重点，仅供了解系统全貌。
