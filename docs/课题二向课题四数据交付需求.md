# 课题二向课题四数据交付需求

版本：v2.0  
日期：2026-06-25  
用途：用于明确课题二向课题四提供的数据范围、采集平台、字段格式与交付方式。

## 一、总体原则

课题四需要课题二提供的是可追溯、可重跑、可构图、可检索的跨平台原始事实数据，而不是已经完成认知分析后的画像数据、知识图谱数据或重点目标识别结果。

课题二侧主要负责数据采集、原始字段保留、媒体文件落地、采集任务元数据记录；课题四侧负责自动标注、信息抽取、实体融合、跨模态检索、画像补全和重点目标识别。

因此，课题二交付给课题四的数据应满足以下原则：

1. 保留平台原始 ID、原始 URL、发布时间、采集时间和原始载荷。
2. 保留内容、账号、账号关系、媒体、新闻报道、采集任务六类数据。
3. 不提前固化课题四算法产出，例如政治倾向、复杂情感、重点目标类型、画像维度等。
4. 所有数据必须可追溯到原始平台页面、原始采集任务和原始字段。
5. 所有时间统一使用 UTC ISO-8601 格式，例如 `2026-06-24T03:25:10Z`。
6. 所有文本统一使用 UTF-8 编码。
7. 媒体文件应提供可访问的对象存储路径或文件交付路径。

## 二、建议采集平台范围

### 2.1 最低必采平台

| 数据类型 | 建议平台 | 课题四用途 |
|---|---|---|
| 社交媒体 | X/Twitter | 跨平台账号识别、叙事传播分析、账号关系网络构建、重点目标识别 |
| 视频平台 | YouTube | 视频叙事分析、评论传播分析、频道画像、跨模态检索 |
| 新闻报道 | 主流新闻网站、RSS、新闻聚合源 | 事件背景、事实核验、叙事源头、实体背景补充 |

### 2.2 建议扩展平台

| 数据类型 | 建议平台 | 说明 |
|---|---|---|
| 短视频平台 | TikTok | 支撑视频、图文混合内容分析和传播路径分析 |
| 社交网络 | Facebook public page/group | 支撑页面、群组、评论、分享链分析 |
| 图片/短视频平台 | Instagram/Reels | 支撑图文混合内容、账号画像和视觉叙事分析 |
| 公开频道/群组 | Telegram public channel/group | 支撑频道传播、转发链、组织化传播分析 |
| 社区论坛 | Reddit、4chan、PTT、Dcard 等 | 支撑社群观点、话题发酵、评论树和群体行为分析 |
| 本地及小语种媒体 | 越南语、菲律宾语、日语、韩语等新闻和社交平台 | 支撑任务书中的多语种和小语种融合指标 |

正式交付时，建议至少覆盖 X/Twitter、YouTube、新闻报道三类来源；根据重点国别、重点地区和任务需要，扩展 TikTok、Facebook、Telegram、Reddit/论坛及本地新闻站点。

## 三、课题二需提供的数据类型

课题二向课题四交付的数据建议统一划分为六类：

| record_type | 数据类型 | 说明 |
|---|---|---|
| `social_content` | 社交媒体内容 | 主贴、频道消息、评论、回复、引用转发、纯转发、图文/视频内容等平台内容对象 |
| `social_account` | 账号/频道/页面/群组快照 | 用户、频道、页面、群组、社区、论坛版块等平台主体的公开资料和统计指标 |
| `account_relation` | 账号关系 | 关注、粉丝、订阅、频道成员、群组成员、管理员、所有者等账号间关系 |
| `media_asset` | 多媒体附件 | 图片、视频、音频、缩略图、OCR、ASR、媒体哈希等 |
| `news_article` | 新闻报道 | 新闻标题、正文、来源、作者、发布时间、语种、地区等 |
| `collection_task` | 采集任务元数据 | 采集关键词、种子账号、时间范围、语种、地区、采集方式等 |

本版本不再将 `interaction` 作为必交付 `record_type`。评论、回复、引用转发、纯转发、频道转发等只要在平台上形成了可定位的内容对象，统一作为 `social_content` 采集；关注、粉丝、订阅、成员等账号间关系统一作为 `account_relation` 采集；点赞、reaction、播放、收藏等弱互动优先作为统计指标进入 `metrics`，不要求逐条明细交付。

## 四、统一外层数据格式

无论是社交内容、账号、账号关系、媒体附件还是新闻报道，建议统一采用以下外层格式：

```json
{
  "schema_version": "kt3_to_kt4_v2",
  "record_type": "social_content",
  "platform": "x",
  "raw_record_id": "x_post_123456",
  "crawl_task_id": "task_20260624_001",
  "collected_at": "2026-06-24T03:25:10Z",
  "source_url": "https://x.com/example/status/123456",
  "language": "en",
  "data": {},
  "raw_payload": {},
  "raw_payload_hash": "sha256..."
}
```

字段说明：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `schema_version` | 是 | 数据交付格式版本，建议固定为 `kt3_to_kt4_v2` |
| `record_type` | 是 | 数据类型，取值见第三节 |
| `platform` | 是 | 平台标识，例如 `x`、`youtube`、`tiktok`、`telegram`、`news` |
| `raw_record_id` | 是 | 课题二生成的全局唯一原始记录 ID |
| `crawl_task_id` | 是 | 采集任务 ID，用于追溯采集批次和采集条件 |
| `collected_at` | 是 | 数据采集时间，UTC ISO-8601 格式 |
| `source_url` | 建议 | 原始内容、账号、新闻或媒体的 URL |
| `language` | 建议 | 语言代码，例如 `en`、`zh`、`ja`、`ko`、`vi`、`tl`、`unknown` |
| `data` | 是 | 课题二与课题四约定的标准字段 |
| `raw_payload` | 是 | 平台原始字段完整保留，不做破坏性改写 |
| `raw_payload_hash` | 是 | 原始载荷哈希，用于去重、追溯和失败重跑 |

说明：统一外层中的 `crawl_task_id` 仅作为采集任务引用，不承载完整采集任务说明。完整采集条件、关键词、种子账号、时间窗口、目标语种和采集器版本等信息，应在 `collection_task` 中单独交付。

## 五、各类数据标准字段

### 5.1 社交媒体内容：social_content

适用范围：主贴、频道消息、评论、回复、引用转发、纯转发、视频平台内容、图文平台内容、论坛楼层等。

`social_content` 关注“平台内容对象本身”。只要平台上形成了一条可定位的内容记录，具备内容 ID、作者、时间、父内容或转发关系，就应作为 `social_content` 采集。即使文本为空、只有表情、只有句号、只有转发关系，也可以作为 `social_content` 采集。

`content_type` 表达内容在平台中的形态或行为形态，不表达媒体模态。视频、图片、音频等媒体信息通过 `media_asset_ids`、`media` 或 `media_types` 表达，不建议使用 `video`、`image` 作为 `content_type`。

```json
{
  "platform_content_id": "123456",
  "author_platform_user_id": "user_001",
  "content_type": "post",
  "text": "原文内容",
  "published_at": "2026-06-24T03:21:00Z",
  "parent_content_id": null,
  "root_content_id": "123456",
  "repost_of_content_id": null,
  "quoted_content_id": null,
  "hashtags": ["tag1", "tag2"],
  "mentions": ["user_002"],
  "external_urls": ["https://example.com"],
  "metrics": {
    "like_count": 0,
    "comment_count": 0,
    "share_count": 0,
    "repost_count": 0,
    "quote_count": 0,
    "view_count": 0,
    "reaction_count": 0
  },
  "media_asset_ids": ["media_001"],
  "media_types": ["video"],
  "media": [
    {
      "asset_id": "media_001",
      "media_type": "video",
      "storage_uri": "s3://kt3-media/2026/06/24/media_001.mp4",
      "sha256": "hash..."
    }
  ]
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `platform_content_id` | 是 | 平台原始内容 ID |
| `author_platform_user_id` | 是 | 作者、频道、页面、群组或发帖主体的原始 ID |
| `content_type` | 是 | 内容形态，建议取值见下表 |
| `text` | 是 | 正文、标题、视频说明、图片配文等文本内容；无文本时为空字符串 |
| `published_at` | 建议 | 内容发布时间 |
| `parent_content_id` | 建议 | 评论、回复、楼中楼等直接上级内容 ID |
| `root_content_id` | 建议 | 会话、评论树或传播链根内容 ID |
| `repost_of_content_id` | 建议 | 被纯转发内容 ID |
| `quoted_content_id` | 建议 | 被引用内容 ID |
| `hashtags` | 建议 | 话题标签 |
| `mentions` | 建议 | 提及的账号 ID 或 handle |
| `external_urls` | 建议 | 正文中的外部链接 |
| `metrics` | 建议 | 点赞、评论、分享、浏览、reaction 等平台统计指标 |
| `media_asset_ids` | 建议 | 关联的媒体附件 ID |
| `media_types` | 建议 | 关联媒体模态，例如 `image`、`video`、`audio` |
| `media` | 可选 | 媒体附件内嵌交付方式；如使用该方式，应至少包含 `asset_id`、`media_type`、`storage_uri`、`sha256` |

建议 `content_type` 取值：

| content_type | 含义 |
|---|---|
| `post` | 普通主贴、平台主内容、视频平台视频条目、图文平台图文条目 |
| `channel_post` | 频道消息或频道发布内容，如 Telegram channel post |
| `comment` | 对主贴、视频、新闻、频道消息等发表评论 |
| `reply` | 对某条评论或回复继续回应 |
| `quote` | 引用转发且有独立内容对象 |
| `repost` | 纯转发、转贴、无新增正文也可保留为空文本内容 |
| `forward` | Telegram 等平台频道转发或消息转发 |
| `forum_thread` | 论坛主题帖 |
| `forum_reply` | 论坛回帖或楼层 |

评论、回复、转发关系恢复规则：

| 场景 | `content_type` | 关键字段 |
|---|---|---|
| 对主贴、视频、新闻发表评论 | `comment` | `parent_content_id` 指向主内容，`root_content_id` 指向主内容 |
| 对某条评论继续回复 | `reply` | `parent_content_id` 指向被回复评论，`root_content_id` 指向主内容 |
| 引用转发并新增文本 | `quote` | `quoted_content_id` 指向被引用内容 |
| 纯转发无新增文本 | `repost` | `text` 可为空，`repost_of_content_id` 指向被转发内容 |
| 频道转发 | `forward` | `repost_of_content_id` 或 `parent_content_id` 指向原消息 |

课题四可从 `author_platform_user_id`、`parent_content_id`、`root_content_id`、`repost_of_content_id`、`quoted_content_id` 推导评论树、回复链、引用链和传播链，因此不再要求课题二额外交付 `interaction` 明细。

### 5.2 账号/频道/页面/群组快照：social_account

适用范围：社交媒体用户、YouTube 频道、Facebook 页面、Telegram 频道/群组、论坛用户、论坛版块、新闻源账号等。

`social_account` 保存平台主体自身资料和统计快照，不保存完整关注列表、粉丝列表、成员列表。关注、粉丝、订阅、成员、管理员、所有者等账号间关系应通过 `account_relation` 单独交付。

```json
{
  "platform_user_id": "user_001",
  "account_entity_type": "channel",
  "platform_native_type": "youtube_channel",
  "handle": "account_name",
  "display_name": "显示名",
  "profile_url": "https://example.com/profile/user_001",
  "bio": "账号简介",
  "avatar_url": "https://example.com/avatar.jpg",
  "verified": false,
  "verified_type": "none",
  "self_declared_location": "用户自填位置",
  "created_at": "2020-01-01T00:00:00Z",
  "snapshot_at": "2026-06-24T03:25:10Z",
  "metrics": {
    "followers_count": 0,
    "following_count": 0,
    "subscriber_count": 0,
    "member_count": 0,
    "post_count": 0,
    "view_count": 0
  }
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `platform_user_id` | 是 | 平台原始用户、频道、页面、群组或版块 ID |
| `account_entity_type` | 建议 | 统一主体类型，建议取值见下表 |
| `platform_native_type` | 建议 | 平台原生类型，如 `youtube_channel`、`telegram_group`、`facebook_page` |
| `handle` | 建议 | 平台用户名或账号名 |
| `display_name` | 建议 | 显示名 |
| `profile_url` | 建议 | 主页 URL |
| `bio` | 建议 | 公开简介 |
| `avatar_url` | 建议 | 头像 URL |
| `verified` | 建议 | 是否认证 |
| `verified_type` | 建议 | `none`、`blue`、`org`、`government`、`media`、`unknown` 等 |
| `self_declared_location` | 建议 | 用户自填位置 |
| `created_at` | 可选 | 平台可见的账号、频道、群组创建时间 |
| `snapshot_at` | 是 | 账号快照采集时间 |
| `metrics` | 建议 | 粉丝数、关注数、订阅数、成员数、发文数等平台指标 |

建议 `account_entity_type` 取值：

| account_entity_type | 含义 |
|---|---|
| `user` | 普通用户账号 |
| `channel` | YouTube、Telegram 等频道 |
| `page` | Facebook page、机构主页等 |
| `group` | 群组 |
| `community` | 社区 |
| `forum_board` | 论坛版块 |
| `news_source` | 新闻源或媒体账号 |
| `unknown` | 无法判断类型 |

频道创建者、群主、管理员、成员等信息如平台可见，不建议塞入 `social_account` 大字段中，建议通过 `account_relation` 表达。

### 5.3 账号关系：account_relation

适用范围：关注列表、粉丝列表、订阅关系、频道成员、群组成员、频道创建者、群主、管理员、共同关注等账号间关系。

```json
{
  "source_platform_user_id": "user_a",
  "target_platform_user_id": "channel_001",
  "relation_type": "subscribe",
  "observed_at": "2026-06-24T03:25:10Z",
  "source": "subscriber_list",
  "confidence": 1.0
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `source_platform_user_id` | 是 | 关系源账号 ID，例如关注者、订阅者、成员账号、管理员账号 |
| `target_platform_user_id` | 是 | 关系目标账号 ID，例如被关注者、频道、群组、页面 |
| `relation_type` | 是 | 关系类型，建议取值见下表 |
| `observed_at` | 是 | 关系被采集观测到的时间 |
| `occurred_at` | 可选 | 关系真实发生时间，如平台可见 |
| `source` | 建议 | 关系来源，例如 `following_list`、`follower_list`、`subscriber_list`、`group_member_list`、`channel_profile` |
| `confidence` | 建议 | 关系置信度，平台原生列表通常为 1.0 |

建议 `relation_type` 取值：

| relation_type | 含义 |
|---|---|
| `following` | source 关注 target |
| `follower` | source 是 target 的粉丝；若可统一方向，优先转为 `following` |
| `subscribe` | source 订阅 target |
| `member_of` | source 是 target 群组/频道/社区成员 |
| `owner_of` | source 是 target 所有者或群主 |
| `creator_of` | source 是 target 创建者 |
| `admin_of` | source 是 target 管理员 |
| `moderator_of` | source 是 target 版主或协管 |
| `common_following` | 共同关注关系或共同关注统计 |

说明：`following` 和 `follower` 是同一关系的两个方向视角。建议课题二统一按“source 关注 target”交付 `following`；若只能从粉丝列表采集，则可交付 `follower` 并说明来源。

对于粉丝量极大的账号，不强制全量交付所有粉丝明细，可采用以下方式：

1. 重点账号全量交付关注列表。
2. 粉丝列表按分页增量交付。
3. 对超大粉丝列表提供抽样数据。
4. 至少提供高频互动账号、共同关注统计或核心粉丝样本。

### 5.4 多媒体附件：media_asset

适用范围：图片、视频、音频、缩略图、帖子附图、视频封面、新闻配图等。

`media_asset` 推荐单独交付，因为媒体文件可能被多个内容复用，且 OCR、ASR、抽帧、图像向量、视频向量、哈希去重等处理链路通常围绕媒体文件本身展开。为降低课题二接入成本，也允许短期将媒体信息内嵌在 `social_content.data.media` 数组中；课题四入库时再拆分为独立媒体资产。

```json
{
  "asset_id": "media_001",
  "media_type": "video",
  "source_url": "https://example.com/media.mp4",
  "storage_uri": "s3://kt3-media/2026/06/24/media_001.mp4",
  "mime_type": "video/mp4",
  "sha256": "hash...",
  "width": 1920,
  "height": 1080,
  "duration_seconds": 60,
  "thumbnail_uri": "s3://kt3-media/2026/06/24/media_001_thumb.jpg",
  "ocr_text": "图片或视频帧中的文字，可选",
  "asr_text": "音视频语音转写，可选"
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `asset_id` | 是 | 媒体附件 ID |
| `media_type` | 是 | `image`、`video`、`audio`、`thumbnail` |
| `source_url` | 建议 | 原始媒体 URL |
| `storage_uri` | 是 | 媒体文件在对象存储或文件系统中的路径 |
| `mime_type` | 建议 | MIME 类型 |
| `sha256` | 是 | 媒体文件哈希，用于去重和追溯 |
| `width` | 可选 | 图片或视频宽度 |
| `height` | 可选 | 图片或视频高度 |
| `duration_seconds` | 可选 | 视频或音频时长 |
| `thumbnail_uri` | 可选 | 缩略图路径 |
| `ocr_text` | 可选 | OCR 识别文本 |
| `asr_text` | 可选 | ASR 转写文本 |

两种交付方式均可接受：

| 方式 | 说明 | 推荐程度 |
|---|---|---|
| 独立 `media_asset` 记录 | 每个媒体文件一条记录，`social_content` 通过 `media_asset_ids` 引用 | 优先推荐 |
| 内嵌 `social_content.data.media[]` | 媒体元数据随内容一起交付，后续由课题四拆分 | 短期兼容 |

无论采用哪种方式，每个媒体对象至少应提供 `asset_id`、`media_type`、`storage_uri`、`sha256`。

### 5.5 新闻报道：news_article

适用范围：新闻网站、RSS、新闻聚合源、本地媒体、小语种媒体等。

```json
{
  "source_name": "BBC",
  "domain": "bbc.com",
  "url": "https://example.com/news/001",
  "title": "新闻标题",
  "body_text": "新闻正文",
  "author": "作者",
  "published_at": "2026-06-24T01:00:00Z",
  "updated_at": "2026-06-24T02:00:00Z",
  "language": "en",
  "country_or_region": "US",
  "section": "world",
  "tags": ["politics", "security"],
  "media_asset_ids": ["media_001"]
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `source_name` | 是 | 新闻来源名称 |
| `domain` | 建议 | 新闻站点域名 |
| `url` | 是 | 新闻 URL |
| `title` | 是 | 标题 |
| `body_text` | 是 | 正文 |
| `author` | 可选 | 作者 |
| `published_at` | 建议 | 发布时间 |
| `updated_at` | 可选 | 更新时间 |
| `language` | 建议 | 语言代码 |
| `country_or_region` | 可选 | 媒体或报道相关地区 |
| `section` | 可选 | 栏目，例如 `world`、`politics` |
| `tags` | 可选 | 新闻标签 |
| `media_asset_ids` | 可选 | 关联媒体附件 ID |

### 5.6 采集任务元数据：collection_task

适用范围：每个采集任务、采集批次、关键词任务、账号任务、URL 任务、话题任务等。

`collection_task` 是任务级元数据，用于解释某一批数据为什么被采、按什么条件采、覆盖哪些平台/语种/地区/时间窗口。它不需要在每条 `social_content`、`social_account` 或 `news_article` 中重复展开；每条记录只需携带统一外层字段 `crawl_task_id` 即可。

```json
{
  "crawl_task_id": "task_20260624_001",
  "collection_method": "api",
  "seed_type": "keyword",
  "seed_value": "关键词或账号列表",
  "query_expression": "实际采集查询表达式",
  "platform": "x",
  "time_window_start": "2026-06-23T00:00:00Z",
  "time_window_end": "2026-06-24T00:00:00Z",
  "target_languages": ["en", "ja", "ko", "vi", "tl"],
  "target_regions": ["US", "JP", "KR", "VN", "PH"],
  "collector_version": "v1.0.0"
}
```

字段要求：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `crawl_task_id` | 是 | 采集任务 ID；应与各类记录外层的 `crawl_task_id` 对应 |
| `collection_method` | 是 | `api`、`crawler`、`rss`、`manual_import` 等 |
| `seed_type` | 是 | `keyword`、`account`、`url`、`topic`、`hashtag` 等 |
| `seed_value` | 是 | 实际采集种子，例如关键词、账号列表、URL 列表 |
| `query_expression` | 建议 | 实际查询表达式 |
| `platform` | 是 | 采集平台 |
| `time_window_start` | 建议 | 采集目标时间窗口开始 |
| `time_window_end` | 建议 | 采集目标时间窗口结束 |
| `target_languages` | 建议 | 目标语种 |
| `target_regions` | 建议 | 目标地区 |
| `collector_version` | 建议 | 采集器版本 |

说明：采集任务元数据用于判断数据覆盖范围、采样偏差、任务相关性和后续评测口径，建议作为必交付数据。若通过离线批量交付，可按批次单独提供 `collection_tasks_YYYY-MM-DD.jsonl`，每个采集任务一行。

## 六、交付方式建议

### 6.1 实时交付

建议使用 Kafka Topic 进行实时或准实时交付。

建议 Topic 划分：

| Topic | 内容 |
|---|---|
| `kt3.social_content` | 社交媒体内容 |
| `kt3.social_account` | 账号/频道/页面/群组快照 |
| `kt3.account_relation` | 账号关系 |
| `kt3.media_asset` | 多媒体附件元数据 |
| `kt3.news_article` | 新闻报道 |
| `kt3.collection_task` | 采集任务元数据 |

Kafka 消息体使用统一 JSON 外层格式，`data` 字段中放置对应类型的标准字段。

### 6.2 离线批量交付

建议使用 JSONL 或 Parquet 文件，按日期、平台和数据类型分区。

推荐目录结构：

```text
kt3_to_kt4/
  date=2026-06-24/
    platform=x/
      record_type=social_content/
        part-0001.jsonl
      record_type=social_account/
        part-0001.jsonl
      record_type=account_relation/
        part-0001.jsonl
    platform=youtube/
      record_type=social_content/
        part-0001.jsonl
      record_type=media_asset/
        part-0001.jsonl
    platform=news/
      record_type=news_article/
        part-0001.jsonl
```

### 6.3 媒体文件交付

媒体文件建议通过对象存储或文件包交付，元数据中通过 `storage_uri` 引用。

推荐路径示例：

```text
s3://kt3-media/2026/06/24/x/media_001.mp4
s3://kt3-media/2026/06/24/youtube/video_001_thumb.jpg
```

如无法使用对象存储，也可以按文件目录交付，但必须保证 `media_asset.storage_uri` 能够定位到对应文件。

## 七、课题二不应作为源数据必填项提供的内容

以下字段属于课题四算法分析和系统加工产出，不建议作为课题二源数据必填项：

1. 政治倾向。
2. 核心立场。
3. 复杂情感，例如怨恨、恐惧、道德义愤等。
4. 叙事 ID 或叙事分类。
5. 重点目标类型。
6. KOL 或关键意见领袖判断。
7. 隐蔽协同关系。
8. 潜在从属关系。
9. MBTI、决策风格、兴趣爱好等画像维度。
10. 影响、支持、煽动、平息等认知关系类型。

如课题二已具备 OCR、ASR、机器翻译、语言识别、基础去重等能力，可以作为可选增强字段提供，但必须同时保留原始文本、原始媒体和原始平台字段。

## 八、验收关注点

课题四接收课题二数据时，建议重点检查以下内容：

1. 是否覆盖指定平台、语种、地区和时间窗口。
2. 是否保留平台原始 ID、原始 URL、发布时间、采集时间。
3. 是否提供完整 `raw_payload` 和 `raw_payload_hash`。
4. 社交内容是否能关联到作者账号。
5. 评论、回复、引用、转发是否能通过 `parent_content_id`、`root_content_id`、`quoted_content_id`、`repost_of_content_id` 恢复传播链。
6. 关注、粉丝、订阅、成员、管理员等关系是否通过 `account_relation` 交付。
7. 媒体内容是否提供可访问的文件路径或对象存储 URI。
8. 采集任务元数据是否能解释数据来源和采样条件。
9. 批量文件或 Kafka 消息是否符合统一 JSON 外层格式。

## 九、简要结论

课题二向课题四交付的数据不应是画像库、知识图谱或认知分析结果，而应是面向课题四后续算法处理的跨平台原始事实数据。

最小交付范围应包括：

1. 社交媒体内容 `social_content`。
2. 账号/频道/页面/群组快照 `social_account`。
3. 账号关系 `account_relation`。
4. 多媒体附件 `media_asset`。
5. 新闻报道 `news_article`。
6. 采集任务元数据 `collection_task`。
