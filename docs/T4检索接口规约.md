# T4 检索接口规约
# 课题四 × T4 算法组接口对接文档

> 当前结论：T4 暂不需要算法组实现独立 HTTP 接口。文本/图像 embedding 生成，以及“文搜图”检索，都由后端内部实现。

---

## 一、当前职责划分

| 能力 | 当前实现方 | 说明 |
|---|---|---|
| 文本 embedding 生成 | 后端 `EmbeddingService` | 直接调用 `llm.embedding.*` 配置指向的 embedding 服务 |
| 图像 embedding 生成 | 后端 `EmbeddingService` | 直接调用同一个 embedding 服务 |
| 文本内容向量入库 | 后端 | `T4IndexingStep` 生成原文标题/摘要/正文向量并写入 Milvus `media_content_embeddings`；同时生成英文 pivot translation 向量并写入 `media_content_pivot_embeddings` |
| 媒体资源向量入库 | 后端 | `ImageEmbeddingService` 生成视觉/OCR/ASR/caption 原文向量并写入 Milvus `media_asset_embeddings`；OCR/ASR/caption 的英文 pivot translation 向量写入 `media_asset_pivot_embeddings` |
| 实体/账号/内容节点向量入库 | 后端 | 后端生成向量并统一写入 `entity_embeddings` |
| 文搜图/视频 | 后端 | 用户文本生成 query embedding 后，后端直接查 Milvus `media_asset_embeddings` 的视觉/OCR/ASR/caption 向量 |

也就是说，T4 不再通过 `AgentProxyClient.call("T4", "generate_text_embedding", ...)` 或 `generate_image_embedding` 这类 action 调用。embedding 生成是后端基础设施能力，不作为算法组接口交付。

---

## 二、后端内部 embedding 能力

后端新增内部服务：

```java
com.idata.profile.infra.embedding.EmbeddingService
```

核心方法：

```java
float[] generateTextEmbedding(String text);
float[] generateImageEmbedding(String imageUrl);
```

配置来源：

```yaml
llm:
  embedding:
    base-url: http://172.16.40.232:18013/v1
    api-key: idata@123456
    model: Qwen3-VL-Embedding-8B
```

调用方式：后端直接请求 `${llm.embedding.base-url}/embeddings`，请求体形如：

```json
{
  "model": "Qwen3-VL-Embedding-8B",
  "input": "文本或图片URL"
}
```

返回的 embedding 维度必须与 Milvus collection 的向量字段一致，当前为 4096。

文本内容有两套 Milvus 向量：

```text
Collection: media_content_embeddings
含义: 原文 title/summary/body embedding
Vector fields: title_embedding / summary_embedding / body_embedding

Collection: media_content_pivot_embeddings
含义: 英文 pivot translation 的 title/summary/body embedding
Vector fields: title_embedding / summary_embedding / body_embedding

Collection: media_asset_embeddings
含义: 媒体资产原文/视觉 embedding
Vector fields: visual_embedding / ocr_embedding / asr_embedding / caption_embedding

Collection: media_asset_pivot_embeddings
含义: 媒体 OCR/ASR/caption 的英文 pivot translation embedding
Vector fields: ocr_embedding / asr_embedding / caption_embedding
```

## 三、文搜图当前实现

“智能融合”里选择 `targetModalities=image` 时，后端执行：

```text
用户文本 query
  -> EmbeddingService.generateTextEmbedding(query)
  -> MilvusVectorService.searchMediaAssetTextEmbeddingsWithScore(...)
  -> 得到 media_assets.id + similarity score
  -> 后端查询 media_assets / media_contents 补齐图片URL和所属贴文信息
  -> 前端以图片网格展示，点击图片进入所属贴文详情
```

Milvus 查询范围包含：

```text
Collection: media_asset_embeddings
Vector fields: visual_embedding / ocr_embedding / asr_embedding / caption_embedding
Output fields: asset_id / segment_id

Collection: media_asset_pivot_embeddings
Vector fields: ocr_embedding / asr_embedding / caption_embedding
Output fields: asset_id / segment_id
```

其中 `asset_id` 对应 PostgreSQL `media_assets.id`。

## 四、接口状态

以下 action 不再作为 T4 算法接口要求：

| action | 状态 | 替代方案 |
|---|---|---|
| `generate_text_embedding` | 不再要求算法组实现 | 后端 `EmbeddingService.generateTextEmbedding` |
| `generate_image_embedding` | 不再要求算法组实现 | 后端 `EmbeddingService.generateImageEmbedding` |
| `search_images_by_text` | 暂不要求算法组实现 | 后端直接生成 query embedding 并查 Milvus `media_asset_embeddings` |

如果后续算法组要在“文搜图”上增加额外重排序、视觉语义解释、跨模态校准等能力，再单独定义新的 T4 检索接口；在当前阶段不需要。

## 五、内容融合检索返回说明

`POST /api/search/hybrid` 在 `targetModalities=all` 或默认模式下，后端按贴文聚合四路召回结果：

| 通道 | 类别 | 说明 |
|---|---|---|
| `ES_POST_KEYWORD` | `TEXT` | ES 检索贴文标题/正文 |
| `MILVUS_POST_SEMANTIC` | `TEXT` | Milvus 检索贴文原文 title/summary/body 向量和英文 pivot translation 向量 |
| `ES_MEDIA_KEYWORD` | `MEDIA` | ES 检索媒体 OCR/ASR/caption 文本 |
| `MILVUS_MEDIA_SEMANTIC` | `MEDIA` | Milvus 检索媒体 visual/OCR/ASR/caption 原文向量和 OCR/ASR/caption 英文 pivot translation 向量 |

后端会对每一路有序结果按 `content_id` 做 RRF 聚合。为了避免一条贴文因为媒体资源数量多而被重复加分，同一个通道内同一个 `content_id` 只贡献一次 RRF；重复命中的媒体资源只作为证据保留。

内容融合检索统一返回 `contentHits`。前端不再使用旧的 `items/imageItems` 作为内容结果列表；如果需要高亮，仍从 `highlights` 中按 `contentId` 读取。

```json
{
  "contentHits": [
    {
      "contentId": "post_uuid",
      "rrfScore": 0.049,
      "rankScore": 0.049,
      "matchLevel": "EXACT_PHRASE",
      "matchLevelRank": 3,
      "dominantHitType": "MEDIA_ASSET",
      "displaySuggestion": "MEDIA_FIRST",
      "contribution": {
        "text": {
          "rrfScore": 0.016,
          "ratio": 0.33
        },
        "media": {
          "rrfScore": 0.033,
          "ratio": 0.67
        }
      },
      "primaryAsset": {
        "entityId": "video:asset_uuid:seg_4",
        "assetId": "asset_uuid",
        "contentId": "post_uuid",
        "mediaType": "video",
        "previewUrl": "https://...",
        "sourceUrl": "https://...",
        "storageUri": "s3://...",
        "minioBucket": "media-assets",
        "minioKey": "media/2026/07/...",
        "mimeType": "video/mp4",
        "width": 1280,
        "height": 720,
        "segmentStartMs": 120000,
        "segmentEndMs": 150000,
        "previewTimeMs": 135000,
        "rrfContribution": 0.016
      },
      "matchedAssets": [],
      "evidences": [
        {
          "channel": "ES_MEDIA_KEYWORD",
          "category": "MEDIA",
          "rank": 1,
          "rrfContribution": 0.016,
          "rawScore": 8.1,
          "hitField": "ocr_text/asr_text/caption_text",
          "contentId": "post_uuid",
          "assetId": "asset_uuid",
          "entityId": "video:asset_uuid:seg_4"
        }
      ],
      "post": {}
    }
  ]
}
```

字段语义：

| 字段 | 说明 |
|---|---|
| `rrfScore` | 四路召回按 RRF 统一后的总分，只用于排序 |
| `rankScore` | 当前版本等同于 `rrfScore`，保留给前端/调试展示；最终排序还会先看 `matchLevelRank` |
| `matchLevel` | 文本命中等级：`EXACT_PHRASE` / `HIGH_TEXT_MATCH` / `TEXT_MATCH` / `HYBRID_RELEVANT` |
| `matchLevelRank` | 文本命中等级排序值。完整短语和高覆盖文本命中优先于普通媒体弱召回 |
| `contribution.text.rrfScore` | 文本侧贡献，来自 ES 贴文关键词和 Milvus 贴文语义 |
| `contribution.media.rrfScore` | 媒体侧贡献，来自 ES 媒体关键词和 Milvus 媒体语义 |
| `ratio` | 当前类别贡献占总贡献比例 |
| `dominantHitType` | `TEXT` / `MEDIA_ASSET` / `MIXED` |
| `displaySuggestion` | `TEXT_FIRST` / `MEDIA_FIRST` / `MIXED`，前端优先使用该字段决定展示样式 |
| `primaryAsset` | 媒体侧贡献最高的命中资源，前端列表优先展示 |
| `matchedAssets` | 其他命中媒体证据 |
| `evidences.rawScore` | ES BM25 或 Milvus cosine 原始分数，仅用于解释和调试，不建议前端直接比较 |

展示建议规则：

```text
mediaRatio >= 0.60 且 primaryAsset 达到强媒体命中阈值 -> MEDIA_FIRST
textRatio >= 0.60                                  -> TEXT_FIRST
否则                                                -> MIXED
```

排序规则：
```text
第一排序键：matchLevelRank，完整短语命中优先
第二排序键：rankScore / rrfScore
```

媒体语义默认准入阈值：
```text
OCR/ASR/caption 语义向量 >= 0.58
visual 视觉向量          >= 0.65
```

音频/视频定位统一使用毫秒字段：

```text
segmentStartMs
segmentEndMs
previewTimeMs
```

前端播放时可将 `previewTimeMs / 1000` 赋给播放器 `currentTime`。

## 六、边界原则

1. 后端负责 embedding 生成、Milvus 写入、Milvus 基础向量召回。
2. 后端负责根据 `assetId` 查询 PG，补齐图片 URL、所属贴文、平台、正文摘要等展示字段。
3. 算法组不需要访问 PostgreSQL、Elasticsearch、Neo4j。
4. 算法组当前不需要实现 T4 HTTP action。
5. 如果未来新增 T4 算法检索接口，响应也应优先返回轻量候选 ID 和分数，业务字段由后端补齐。
