> 版本：v2.0 · 2026-06-25
> 对应数据库设计：课题四_数据库设计_v3.md（21张表）
> 变更：T3融合冲突直接覆盖（不审核）；画像由定时任务批量生成（不做补全）；补充account_relations回填机制

## 一、总体说明

### 1.1 六种 record_type 的处理路径

| record_type | Kafka Topic | 是否走 T1-T4 | 终态 |
|---|---|---|---|
| `social_content` | kt3.social_content | ✅ 完整走 T1→T2→T3→T4 | T4_INDEXED |
| `news_article` | kt3.news_article | ✅ 完整走 T1→T2→T3→T4 | T4_INDEXED |
| `social_account` | kt3.social_account | ❌ 只做标准化映射 | NORMALIZED |
| `account_relation` | kt3.account_relation | ❌ 只做标准化映射 | NORMALIZED（后续有独立的UUID回填批处理） |
| `media_asset` | kt3.media_asset | ❌ 只做标准化映射 | NORMALIZED |
| `collection_task` | kt3.collection_task | ❌ 只做标准化映射 | NORMALIZED |

### 1.2 pipeline_status 状态机

```
RECEIVED → NORMALIZED → T1_DONE → T2_DONE → T3_DONE → T4_INDEXED
                                                              ↑
                                                           终态（social_content / news_article）

RECEIVED → NORMALIZED
                ↑
             终态（其余四种类型）

任意步骤失败 → FAILED（重试超3次后）
```

### 1.3 核心设计原则

**1. raw_records 必须第一个写。**
不管后续 T1-T4 是否成功，原始消息必须先持久化。任何步骤失败都可以从 raw_records 重跑，不需要重消费 Kafka。

**2. 每步 T 的完整输出存 raw_records 的 JSONB 字段。**
t1_output / t2_output / t3_output / t4_output 四个 JSONB 字段存对应步骤的完整响应，单行即可追溯全部处理过程：
```sql
SELECT t1_output, t2_output, t3_output, t4_output
FROM raw_records
WHERE source_record_id = 'x_post_123456';
```

**3. 平台原始 ID 先存，UUID FK 后回填。**
account_relations、media_contents.author_account_id 等字段初始为 null 或存字符串，不要求入库时立刻匹配 UUID 外键。专门的回填批处理负责后续补齐。这样不管课题三数据到达顺序如何，都不会因 FK 约束阻塞入库。

**4. T4 可与 T3 并行。**
T4 只依赖 body_text，不依赖 T3 的融合结果，两者可以同时触发，节省约 30-40% 的处理时间。

**5. T3 融合冲突直接覆盖，不审核。**
同一实体的字段如果新证据和已有值不一致（比如国籍字段从"美国"变成"英国"），T3 直接用新值覆盖旧值，不暂存、不推人工审核队列。实体归一（merge_history）逻辑不受影响，依然正常记录被合并的旧实体ID。

**6. 画像由定时任务批量生成，不做差量补全。**
T5不再是"用户查询时按需触发补全"，而是定时任务按 importance_score 排序，批量调用T5为每个实体一次性生成全量15维度画像，写入新版本，旧版本自动归档。

## 二、social_content 处理流程

**适用范围：** 主贴、评论、回复、转发、引用、频道消息、论坛帖子等平台内容对象。
**处理路径：** 走完整 T1→T2→T3→T4 四步流水线。

---

### Step 1：Schema 校验 + 去重检查

**执行方：** data-ingestion-service（接入适配器）
**操作：** 纯内存，不写任何表

**校验内容：**
- 必填字段不为空：`record_type`、`source_record_id`、`payload_hash`、`platform`、`data.platform_content_id`、`data.author_platform_user_id`
- 字段类型正确：时间字段为 ISO-8601 格式、数值字段非负
- `schema_version` 为已知版本（kt3_to_kt4_v1）

**去重查询：**
```sql
-- 任意一个命中即为重复，直接 ack 跳过
SELECT 1 FROM raw_records WHERE source_record_id = ? LIMIT 1;
SELECT 1 FROM raw_records WHERE payload_hash = ? LIMIT 1;
```

**校验失败处理：** 写错误日志，消息 ack，不进入后续流程，不重试。
**重复处理：** 直接 ack 跳过，不报错。

### Step 6：T2 信息抽取

**执行方：** pipeline-service → agent-proxy-service → T2 Agent（HTTP 调用）
**输入：** body_text + t1_output 中的 entities_hint + narrative_hint + 传播链字段（hashtags / mentions / parent_content_id 等）
**输出：** 识别出的实体列表（人物/组织/事件/叙事）、实体关系、传播链关系推导结果

#### T2 核心任务说明

T2 从传播链字段推导关系，不需要课题三单独交付 interaction 明细：
- `parent_content_id` → 推导评论/回复关系
- `repost_of_content_id` → 推导转发关系
- `quoted_content_id` → 推导引用关系
- `mentions` → 推导提及关系

**写表 1：** `persons` / `organizations` / `events` / `narratives`（INSERT 或 UPDATE，精简字段）

| 字段 | 值 | 说明 |
|------|-----|------|

| canonical_name | 拜登 | T2 识别的标准名 |

### Step 7：T3 信息融合（与 T4 并行触发）

**执行方：** pipeline-service → agent-proxy-service → T3 Agent（HTTP 调用）
**输入：** t2_output 中的实体列表 + 现有图谱中的已知实体
**输出：** 实体归一结果（多来源的同一实体合并为一个 ID）、实体关系、叙事聚类

**融合策略说明：** T3做两类工作，处理方式不同：
- **实体归一（merge）**：发现"拜登"和"Biden"是同一实体时，合并为一个节点，旧UUID记录进`merge_history`。这是正常归一逻辑，必须保留追溯链。
- **字段冲突覆盖**：发现某实体的`nationality`字段，旧值"美国"和新证据支持的"英国"不一致时，**直接用新值覆盖**，不暂存、不审核、不记录冲突历史。

**写表 1：** `Neo4j`（写入/更新节点和关系边）

```cypher
-- 写入/更新人物节点（含完整属性，新值直接覆盖旧值）
MERGE (p:Person {id: $person_id})
SET p.canonical_name = $name,
    p.aliases = $aliases,
    p.nationality = $nationality,   -- 冲突时新值直接覆盖
    p.importance_score = $score

-- 写入关系边
MERGE (p1:Person {id: $id1})-[:AFFILIATED_WITH {
    since: date($since), confidence: $confidence
}]->(o:Organization {id: $org_id})

-- 写入账号参与叙事的关系
MERGE (a:SocialAccount {id: $account_id})-[:PARTICIPATES_IN_NARRATIVE {
    post_count: $count, role: 'amplifier', confidence: $conf
}]->(n:Narrative {id: $narrative_id})
```

**写表 2：** `persons` / `organizations` / `narratives`（UPDATE）

| 字段 | 值 | 场景 |
|------|-----|------|
| merge_history | [旧UUID1, 旧UUID2] | T3 把多个实体归一时，记录被合并的旧 ID（实体归一逻辑，非冲突覆盖） |
| last_seen_at | NOW() | 每次归一处理时更新 |

**写表 3：** `raw_records`（UPDATE）

| 字段 | 值 |
|------|-----|
| t3_output | T3 Agent 完整响应（JSONB），含实体归一 ID 映射 |
| pipeline_status | T3_DONE |

**写表 4：** `pipeline_tasks`（UPDATE）

| 字段 | 值 |
|------|-----|
| t3_status | done |
| t3_duration_ms | 实际耗时 |

**状态变化：** `T2_DONE → T3_DONE`

**适用范围：** 关注/粉丝/订阅/成员/管理员等账号间关系。课题三 v2.0 新增类型，替代原 interaction 中的账号关系部分。
**处理路径：** 不走 T1-T4。独立的回填批处理负责补齐 UUID 并写 Neo4j。

---

### Step 2：写 raw_records

关键字段值：`record_type = account_relation`，`pipeline_status = RECEIVED`

---

|------|-----|
| pipeline_status | NORMALIZED |

**状态变化：** `RECEIVED → NORMALIZED`（入库完成）

---

### Step 4：为什么 from_account_id / to_account_id 会是空的

**根本原因：Kafka 消息到达顺序不保证。**

`account_relation` 消息完全可能先于对应账号的 `social_account` 消息到达，或者目标账号课题三还没采集到（比如是个私密账号）。

如果入库时强行要求这两个字段必须是合法的 `social_accounts.id` 外键，会出现两种问题：要么入库直接失败，要么必须等账号数据到齐才能写关系，两种都会阻塞数据流。

**所以设计上选择：关系先存下来，外键留空，账号数据到位后用独立批处理回填。**

---

### Step 5：UUID 回填批处理（独立任务，不在入库主流程内）

**触发方式：** 两种方式任选其一
- **方式A：** 跟随 T3 节奏。T3 本来就在批量处理实体，每轮 T3 顺带跑一遍回填 SQL
- **方式B：** 独立定时任务，比如每 5 分钟跑一次，专门负责 UUID 对齐，和 T3 解耦

**写表 1：** `account_relations`（UPDATE，增量执行，只处理还没回填的记录）

```sql
-- 回填 from_account_id：按 (platform, platform_user_id) 匹配
UPDATE account_relations ar
SET from_account_id = sa.id
FROM social_accounts sa
WHERE ar.from_account_id IS NULL
  AND sa.platform = ar.platform
  AND sa.platform_user_id = ar.source_platform_user_id;

-- 回填 to_account_id
UPDATE account_relations ar
SET to_account_id = sa.id
FROM social_accounts sa
WHERE ar.to_account_id IS NULL
  AND sa.platform = ar.platform
  AND sa.platform_user_id = ar.target_platform_user_id;
```

**写表 2：** `Neo4j`（写入关系边，仅处理刚回填完成、两端UUID都已就绪的记录）

```sql
-- 找出回填完成、可以写图谱的记录
SELECT id, from_account_id, to_account_id, relation_type, observed_at, source
FROM account_relations
WHERE from_account_id IS NOT NULL
  AND to_account_id IS NOT NULL;
  -- 实际实现建议加一个 synced_to_neo4j BOOLEAN 标记字段，避免重复写入
```

```cypher
-- 根据 relation_type 写对应的关系边（事实关系，置信度1.0）
MATCH (a:SocialAccount {id: $from_id}), (b:SocialAccount {id: $to_id})
MERGE (a)-[:FOLLOWS {
    observed_at: datetime($observed_at),
    source: $source,
    confidence: 1.0
}]->(b)

-- 其他关系类型按 relation_type 映射：
-- subscribe     → SUBSCRIBES_TO
-- member_of     → MEMBER_OF_GROUP
-- admin_of      → ADMIN_OF
-- owner_of      → OWNER_OF
```

> **注意：** 这些来自课题三的事实关系（FOLLOWS / SUBSCRIBES_TO 等）和 T6 推断的协同关系（COORDINATES_WITH）是两类不同的关系，在 Neo4j 中严格区分，COORDINATES_WITH 带 `detected_by: "T6"` 标记。

---

### Step 6：无法回填的"孤儿关系"如何处理

如果某条关系的 `target_platform_user_id` 对应的账号课题三一直没有交付（比如私密群组、平台限制无法采集），这条关系的 `to_account_id` 会**永久为 NULL**。

这不影响入库状态——这条记录在 `NORMALIZED` 阶段已经算完成，只是暂时无法在 Neo4j 里建立完整的关系边。

**建议监控指标：**
```sql
-- 统计回填超过7天仍未完成的关系数量，超过阈值需要人工核实
SELECT relation_type, COUNT(*) AS orphan_count
FROM account_relations
WHERE (from_account_id IS NULL OR to_account_id IS NULL)
  AND created_at < NOW() - INTERVAL '7 days'
GROUP BY relation_type;
```

数量过多通常说明：要么课题三某些账号类型的数据确实没交付全，要么 `platform_user_id` 的命名规则在两类消息（social_account 和 account_relation）里不一致，需要联调时核对。

---

### account_relation 写表汇总

| 阶段 | 写表 | 操作 |
|------|------|------|

## 八、画像生成流程（定时任务，不属于入库流程）

**触发方式：** 定时任务（每日凌晨或按配置周期），不是用户触发，不做差量补全。

---

### Step 1：按重要性排序，选取待生成画像的实体

```sql
SELECT id, canonical_name, importance_score
FROM persons
WHERE id NOT IN (
    -- 排除7天内已生成过 active 画像的实体
    SELECT person_id FROM person_profiles
    WHERE status = 'active' AND generated_at > NOW() - INTERVAL '7 days'
)
ORDER BY importance_score DESC
LIMIT 100;  -- 每轮处理数量，避免一次性压垮 T5
```

---

### Step 2：调用 T5 生成全量画像

**执行方：** 定时任务 → agent-proxy-service → T5 Agent（HTTP 调用）
**输入：** 该人物在 Neo4j 中的完整属性、关联的内容、关系网络
**输出：** 15 维度全量画像数据（一次性生成，非差量）

**写表：** `person_profiles`（INSERT 新版本）

| 字段 | 值 | 说明 |
|------|-----|------|
| person_id | persons.id | |
| portrait_version | 旧版本号+1 | 若无旧版本则为1 |
| status | active | 新版本直接激活 |
| political_orientation / political_score / ... | T5输出 | 15个维度的全部字段 |
| generated_at | NOW() | |

**写表：** `person_profiles`（UPDATE，归档旧版本）

```sql
UPDATE person_profiles
SET status = 'archived'
WHERE person_id = ? AND status = 'active' AND id != ?;  -- 排除刚插入的新版本
```

> **简化说明：** 不再有"缺失维度检测""候选项生成""冲突复核"这些中间步骤。T5 一次性产出全部15个维度，写入即为最终版本，旧版本直接归档。如发现明显错误，由人工直接编辑 `person_profiles` 对应字段，`reviewed_at`/`reviewer_id` 记录修正信息。

## 九、失败重试机制

所有类型的处理失败，统一按以下策略处理：

| 场景 | 处理方式 |
|------|--------- |
| Schema 校验失败 | 写错误日志，消息 ack，不重试 |
| raw_records 写入失败 | 消息不 ack，Kafka 重新投递（最多3次） |
| L1 标准化写入失败 | pipeline_status = FAILED，写 error_message |
| T1 调用失败 | t1_status = failed，retry_count +1，指数退避重试，超3次推死信队列 |
| T2 调用失败 | 同 T1，支持从 T2 步骤单独重跑（raw_records 已落，不需重新采集） |
| T3 调用失败 | 同 T1，支持从 T3 步骤重跑 |
| T4 调用失败 | 同 T1，支持从 T4 步骤重跑 |
| account_relations 回填长期未完成 | 不算失败，定期监控孤儿关系数量，超阈值人工核实 |
| T5 画像生成失败 | 记录失败实体，下一轮定时任务重新尝试，不影响其他实体 |

**重跑方式：**

```sql
-- 查找失败记录
SELECT id, source_record_id, error_step, error_message
FROM pipeline_tasks
WHERE status = 'FAILED'
ORDER BY created_at DESC;

-- 从指定步骤重跑（以 T3 为例）
UPDATE pipeline_tasks
SET t3_status = 'pending', status = 'PENDING', retry_count = 0
WHERE id = ?;
-- pipeline-service 定时扫描 PENDING 状态的任务，自动重新触发
```

## 十、数据追溯方式

### 10.1 查一条数据的完整处理过程

```sql
-- 单行查全部处理步骤（不需要跨表JOIN）
SELECT
    id,
    source_record_id,
    pipeline_status,
    t1_output,    -- T1 标注的完整输出
    t2_output,    -- T2 抽取的实体/关系
    t3_output,    -- T3 融合的归一结果（含实体ID映射）
    t4_output     -- T4 写入的向量ID、ES文档ID
FROM raw_records
WHERE source_record_id = 'x_post_123456';
```

### 10.2 查一个采集批次的数据

```sql
-- 查某个采集任务的所有数据
SELECT r.source_record_id, r.pipeline_status, r.platform
FROM raw_records r
WHERE r.crawl_task_id = 'task_20260624_001'
ORDER BY r.created_at;

-- 查采集任务的覆盖情况
SELECT ct.platform, ct.target_languages, ct.records_collected,
       COUNT(r.id) AS actually_ingested
FROM collection_tasks ct
LEFT JOIN raw_records r ON r.crawl_task_id = ct.crawl_task_id
GROUP BY ct.id;
```

### 10.3 查流水线各步骤性能

```sql
SELECT
    AVG(t1_duration_ms) AS avg_t1_ms,
    AVG(t2_duration_ms) AS avg_t2_ms,
    AVG(t3_duration_ms) AS avg_t3_ms,
    AVG(t4_duration_ms) AS avg_t4_ms,
    COUNT(*) FILTER (WHERE status = 'DONE')   AS done,
    COUNT(*) FILTER (WHERE status = 'FAILED') AS failed
FROM pipeline_tasks
WHERE created_at > NOW() - INTERVAL '24 hours';
```

### 10.4 查 account_relations 的回填进度

```sql
SELECT
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE from_account_id IS NOT NULL AND to_account_id IS NOT NULL) AS fully_linked,
    COUNT(*) FILTER (WHERE from_account_id IS NULL OR to_account_id IS NULL) AS pending_or_orphan
FROM account_relations;
```
