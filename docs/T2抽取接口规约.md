# T2 实体关系抽取接口规约
# 课题四 × T2 算法组接口对接文档

> 本文档专门给 T2 算法组用，只讲 T2 自己要实现的接口。T2 输出结果之后后端怎么做候选召回、T3 怎么判断合并/新建、关系怎么落图——这些是后端和 T3 的事，T2 不需要关心，如果想了解完整链路，参考 `docs/Kg融合接口规约 v1.md`。

---

## 一、T2 在整个流水线里的位置

```
Kafka消息（社交内容）→ T1 标注 → T2 实体关系抽取 → 后端候选召回 → T3 判断合并/新建 → 关系落图 → T4 向量化索引
```

T2 拿到一段文本（可能带标题、可能带 T1 的标注结果作为参考），任务是**从文本里识别出人物/组织/事件/地点这些实体，以及它们之间的关系**，输出的是**mention 级结果**——也就是说，T2 只负责"这段文字里出现了什么、它们之间什么关系"，不负责判断"这个实体是不是图谱里已经存在的某个实体"（那是 T3 的活）。所以 T2 输出的每个实体都有一个自己生成的临时 ID（`mentionId`），不是图谱里的正式实体 ID。

---

## 二、接口

### POST（action=`extract_entities`）

#### 统一响应包裹

T2 接口的 HTTP 响应体统一使用 `code/msg/data` 包裹：

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "...": "原业务响应体"
  }
}
```

约定：

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | number | `0` 表示业务成功；非 `0` 表示业务失败 |
| `msg` | string | 成功时建议固定为 `success`；失败时返回简短错误原因 |
| `data` | object\|null | 成功时为原来的 T2 抽取结果；失败时可以为 `null` |

HTTP 4xx/5xx 仍表示协议级或服务级失败。`code != 0` 表示 HTTP 请求成功到达算法服务，但业务处理失败，例如输入不合法、模型不可用、schema 不支持等。

```json
    "schemaVersion": "t1_annotation_v0.6",

      {

        "mediaId": null,

        "timeRange": null,

    "contentId": "内容ID，回显用",

```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `title` | string | 否 | 新闻类内容标题，社交媒体帖子通常为空 |
| `text` | string | 是 | 待抽取的正文文本 |
| `t1Annotation` | object | 否 | T1 `annotate_content` 的完整业务响应体，供 T2 参考；包含 `aigcDetection`、`annotations.basicObjective.entitiesHint`、`evidenceClues` 等字段，不是强制要求必须采纳 |
| `context.contentId` | string | 否 | 内容ID，原样回显在响应体的 `data.contentId` 字段，方便后端对应 |
| `context.platform` | string | 否 | 来源平台代码 |
| `context.hashtags`/`mentions` | array | 否 | 话题标签、@提及，可以辅助判断实体和话题相关性 |
| `context.parentContentId`/`repostOfContentId`/`quotedContentId` | string | 否 | 传播链信息（回复/转发/引用的上级内容ID），T2 不需要处理这些关系，后端会单独处理内容间的传播关系 |
| `language` | string | 否 | 语言代码 |

以下字段均位于成功响应的 `data` 内。

**`entities[]` 字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `mentionId` | string | 是 | 本条响应内唯一的临时ID（如 `m1`/`e1`），算法组自行生成，用于 `relations` 里引用，不是图谱正式实体ID |
| `name` | string | 是 | 原文出现的名称（surface form），比如文本里写的"CENTCOM"就填"CENTCOM" |
| `canonicalName` | string | 是 | 标准名（跨语言归一），这个名字如果后续判断要新建实体，会直接作为图谱里的正式名称，务必准确、规范 |
| `type` | string | 是 | 实体类型，取值：`person`/`organization`/`event`/`location`。**没有 `narrative` 这个类型**，叙事/议题不作为独立实体抽取 |
| `span` | object | 否 | 这个实体在 `text` 里的字符位置，`{"start": 数字, "end": 数字}`，不是数组 |
| `aliases` | array | 否 | 同一实体的其他表达方式/别名 |
| `importanceScore` | number | 否 | 0-100，这个实体在当前内容里的重要程度 |
| `confidence` | number | 否 | 0-1，抽取置信度 |
| `attributes` | object | 否 | 扩展属性，按 `type` 不同填不同内容，见下方"`attributes` 里认知领域类别属性"一节 |

#### `attributes` 里的认知领域类别属性

任务书要求"不少于10类认知领域实体"（境外政要、关键意见领袖、国家机构、媒体机构……），这些类别标签**不是 `type` 字段的取值**（`type` 永远只是 `person`/`organization`/`event`/`location` 这4种），而是放在 `attributes` 里的细分属性：

| `type` | `attributes` 里的字段 | 类型 | 说明 |
|---|---|---|---|
| `person` | `personCategories` | 字符串数组 | 人物的认知领域类别，**允许多标签并存**（比如一个人可以同时是"领域专家"又是"关键意见领袖"），不确定或者信息不够时可以给 `["general_person"]` 或者干脆不填这个属性，不要为了贴标签硬猜。取值范围（8个）：`foreign_public_official`（境外政要）/`key_influencer`（关键意见领袖）/`domain_expert`（领域专家）/`media_person`（媒体人物）/`business_figure`（商业人物）/`civic_actor`（社会活动人物）/`active_netizen`（活跃网民）/`general_person`（未明确类别，兜底） |
| `organization` | `organizationCategories` | 字符串数组 | 组织的认知领域类别，同样允许多标签。取值范围（5个）：`state_institution`（国家机构）/`political_organization`（政治组织）/`social_organization`（社会组织，含NGO/智库）/`media_organization`（媒体机构）/`enterprise`（企业机构，含平台公司） |
| `event` | `eventType`/`eventTimeStart` | string | 不变，见响应体示例 |
| `location` | （无） | | `location` 不需要额外的类别细分，`attributes` 传空对象 `{}` 即可 |

`event.attributes.eventType` 取值范围：`politics` / `military` / `economy` / `society` / `culture` / `science_tech` / `security` / `nature` / `other`。无法归入前 8 类时使用 `other`。

每个类别的详细定义、典型对象举例、判断边界，见 `docs/T2抽取实体与关系说明.md` 第一节，这里不重复贴。

**`relations[]` 字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `relationMentionId` | string | 是 | 本条响应内唯一ID |
| `subjectMentionId`/`objectMentionId` | string | 是 | 引用 `entities[].mentionId`，表示这条关系的主体和客体 |
| `predicate` | string | 是 | 关系类型，**必须**从 `docs/关系词表与头尾实体类型说明.md` 定义的16个关系类型里选取，不在词表内的会被后端过滤掉、不会写入图谱，等于白抽 |
| `confidence` | number | 否 | 0-1 |
| `evidence` | string | 否 | 支撑这条关系判断的原文片段 |

**`resolvedAuthorAccountId`**：如果 T2 判断出这条内容的作者账号对应图谱里的某个已知账号ID，可以在这里返回；不确定就返回 `null`，后端有自己的账号关联逻辑兜底，不强制要求 T2 必须填。

---

## 三、关键约束

1. **`predicate` 必须在16个关系词表内**，具体是哪16个、每个的头尾实体类型限制，看 `docs/关系词表与头尾实体类型说明.md`，这份文档是唯一权威来源，不要按旧文档或者训练数据记忆的关系名硬编。
2. **不要输出 `type=narrative` 的实体**，叙事/议题类内容如果需要表达，通过 `attributes` 或者关系描述承载，不再是独立实体。
3. **`canonicalName` 必须填**，这是后续实体消歧、新建实体的关键字段，不能留空或者跟 `name` 完全一样却明显不是标准名（比如原文缩写、错别字）。
4. **`span` 建议填**，虽然不是强制必填，但没有这个字段后续没法在原文里定位证据、做高亮展示，建议尽量给。
5. **不用自己输出"内容指向实体"的关系**——每条内容成功解析出来的实体，后端会自动补一条 `(MediaContent)-[DESCRIBES]->(Event)` 或 `(MediaContent)-[MENTIONS]->(Person|Organization|Location)`，`relations[]` 里只需要给实体和实体之间的关系（比如"谁影响了谁""谁支持谁"），不用再单独输出一条"这条内容提到了这个实体"，后端会处理，重复输出也没关系，会被去重。
6. **`personCategories`/`organizationCategories` 允许多标签、允许不填**——不是必填字段，信息不够判断类别时，宁可不填或者填 `general_person` 兜底，不要为了凑齐"10类"去猜；这两个字段影响的是任务书验收口径下的统计维度，不影响实体本身能不能正确落图。

## 四、不再使用的旧字段/概念（提醒，避免误实现）

- ~~`sourceName`/`targetName`/`relationType`~~ → 改成 `subjectMentionId`/`objectMentionId`/`predicate`，且都是引用 `mentionId` 不是直接给实体名字。
- ~~独立的 `events` 数组~~ → event 现在混在 `entities` 里，`type=event`。
- ~~`docId`~~ → 改名 `context.contentId`。
- ~~`sourceInfo`~~ → 改名 `context`。
