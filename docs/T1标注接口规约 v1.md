# T1 自动标注接口规约
# 课题四 × T1 算法组接口对接文档

> **重要变更说明**：本文档相对旧版（v1.1）做了架构级重写，不是字段增补。旧版的 `/t1/annotate_text`、`/t1/annotate_image`、`/t1/batch_annotate` 三个接口已经废弃，现在统一成一个 `annotate` 接口，同时支持纯文本、纯图片/视频、图文混合三种输入。另外新增了两个独立接口：`annotate_account`（账号类别判断）、`annotate_event_heat`（事件热度判断），这两个不是"内容标注"的一部分，输入输出结构完全不同，见本文档第三、四节。
>
> `annotate` 接口的响应体字段字典（15个标注维度的枚举值、字段说明）已经在 `docs/T1_annotation_v0.5_README.md` 里非常详细地写过了，本文档不重复列举，只讲接口调用层面的东西（URL、请求体、跟 README 的对应关系）。**两份文档配合看**：这份看"怎么调接口"，`T1_annotation_v0.5_README.md` 看"响应体每个字段具体什么意思、枚举值有哪些"。

## 一、接口总览

| 接口 | action | 用途 |
|---|---|---|
| `annotate` | `annotate` | 内容标注（文本/图片/视频/图文混合），15个标注维度 + AIGC检测 |
| `annotate_account` | `annotate_account` | 账号类别判断（独立于内容，基于账号资料） |
| `annotate_event_heat` | `annotate_event_heat` | 事件热度判断（独立于单条内容，基于图谱聚合数据） |

**URL 拼接规则不变**：`完整请求URL = sub_agent_registry.mock_url（或base_url） + "/" + action`。三个接口共用同一个 `T1` agent_code，通过 `action` 区分。

**为什么内容标注只有一个接口，不再区分文本/图片**：任务书要求支持"文本、图像、视频及图文混合内容"的AIGC检测，图文混合场景下的一致性判断（比如"这张图和这段文字是否匹配、是否有拼接伪造痕迹"）必须让文本和图像在同一次调用里一起看，分开调用天然做不到。所以这次把输入合并成一个统一结构，调用方按当时手头有什么数据（只有文字、只有图、文字+图都有）自行传入，不需要凑齐了才能调用——响应体里对应模态字段用 `not_applicable` 表示"这次没有这个模态的数据"。

## 二、内容标注接口

### POST（action=`annotate`）

#### 请求体

```json
{
  "title": "文章标题，社交内容通常为空",
  "text": "正文文本，可以为空（纯图片/视频内容时）",
  "language": "zh",
  "medias": [
    {
      "id": "对应 media_assets.id，用于结果里 evidence_clues.mediaId 回填定位",
      "url": "MinIO 可访问地址",
      "mediaType": "image | video"
    }
  ],
  "context": {
    "contentId": "内容ID，用于日志追踪",
    "platform": "来源平台代码，如 x / weibo / telegram",
    "url": "内容原始链接",
    "contentType": "post | comment | reply | article",
    "authorHandle": "作者账号标识",
    "publishedAt": "ISO8601时间",
    "hashtags": ["话题标签数组"],
    "likeCount": 0,
    "commentCount": 0,
    "shareCount": 0,
    "repostCount": 0,
    "viewCount": 0,
    "parentContentId": "父内容ID，回复/评论场景"
  }
}
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `title` | string | 否 | 新闻类内容的标题，社交媒体帖子通常没有，传空即可 |
| `text` | string | 否 | 正文文本；`text` 和 `medias` 至少要有一个非空，不能两个都空 |
| `language` | string | 否 | 语言代码，不传时按文本自动判断 |
| `medias` | array | 否 | 图片/视频列表，可以为空、一张、多张混合（图和视频可以同时存在） |
| `medias[].id` | string | 是（有media时） | 用于响应体 `evidenceClues[].mediaId` 回填，标识这条证据来自哪个媒体文件 |
| `medias[].mediaType` | string | 是（有media时） | `image` 或 `video`，决定这个文件走哪套 AIGC 检测逻辑 |
| `context.likeCount`/`commentCount`/`shareCount`/`repostCount`/`viewCount` | number | 否 | 互动数据，供 `risk_level`/`opinion_emotion` 等维度参考（比如互动量极高+负面情绪，风险等级应该相应提高） |

**关于 `event_heat`**：内容标注这次**不再输出事件热度**，这个维度已经独立成 `annotate_event_heat` 接口（见第四节），原因是单条内容的"热度感知"信息量太有限，判断不准，真正的事件热度需要图谱里聚合足够多关联内容之后才能算。

#### 响应体

响应体结构（`schemaVersion`/`inputReference`/`aigcDetection`/`annotations`/`evidenceClues`/`qualityControl`/`overallConfidence`/`processedAt`）、15个标注维度的完整字段字典和枚举值，见 `docs/T1_annotation_v0.5_README.md`。

**唯一需要补充的是 `inputReference.contentType` 的取值规则**（这个字段由后端根据这次请求实际传了什么内容拼出来，算法组只需要按需返回，不需要自己判断）：

| 输入情况 | `inputReference.contentType` |
|---|---|
| 只有 `text`，没有 `medias` | `text` |
| 只有 `medias`（图片），没有 `text` | `image` |
| 只有 `medias`（视频），没有 `text` | `video` |
| `text` + `medias` 都有 | `text_image_mixed` |

## 三、账号类别判断接口

### POST（action=`annotate_account`）

**用途**：判断一个社交账号属于任务书要求的哪一类主体（个人/媒体/政府机构/KOL/疑似机器人等）。这是**账号级**判断，跟内容标注是两回事——内容标注里也有一个 `account_type` 字段（属于15个维度之一），但那个是"只看这一条内容，弱信号兜底判断"，权威判断走这个独立接口，基于账号完整资料（简介、认证状态、粉丝数等）。

**触发时机**：账号摄入时（`SocialAccountConsumer`）同步调用，判断结果写入 `social_accounts.account_type`/`account_type_confidence`。

#### 请求体

```json
{
  "platform": "x",
  "platformUserId": "平台原始用户ID",
  "accountEntityType": "user | channel | page | group | community | forum_board | news_source",
  "platformNativeType": "平台自己的账号类型标识（如果有）",
  "handle": "@账号名",
  "displayName": "显示名称",
  "bio": "账号简介",
  "selfDeclaredLocation": "账号自己填写的地区",
  "verified": true,
  "verifiedType": "none | blue | org | government | media",
  "isSuspended": false,
  "accountCreatedAt": "ISO8601时间",
  "followersCount": 0,
  "followingCount": 0,
  "subscriberCount": 0,
  "memberCount": 0,
  "postCount": 0,
  "viewCount": 0,
  "recentPostSamples": ["该账号近期发帖文本样本，没有就传空数组，不是必填"]
}
```

#### 响应体

```json
{
  "schemaVersion": "t1_account_annotation_v0.5",
  "accountType": {
    "primaryAccountCategory": "ordinary_user | news_media | state_affiliated_media | government_agency | political_actor | political_party_or_campaign | military_security_agency | international_organization | ngo_or_civil_society | academic_or_expert | commercial_brand | platform_official | influencer_kol | community_group | anonymous_account | suspected_bot_or_automated | unknown | other",
    "accountSubtypeTags": ["更细分的标签，可选"],
    "automationSuspicion": "none | low | medium | high | unclear",
    "accountTypeConfidence": 0.0,
    "evidenceIds": ["ev_acc_001"]
  },
  "evidenceClues": [
    {"evidenceId": "ev_acc_001", "evidenceType": "metadata", "source": "metadata", "evidenceText": "支撑判断的具体依据，比如简介原文或认证状态"}
  ],
  "qualityControl": {
    "needHumanReview": false,
    "reviewReasons": ["none"],
    "failedModules": ["none"]
  },
  "overallConfidence": 0.0,
  "processedAt": "ISO8601时间"
}
```

**判断依据的优先级建议**：`bio`/`displayName` 里如果有明确的身份自陈（比如"XX部门官方账号"），优先按这个判断；没有明确自陈的，参考 `verified`/`verifiedType`/粉丝量级/`accountEntityType` 综合判断；信息不足时老实标 `unknown`，不要瞎猜。

## 四、事件热度判断接口

### POST（action=`annotate_event_heat`）

**用途**：判断图谱里某个 `Event` 实体当前的热度等级。这是**事件级**判断，输入不是某一条内容，而是"这个事件 + 图谱里跟它关联的一跳实体（人物/组织/地点/账号/抽样后的相关内容）+ 后端算好的聚合统计数字"。

**触发时机**：不是实时触发，由后端定时任务 `EventHeatJob` 周期性扫描需要（重新）计算热度的事件，批量调用。

#### 请求体

```json
{
  "event": {
    "eventId": "事件ID",
    "canonicalName": "事件标准名称",
    "eventType": "election | military | diplomatic | protest | disaster | other",
    "occurredAtStart": "ISO8601时间",
    "occurredAtEnd": "ISO8601时间",
    "country": "国家/地区"
  },
  "relatedEntities": [
    {
      "entityId": "关联实体ID",
      "entityType": "person | organization | location | media_content | social_account",
      "name": "实体名称（entityType=media_content时此字段为空）",
      "publishedAt": "仅entityType=media_content时有值",
      "platform": "仅entityType=media_content时有值",
      "likeCount": 0,
      "commentCount": 0,
      "shareCount": 0,
      "repostCount": 0,
      "viewCount": 0
    }
  ],
  "aggregateStats": {
    "totalRelatedContentCount": 0,
    "totalEngagement": 0,
    "distinctPlatformCount": 0,
    "earliestContentAt": "ISO8601时间",
    "latestContentAt": "ISO8601时间"
  }
}
```

**注意**：`relatedEntities` 里的 `media_content` 类型条目，是后端按互动量从图谱里全部关联内容中抽样出来的 **Top 20**，不是全量——真实的关联内容总数看 `aggregateStats.totalRelatedContentCount`，不要用 `relatedEntities` 里 `media_content` 条目的数量当总数。

#### 响应体

```json
{
  "schemaVersion": "t1_event_heat_v1",
  "eventId": "事件ID，跟请求里的对应",
  "eventHeat": {
    "heatLevel": "low | medium | high | explosive | unclear",
    "heatScore": 0.0,
    "heatSignalTypes": ["content_volume | engagement_surge | rapid_growth | wide_platform_spread | sustained_attention | declining | insufficient_data | unclear"],
    "reasoning": "简要判断理由",
    "confidence": 0.0
  },
  "overallConfidence": 0.0,
  "processedAt": "ISO8601时间"
}
```

**判断建议**：`aggregateStats.totalRelatedContentCount` 为 0 时（图谱里还没有任何内容关联到这个事件），应该返回 `heatLevel=unclear`、`heatSignalTypes=["insufficient_data"]`，不要瞎猜一个热度等级出来。

## 五、后端消费 T1 结果的方式（供理解，不要求算法组处理）

- `annotate` 的完整响应体，后端整段存成 JSON（PG `media_contents.t1_annotation` 一个字段，Neo4j `MediaContent` 节点 `t1Annotation` 一个属性），不再逐字段拆解落库，算法组不需要关心具体哪个字段对应哪一列。
- `annotate_account` 的结果写入 `social_accounts.account_type`/`account_type_confidence`。
- `annotate_event_heat` 的结果写入 `events.event_heat_level`/`event_heat_score`/`event_heat_confidence`/`event_related_content_count`/`event_heat_computed_at`，以及 Neo4j `Event` 节点的 `eventHeatLevel`/`eventHeatScore` 属性。
