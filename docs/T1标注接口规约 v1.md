# T1 自动标注接口规约
# 课题四 × T1 算法组接口对接文档

> 本文档描述 T1 三个接口的调用方式（URL、请求体、响应体结构）。字段字典（每个字段的完整枚举值和含义）见 `docs/T1_annotation_v0.6.json`/`T1_annotation_v0.6_README.md`，两份文档配合看：这份看"怎么调接口"，README 看"每个字段具体什么意思"。
>
> **当前后端代码尚未迁移到本文档描述的结构**，迁移是单独安排的一轮工作，迁移完成前 `sub_agent_registry` 里 `active_url_type=mock` 返回的示例数据暂时还是旧结构，不代表最终形态。

## 一、接口总览

| 接口 | action | 用途 |
|---|---|---|
| `annotate` | `annotate` | 内容标注（文本/图片/视频/图文混合），AIGC检测 + 6个高价值主观维度 + 5个基础客观维度 |
| `annotate_account` | `annotate_account` | 账号类别判断（独立于内容，基于账号主页/快照资料） |
| `annotate_event_heat` | `annotate_event_heat` | 事件热度判断（独立于单条内容，基于已聚合的图谱数据） |

**URL 拼接规则**：`完整请求URL = sub_agent_registry.mock_url（或base_url） + "/" + action`。三个接口共用同一个 `T1` agent_code，通过 `action` 区分。

三个接口统一使用 `"schemaVersion": "t1_annotation_v0.6"`，同步维护和升级，不分别维护独立版本号。

**为什么内容标注只有一个接口，不区分文本/图片**：图文混合场景下的一致性判断（比如"这张图和这段文字是否匹配"）必须让文本和图像在同一次调用里一起看。调用方按当时手头有什么数据（只有文字、只有图、文字+图都有）自行传入，不需要凑齐了才能调用——响应体里对应模态字段用 `not_applicable` 表示"这次没有这个模态的数据"。

## 二、内容标注接口

### POST（action=`annotate`）

#### 请求体

```json
{
  "title": "文章标题，社交内容通常为空",
  "text": "正文文本，可以为空（纯图片/视频内容时）",
  "language": "zh",
  "medias": [
    {
      "id": "对应 media_assets.id，用于结果里 evidence_clues.mediaId 回填定位",
      "url": "MinIO 可访问地址",
      "mediaType": "image | video"
    }
  ],
  "context": {
    "contentId": "内容ID",
    "platform": "来源平台代码，如 x / weibo / telegram",
    "url": "内容原始链接",
    "contentType": "post | comment | reply | article",
    "authorHandle": "作者账号标识",
    "publishedAt": "ISO8601时间",
    "hashtags": ["话题标签数组"],
    "likeCount": 0,
    "commentCount": 0,
    "shareCount": 0,
    "repostCount": 0,
    "viewCount": 0,
    "parentContentId": "父内容ID，回复/评论场景"
  }
}
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `title` | string | 否 | 新闻类内容的标题，社交媒体帖子通常没有，传空即可 |
| `text` | string | 否 | 正文文本；`text` 和 `medias` 至少要有一个非空 |
| `language` | string | 否 | 语言代码，不传时按文本自动判断 |
| `medias` | array | 否 | 图片/视频列表，可以为空、一张、多张混合 |
| `medias[].id` | string | 是（有media时） | 用于响应体 `evidenceClues[].mediaId` 回填 |
| `medias[].mediaType` | string | 是（有media时） | `image` 或 `video` |
| `context.likeCount`/`commentCount`/`shareCount`/`repostCount`/`viewCount` | number | 否 | 互动数据，供 `riskLevel`/`opinionEmotion` 等维度参考 |

#### 响应体

顶层结构：

```
annotate
├── schemaVersion
├── inputReference
├── language
├── aigcDetection
├── annotations
│   ├── highValueSubjective
│   └── basicObjective
├── evidenceClues
├── qualityControl
├── overallConfidence
└── processedAt
```

##### `inputReference`

只保存原始内容引用和基础元数据，不复制完整原文或媒体文件。

| 字段 | 类型 | 说明 |
|---|---|---|
| `contentId` | string | 原始内容唯一标识 |
| `contentType` | 枚举 | `post`/`comment`/`reply`/`article`，原样回显请求里 `context.contentType` |
| `modalityCombination` | 枚举 | `text`/`image`/`video`/`text_image`/`text_video`/`image_video`/`text_image_video`，由后端根据这次请求实际传了 `text`/`medias` 里的哪些模态算出来告诉算法组 |
| `platform` | string\|null | 来源平台 |
| `url` | string\|null | 原始内容URL |
| `authorId` | string\|null | 作者或发布账号标识 |
| `createdAt` | string\|null | 原始内容创建时间 |

##### `annotations.highValueSubjective`：6个维度

```
ideology
coreStance
opinionEmotion
languageStyle
manipulationMethod
riskLevel
```

`coreStance` 结构（`stanceTarget` 表示这条内容的核心立场指向的对象，是内容级的泛化描述，不解析到具体图谱实体）：

```json
"coreStance": {
  "stanceTarget": {
    "targetType": "event | issue | policy | action | person | organization | country_or_region | ideology_or_value | other | unclear",
    "targetText": "立场指向的对象，原文表达，比如\"这项制裁政策\""
  },
  "stanceLabel": "support | oppose | neutral | mixed | unclear",
  "stanceStrength": "weak | medium | strong | unclear",
  "coreStanceConfidence": 0.0,
  "evidenceIds": []
}
```

`manipulationMethod` 结构（BEND八类固定方法，可多选，也可以是空数组）：

```json
"manipulationMethod": {
  "methodLabels": ["engage | explain | excite | enhance | dismiss | distort | dismay | distract"],
  "manipulationMethodConfidence": 0.0,
  "evidenceIds": []
}
```

##### `annotations.basicObjective`：5个维度

```
topicTags
entitiesHint
keywords
summary
eventType
```

`topicTags` 只输出一个 `primaryDomain`（不再细分子话题）。`entitiesHint` 每条包含 `entityHintId`/`text`/`typeHint`/`entityHintConfidence`/`evidenceIds`（实体在原文里的位置通过 `evidenceIds` 指向证据池，不单独设 `span` 字段）。`keywords` 每条包含 `keywordText`/`keywordConfidence`/`evidenceIds`（关键词来源和位置同样由证据池表达）。`summary` 只有 `summaryText`/`summaryConfidence`，不设 `evidenceIds`（摘要依据整条多模态输入生成，不指向某一处具体证据）。

##### `evidenceClues`

```
evidenceClues[]
├── evidenceId
├── evidenceType     (text_span | image_region | video_segment | video_frame_region | metadata | model_signal)
├── source           (text | image | video | audio | ocr | subtitle | metadata | aigc_detector)
├── evidenceText
├── span             ([start, end] 字符区间)
├── mediaId
├── region           (x/y/width/height，建议0.0-1.0归一化坐标)
├── timeRange        (start/end，单位秒)
├── metadataSnapshot
└── modelSignal
```

证据必须来自原始输入、原始元数据或检测器原始输出，不把模型生成的解释文本当作原始证据；不适用的定位字段用 `null`。

##### `qualityControl`

```
needHumanReview   布尔值
reviewReasons     [low_confidence, insufficient_context, conflicting_signals, multimodal_inconsistency,
                    aigc_suspicious, high_risk_content, missing_metadata, module_failure,
                    too_short_input, manual_policy_required, other]
failedModules     [aigcDetection, textAigcDetection, imageAigcDetection, videoAigcDetection,
                    multimodalAigcDetection, ideology, coreStance, opinionEmotion, languageStyle,
                    manipulationMethod, riskLevel, topicTags, entitiesHint, keywords, summary,
                    eventType, other]
```

模态不存在使用 `not_applicable` 不算模块失败；输出 `unclear` 不自动等于模块失败；`failedModules` 非空时通常应设 `needHumanReview=true`、`reviewReasons` 包含 `module_failure`。

## 三、账号类别判断接口

### POST（action=`annotate_account`）

**用途**：基于账号主页或账号快照的完整资料，判断账号的主要类别、细分标签、自动化嫌疑。这是**账号级**判断，不是基于单条内容。

**触发时机**：账号摄入时（`SocialAccountConsumer`）同步调用，判断结果写入 `social_accounts.account_type`/`account_type_confidence`。

#### 请求体

```json
{
  "platform": "x",
  "platformUserId": "平台原始用户ID",
  "accountEntityType": "user | channel | page | group | community | forum_board | news_source",
  "platformNativeType": "平台自己的账号类型标识（如果有）",
  "handle": "@账号名",
  "displayName": "显示名称",
  "bio": "账号简介",
  "selfDeclaredLocation": "账号自己填写的地区",
  "verified": true,
  "verifiedType": "none | blue | org | government | media",
  "isSuspended": false,
  "accountCreatedAt": "ISO8601时间",
  "followersCount": 0,
  "followingCount": 0,
  "subscriberCount": 0,
  "memberCount": 0,
  "postCount": 0,
  "viewCount": 0,
  "recentPostSamples": ["该账号近期发帖文本样本，没有就传空数组，不是必填"]
}
```

`accountEntityType` 表示平台对象形态（用户/频道/主页/群组等），不等于最终账户类别判断结果。

#### 响应体

```json
{
  "schemaVersion": "t1_annotation_v0.6",
  "accountReference": {
    "platform": "x",
    "platformUserId": "回显请求里的原值",
    "accountEntityType": "user",
    "platformNativeType": "回显",
    "handle": "回显",
    "displayName": "回显"
  },
  "accountType": {
    "primaryAccountCategory": {
      "categoryLabel": "ordinary_user | news_media | state_affiliated_media | government_agency | political_actor | political_party_or_campaign | military_security_agency | international_organization | ngo_or_civil_society | academic_or_expert | commercial_brand | platform_official | influencer_kol | community_group | anonymous_account | suspected_bot_or_automated | unknown | other",
      "evidenceIds": ["ev_account_001"]
    },
    "accountSubtypeTags": [
      {"subtypeTag": "更细粒度的身份标签，受控字符串，非固定枚举", "evidenceIds": ["ev_account_001"]}
    ],
    "automationSuspicion": {
      "suspicionLevel": "none | low | medium | high | unclear",
      "evidenceIds": ["ev_account_002"]
    }
  },
  "evidenceClues": [
    {
      "evidenceId": "ev_account_001",
      "evidenceType": "profile_text | verification_info | account_metadata | activity_statistics | recent_post_sample | platform_label | other",
      "sourceField": "display_name | bio | self_declared_location | verified | verified_type | account_entity_type | platform_native_type | account_created_at | followers_count | following_count | subscriber_count | member_count | post_count | view_count | recent_post_sample | other",
      "metadataSnapshot": {"value": "支撑判断的最小原始值快照"}
    }
  ],
  "qualityControl": {
    "needHumanReview": false,
    "reviewReasons": [],
    "failedModules": []
  },
  "overallConfidence": 0.0,
  "processedAt": "ISO8601时间"
}
```

**`accountReference`**：回显请求里的关键身份字段，不复制完整账户资料，方便调用方核对这个结果对应哪个账号。

**`primaryAccountCategory` 选择原则**：优先依据主体身份而不是粉丝量/传播规模；优先选择更具体的类别；账号名称、简介、认证和平台标签应相互印证；资料不足用 `unknown`；身份明确但词表未覆盖用 `other`。

**`accountSubtypeTags[].subtypeTag`** 不是固定枚举，是受控字符串，没有可靠细分类时返回空数组。

**证据结构专用于账号场景，不通用**：`evidenceType`/`sourceField` 都是账号专属枚举，跟内容标注的 `text_span`/`image_region` 不是一回事；**账号证据没有 `evidenceText` 字段**，所有原始值统一放 `metadataSnapshot`，统计类证据可以同时给 `rawValues`（原始值）和 `derivedValues`（派生计算值，比如日均发帖数）。

**该接口不设置 `accountTypeConfidence` 或各子标签各自的置信度**，顶层 `overallConfidence` 统一表示对 `primaryAccountCategory`+`accountSubtypeTags`+`automationSuspicion` 这一整套结果的总体确信程度。

`qualityControl.reviewReasons`/`failedModules` 是账号场景专用枚举（`insufficient_profile_information`/`conflicting_identity_signals`/`ambiguous_account_category`/`verification_conflict`/`insufficient_activity_data`/`insufficient_post_samples`/`automation_judgment_uncertain`等），跟内容标注接口不通用。

## 四、事件热度判断接口

### POST（action=`annotate_event_heat`）

**用途**：判断图谱里某个 `Event` 实体当前的综合热度。输入不是某一条内容，而是"事件基础信息 + 图谱里跟它关联的一跳实体和抽样内容 + 后端算好的聚合统计"。接口不负责事件识别、实体抽取、内容关联和统计聚合，这些都是后端已经做好才传进来的。

**触发时机**：不是实时触发，由后端定时任务 `EventHeatJob` 周期性扫描需要（重新）计算热度的事件，批量调用。

#### 请求体

```json
{
  "event": {
    "eventId": "事件唯一标识",
    "canonicalName": "事件标准名称",
    "eventType": "election | military | diplomatic | protest | disaster | other",
    "occurredAtStart": "事件开始时间，ISO8601",
    "occurredAtEnd": "事件结束时间，ISO8601",
    "country": "主要关联国家/地区"
  },
  "relatedEntities": [
    {
      "entityId": "关联实体ID",
      "entityType": "person | organization | location | media_content | social_account",
      "name": "实体名称（entityType=media_content时此字段为空）",
      "publishedAt": "仅entityType=media_content时有值",
      "platform": "仅entityType=media_content时有值",
      "likeCount": 0,
      "commentCount": 0,
      "shareCount": 0,
      "repostCount": 0,
      "viewCount": 0
    }
  ],
  "aggregateStats": {
    "totalRelatedContentCount": 0,
    "totalEngagement": 0,
    "distinctPlatformCount": 0,
    "earliestContentAt": "最早关联内容时间",
    "latestContentAt": "最新关联内容时间"
  }
}
```

`event.occurredAtStart`/`occurredAtEnd` 是事件本身何时发生，`aggregateStats.earliestContentAt`/`latestContentAt` 是系统关联到的内容何时发布，两者不是一回事。`relatedEntities` 里的 `media_content` 条目是后端按互动量抽样出来的辅助样本，事件关联内容的真实总规模以 `aggregateStats.totalRelatedContentCount` 为准，不要用 `relatedEntities` 里 media_content 条目的数量当总数。

#### 响应体

```json
{
  "schemaVersion": "t1_annotation_v0.6",
  "eventHeat": {
    "heatLevel": "low | medium | high | explosive | unclear",
    "heatScore": 0.0,
    "heatSignalTypes": ["content_volume | engagement_surge | rapid_growth | wide_platform_spread | sustained_attention | declining | insufficient_data | unclear"],
    "reasoning": "面向系统展示和人工复核的简短依据摘要，不记录模型内部推理过程"
  },
  "overallConfidence": 0.0,
  "processedAt": "ISO8601时间"
}
```

**响应不重复返回事件引用对象**，调用方（后端）通过请求上下文中的 `event.eventId` 关联结果，不需要响应体回显。`heatScore` 取值 `0.0-1.0`，表示事件当前有多热；`overallConfidence` 表示对整套热度判断结果的确信程度，两者含义不同，不要混用。`eventHeat` 内部不单独设置置信度字段。

`heatLevel=explosive` 必须体现短时间内的快速增长或互动激增，不能仅依据历史累计总量较高。`heatSignalTypes` 里 `engagement_surge`/`rapid_growth`/`declining` 应该有不同时间窗口的对比数据支撑，只有累计总量时不应直接输出这几个。

**没有任何关联内容时的标准输出**：

```json
{
  "heatLevel": "unclear",
  "heatScore": null,
  "heatSignalTypes": ["insufficient_data"],
  "reasoning": "当前事件尚未关联到有效内容，无法形成可靠的事件热度判断。"
}
```

## 五、三接口字段边界

| 内容 | 所属接口 |
|---|---|
| 单条内容的意识形态、核心立场、情绪和语言风格 | `annotate` |
| 单条内容的BEND方法（`manipulationMethod`）和风险等级 | `annotate` |
| 单条内容的AIGC检测 | `annotate` |
| 账户的主要类别和细分类标签 | `annotate_account` |
| 账户的自动化嫌疑 | `annotate_account` |
| 已聚合事件的整体热度 | `annotate_event_heat` |

边界原则：不用单条贴文代替账户主页快照做账户类别判断；不在单条内容首次进入系统时直接输出事件级热度；`entitiesHint` 只提供实体线索，不替代正式实体抽取（正式抽取由 T2 负责，见《T2抽取接口规约》）；AIGC检测与内容风险判断相互独立；事件热度基于聚合数据，不基于单条内容的互动量直接判断。

---

## 六、通用规则

- **`score` 与 `confidence`**：两者都用 `0.0-1.0`，但含义不同——`score` 表示目标属性的程度（比如"这条内容有多像AI生成"），`confidence` 表示模型对判断结果的确信程度。无法计算或无法形成有效判断时用 `null`。
- **空数组 `[]`**：表示模块正常执行，但没有相应结果（比如没有识别到明确的BEND手法）。
- **`unclear`**：任务适用于当前输入，但信息不足/信号冲突/上下文不完整，无法可靠判断。
- **`not_applicable`**：当前输入不存在对应模态，或该字段对当前输入不适用（比如纯文本输入里 `videoAigcLabel` 应该是 `not_applicable`），这不算模块失败。
- **`other` 与 `unknown`**：`other` 表示已经能判断对象性质，但固定词表没有对应类别；`unknown` 表示信息不足，无法判断对象类别。
- **证据引用**：标注字段的 `evidenceIds` 必须能在同一响应的 `evidenceClues` 里找到，不同标签可以复用同一条证据。
- 所有枚举字段只能使用规定的固定值；非固定字段（如 `accountSubtypeTags[].subtypeTag`）用项目维护的受控字符串；时间字段统一 ISO 8601；不要在响应里增加未定义的同义置信度字段；不要把 `not_applicable` 写进 `failedModules`；不要把模型生成的解释当作原始证据。

## 七、后端消费 T1 结果的方式（供理解，不要求算法组处理）

- `annotate` 的完整响应体，后端整段存成 JSON（PG `media_contents.t1_annotation`、Neo4j `MediaContent.t1Annotation`），不逐字段拆解落库。
- `annotate_account` 的结果写入 `social_accounts.account_type`/`account_type_confidence`（取 `accountType.primaryAccountCategory.categoryLabel` 和顶层 `overallConfidence`）。
- `annotate_event_heat` 的结果写入 `events.event_heat_level`/`event_heat_score`/`event_heat_confidence`（取顶层 `overallConfidence`）等字段，以及 Neo4j `Event` 节点属性。
