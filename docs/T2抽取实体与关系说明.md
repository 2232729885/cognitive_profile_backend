# T2 需要抽取的实体与关系
# 课题四 × T2 算法组

> 本文档讲**抽取什么**（实体类别、关系类型、判断标准），接口怎么调（请求/响应格式、字段类型）见《T2抽取接口规约.md》，两份文档配合看。

---

## 一、实体抽取

### 1.1 图谱基础类型（`entities[].type`，只有这4种）

```
person | organization | event | location
```

**没有 `narrative`（叙事）类型**——叙事、议题、主张、口号这类对象边界不稳定、上下文依赖强（同一句话可能是观点、也可能是宣传语、也可能是谈判立场，很难靠局部文本稳定归类），不作为 T2 的原子实体抽取。这类信息如果需要体现，通过 `keywords`/`topicTags`（T1标注维度）或者内容摘要承载，不要在 T2 这边输出成实体。

### 1.2 为什么只有4种基础类型，但任务书要求"不少于10类认知领域实体"

任务书要求的"10类认知领域实体"（境外政要、关键意见领袖、国家机构、媒体机构……）是**认知领域类别标签**，不等于要建10个平行的顶层实体类型。如果每个类别都建一个顶层类，下游"只要查所有人物"这种最基本的调用都会变得很麻烦。

所以采用"两层口径"：
- **图谱基础类型**（`type` 字段）：`person`/`organization`/`event`/`location`，保持稳定、少而清楚
- **认知领域类别**（`attributes` 里的细分标签）：落在基础类型之上，通过 `attributes.personCategories`/`attributes.organizationCategories` 表达

也就是说，任务书要的"境外政要""关键意见领袖"这些，**不是** `type` 字段的取值，`type` 永远是 `person`；类别信息放在 `attributes` 里。

### 1.3 `person` 类型的认知领域细分（`attributes.personCategories`，字符串数组，允许多标签）

一个人物可以同时属于多个类别（比如某学者既是"领域专家"又是"关键意见领袖"），所以这是**数组**，不是单选。

| 枚举值 | 中文名 | 说明 | 规模参考 |
|---|---|---|---|
| `foreign_public_official` | 境外政要 | 在境外政府、军队、政党、国际组织等公共权力体系中具有明确职务、且与关注议题直接相关的人物。总统、总理、部长、议员、外交官、军方负责人、政府发言人等 | 不少于1万个 |
| `key_influencer` | 关键意见领袖 | 在特定议题中具有显著传播影响力、受众号召力或议程带动能力的人物。知名评论员、大V、头部博主、强影响力自媒体人物等 | 不少于2万个 |
| `domain_expert` | 领域专家 | 在特定专业领域具有解释权、专业背书能力或政策分析能力的人物。学者、研究员、智库专家、行业分析师等。可以和"关键意见领袖"重叠 | |
| `media_person` | 媒体人物 | 以新闻主持、采访报道、评论分析、节目表达为主要公众角色的人物。主持人、记者、评论员等。**注意**：主持人本人是 `person`，他所属的媒体机构是 `organization`，不要混在一起 | |
| `business_figure` | 商业人物 | 在企业、金融机构、商业集团等盈利组织中具有代表性、决策力或公众影响力的人物。创始人、CEO、董事长、高管、投资人等 | |
| `civic_actor` | 社会活动人物 | 在公益、倡议、社会运动、非政府行动或国际公共事务中具有代表性的人物。NGO发起人、社会运动组织者等。**注意**：组织本身是 `organization`，发起人/代表人是 `person` | |
| `active_netizen` | 活跃网民 | 长期活跃于社交媒体、论坛、评论区等线上空间，持续发布/转发/评论的普通或半职业化网民。强调"活跃行为"本身，不预设影响力或身份 | 不少于1000万个 |
| `general_person` | 未明确类别人物 | 能确认是具体自然人，但暂时无法判断公共角色/职业/类别的兜底类别。**信息不够就用这个，不要为了贴标签过度猜测** | |

### 1.4 `organization` 类型的认知领域细分（`attributes.organizationCategories`，字符串数组）

| 枚举值 | 中文名 | 说明 |
|---|---|---|
| `state_institution` | 国家机构 | 代表国家权力、公共治理、军事安全、外交事务或执法体系的正式机构。政府部门、外交部、国防部、军队、情报机构、使领馆等 |
| `political_organization` | 政治组织 | 围绕政治竞争、政策主张、意识形态传播或社会动员建立的政治性组织。政党、竞选团队、政治行动委员会、政治联盟等 |
| `social_organization` | 社会组织 | 不直接属于国家机关或政党，以研究、倡议、公益、行业协作或国际合作为目的。NGO、基金会、智库、协会等 |
| `media_organization` | 媒体机构 | 以新闻采编、内容发布、评论传播或信息分发为主要职能。通讯社、报社、电视台、新闻网站、媒体栏目等 |
| `enterprise` | 企业机构 | 以营利、商业经营、资本运作、产品服务供给为主要目的。苹果、花旗银行、OpenAI、亚马逊、谷歌、X平台、TikTok等都属于这类。**平台公司只是企业机构的一个子类，不单独作为类别** |

### 1.5 `event`/`location` 类型

- `event`：`attributes` 里带 `eventType`（`politics`/`military`/`economy`/`society`/`culture`/`science_tech`/`security`/`nature`/`other`）和 `eventTimeStart`，具体枚举见《T2抽取接口规约.md》。
- `location`：不需要额外的 `attributes` 细分类别，`canonicalName` 给标准地名即可。

### 1.6 不属于 T2 抽取范围的对象

- **叙事/议题/主张/口号**：不抽取成实体（见1.1）。
- **账号/频道/社群**：`SocialAccount` 是图谱基础类之一，但**不是 T2 从文本里抽取的**——账号节点由账号摄入流程（课题三 `social_account` 消息）直接产生，账号的类别判断（个人/媒体/政府等18类）由 T1 `annotate_account` 接口判断，不需要 T2 在正文里识别"这条内容提到了哪个账号"。文本里 `@某账号` 这类提及，由内容摄入时的 `mentions` 字段处理，不经过 T2 实体抽取这条路径。

## 二、关系抽取

### 2.1 可以使用的16个关系类型

完整头尾实体类型定义、方向约定见 `docs/关系词表与头尾实体类型说明.md`（唯一权威来源），这里只列一遍名字：

```
账号关联类：HAS_ACCOUNT
归属组织类：BELONGS_TO、PART_OF
内容链路类：PUBLISHED_BY、REPLY_TO、REPOSTS、MENTIONS
事件地点类：DESCRIBES、EVENT_OCCURRED_AT、EVENT_INVOLVES_ENTITY、LOCATED_IN
立场认知类：SUPPORTS、OPPOSES、QUESTIONS、INCITES、DE_ESCALATES
```

**`predicate` 必须是这16个之一，不在词表内的会被后端过滤掉，等于白抽。** 也不要按训练数据里的记忆去用 `AFFILIATED_WITH`/`OWNS`/`MEMBER_OF`/`AUTHORED`/`COORDINATES_WITH` 这些已经废弃的旧关系名，这些都已经从词表里删除了。

### 2.2 哪些关系需要 T2 自己判断输出，哪些不需要

`HAS_ACCOUNT`/`PUBLISHED_BY`/`REPLY_TO`/`REPOSTS`（内容和它作者/上级内容的关系）、`DESCRIBES`/`MENTIONS`（内容和它提到的实体之间的关系）**这几个后端会自动补写，不需要 T2 在 `relations[]` 里输出**——后端已经知道"这条内容的作者是谁""这条内容回复/转发了谁""这条内容成功解析出了哪些实体"，重复输出也没关系（会被去重），但不是必须的。

**T2 真正需要重点判断的是这6类"认知关系"**（对应任务书要求的核心关系能力），下面逐个说判断标准：

### 2.3 `SUPPORTS`（支持背书）

主体 A 对主体 B、事件 B 或主张 B 表达**正向**立场、认可、声援、背书、帮助或维护。

典型表现：某组织支持某政策、某意见领袖声援某群体、某媒体认可某主张、某政党为某行动提供政治背书。

**边界：普通转发或提及不算支持背书**，必须有明确的正向态度或帮助行为才判定为 `SUPPORTS`。

### 2.4 `OPPOSES`（反对批评）

主体 A 对主体 B、事件 B 或主张 B 表达**负向**立场、批评、质疑、否定、谴责、抵制或驳斥。

典型表现：某政府谴责某行动、某媒体批评某政策、某群体抵制某组织。

**边界：文本整体情绪负面不等于存在反对关系**，必须确认负面态度指向的具体对象才判定为 `OPPOSES`。

### 2.5 `QUESTIONS`（质疑）

主体 A 对主体 B、事件 B 或相关说法表达怀疑、追问、挑战、求证、削弱可信度或提出不确定性判断——**跟 `OPPOSES` 的区别是没有明确否定/谴责，只是提出怀疑或要求进一步证实**。

### 2.6 `INCITES`（煽动放大）

主体 A 对事件 B、叙事 B、内容 B 或群体情绪 B 进行**动员、鼓动、扩散、放大、激化**或推动传播。

典型表现：号召大量转发、集中传播同一口号、放大争议性内容、鼓动群体行动、利用情绪化表达推动议题升温。

**边界：单次普通转载通常只说明内容传播，不必然构成煽动放大**，需要有动员、扩散、激化或策略性放大的证据。

### 2.7 `DE_ESCALATES`（平息缓和）

主体 A 对冲突 B、争议 B、恐慌 B 或升级风险 B 进行**降温、澄清、调停、事实核查、缓和或阻断扩散**。

典型表现：呼吁冷静、推动停火谈判、澄清虚假消息、发布事实核查、调停冲突双方。

**边界：单纯反对某一叙事不等于平息缓和**，只有表达或行动的作用是降低冲突、减少误导、缓和情绪或阻止升级时，才判定为 `DE_ESCALATES`。

### 2.8 `EVENT_INVOLVES_ENTITY`（事件涉及实体，承担"影响"关系的表达角色）

表示某个事件涉及某个人物、组织或账号作为参与方、对象或受影响方。**这个关系在当前口径下同时承担"主体 A 影响/引导主体 B"这种更广泛的"影响"语义**，不单独恢复 `INFLUENCES` 这个关系类型。

典型表现：某政要讲话影响公众态度、某专家评论引导媒体框架、某组织倡议推动群体行动——这类"影响引导"关系统一落在 `EVENT_INVOLVES_ENTITY` 上表达，不要输出成词表外的关系名。

**边界：仅仅共同出现、共同参与某事件，不能直接判定为有影响关系**，必须有明确的影响方向或作用描述才输出这条关系。

### 2.9 `BELONGS_TO`/`PART_OF`（组织关联与从属控制）

表示主体 A 与主体 B 之间存在明确的组织归属、成员身份、管理控制、资助支持、账号归属或上下级从属关系——这是**相对稳定的结构性连接**，不是某一次发声、转发或立场表达。

典型表现：某人物任职于某机构（`Person -> BELONGS_TO -> Organization`）、某组织受另一组织资助或控制、某下属机构接受上级机构指挥（`Organization -> PART_OF -> Organization`）。

**边界：支持某一观点、共同参与某一事件或转发同一内容，都不能直接判定为组织关联**，必须有任职、隶属、管理、运营、资助、控制等明确证据。

## 三、小结：一条内容大概要抽取成什么样子

```json
{
  "entities": [
    {
      "mentionId": "m1",
      "name": "某某",
      "canonicalName": "某某",
      "type": "person",
      "attributes": { "personCategories": ["key_influencer", "domain_expert"] }
    },
    {
      "mentionId": "m2",
      "name": "某某智库",
      "canonicalName": "某某智库",
      "type": "organization",
      "attributes": { "organizationCategories": ["social_organization"] }
    }
  ],
  "relations": [
    { "subjectMentionId": "m1", "predicate": "BELONGS_TO", "objectMentionId": "m2" },
    { "subjectMentionId": "m1", "predicate": "SUPPORTS", "objectMentionId": "..." }
  ]
}
```

字段的完整类型定义、`span`/`evidence` 这些细节，见《T2抽取接口规约.md》。
