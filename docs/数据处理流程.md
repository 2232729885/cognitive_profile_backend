## 一、总体说明

### 1.1 六种 record_type 的处理路径

| record_type | Kafka Topic | 是否走 T1-T4 | 终态 |
|---|---|---|---|
| `social_content` | kt3.social_content | ✅ 完整走 T1→T2→T3→T4 | T4_INDEXED |
| `news_article` | kt3.news_article | ✅ 完整走 T1→T2→T3→T4 | T4_INDEXED |
| `social_account` | kt3.social_account | ❌ 只做标准化映射 | NORMALIZED |
| `account_relation` | kt3.account_relation | ❌ 只做标准化映射 | NORMALIZED（后续有独立的UUID回填批处理） |
| `media_asset` | kt3.media_asset | ❌ 只做标准化映射 | NORMALIZED |
| `collection_task` | kt3.collection_task | ❌ 只做标准化映射 | NORMALIZED |

### 1.2 pipeline_status 状态机

```
RECEIVED → NORMALIZED → T1_DONE → T2_DONE → T4_INDEXED
                                                    ↑
                                                 终态（social_content / news_article）

RECEIVED → NORMALIZED
                ↑
             终态（其余四种类型）

任意步骤失败 → FAILED（重试超3次后）
```

**没有单独的 `T3_DONE`**——`PipelineStatus` 枚举里还留着这个值，但实际代码从来没有任何地方设置过它，是历史遗留。原因见 1.3 第4条：T3（`resolve_batch`）不是一个独立的流水线步骤，是嵌在 T2 这一步内部同步调用的，`T2ExtractionStep` 处理完（含调用T3判断每个实体）之后直接把状态设成 `T2_DONE`，下一步就是 `T4_INDEXED`。

### 1.3 核心设计原则

**1. raw_records 必须第一个写。**
不管后续 T1-T4 是否成功，原始消息必须先持久化。任何步骤失败都可以从 raw_records 重跑，不需要重消费 Kafka。

**2. 每步 T 的完整输出存 raw_records 的 JSONB 字段。**
t1_output / t2_output / t3_output / t4_output 四个 JSONB 字段存对应步骤的完整响应，单行即可追溯全部处理过程：
```sql
SELECT t1_output, t2_output, t3_output, t4_output
FROM raw_records
WHERE source_record_id = 'x_post_123456';
```

**3. 平台原始 ID 先存，UUID FK 后回填。**
account_relations、media_contents.author_account_id 等字段初始为 null 或存字符串，不要求入库时立刻匹配 UUID 外键。专门的回填批处理负责后续补齐。这样不管课题三数据到达顺序如何，都不会因 FK 约束阻塞入库。

**4. T3 不是独立步骤，嵌在 T2 内部同步调用；T4 与 T2/T3 这一整块可以并行。**
T2 输出的是 mention 级实体（还没判断是不是图谱里已有的实体），`T2ExtractionStep` 在同一步内接着做：候选召回（ES+Milvus） → 调 T3 `resolve_batch` 判断每个 mention 该合并/新建/转审核 → 按结果写 PG/Neo4j/ES/Milvus。这一整块完成后状态直接置为 `T2_DONE`，不会经过 `T3_DONE`。T4 只依赖 body_text，不依赖 T2/T3 的实体结果，可以和这一整块并行触发，节省处理时间。

**5. 实体融合改为后台定时批量处理。**
T2 每次识别到实体都直接 INSERT 新记录，`dedup_status='pending'`。T3 只负责写入本次 Neo4j 节点和关系，不再实时合并 PG 实体记录；`merge_history` 由 EntityDeduplicationJob 融合时记录。

**6. 画像由定时任务批量生成，不做差量补全。**
T5不再是"用户查询时按需触发补全"，而是定时任务按 importance_score 排序，批量调用T5为每个实体一次性生成全量15维度画像，写入新版本，旧版本自动归档。

## 二、social_content 处理流程

**适用范围：** 主贴、评论、回复、转发、引用、频道消息、论坛帖子等平台内容对象。
**处理路径：** 走完整 T1→T2→T3→T4 四步流水线。

---

### Step 1：Schema 校验 + 去重检查

**执行方：** data-ingestion-service（接入适配器）
**操作：** 纯内存，不写任何表

**校验内容：**
- 必填字段不为空：`record_type`、`source_record_id`、`payload_hash`、`platform`、`data.platform_content_id`、`data.author_platform_user_id`
- 字段类型正确：时间字段为 ISO-8601 格式、数值字段非负
- `schema_version` 为已知版本（kt3_to_kt4_v1）

**去重查询：**
```sql
-- 任意一个命中即为重复，直接 ack 跳过
SELECT 1 FROM raw_records WHERE source_record_id = ? LIMIT 1;
SELECT 1 FROM raw_records WHERE payload_hash = ? LIMIT 1;
```

**校验失败处理：** 写错误日志，消息 ack，不进入后续流程，不重试。
**重复处理：** 直接 ack 跳过，不报错。

### Step 5：T1 自动标注

**执行方：** pipeline-service → agent-proxy-service → T1 Agent（HTTP 调用，`action=annotate`）
**输入：** `title`/`text`（`media_contents.title`/`body_text`）+ `medias`（关联的图片/视频，取自 `media_asset_ids` 已经落库的部分）+ `context`（`contentId`/`platform`/`url`/`contentType`/`authorHandle`/`publishedAt`/`hashtags`/互动数据/`parentContentId`）
**输出：** 当前规约 `T1_annotation_v0.6`（`docs/T1标注接口规约.md`）——AIGC检测 + 6个高价值主观维度（ideology/coreStance/opinionEmotion/languageStyle/manipulationMethod/riskLevel）+ 5个基础客观维度（topicTags/entitiesHint/keywords/summary/eventType）+ 证据池 + 质量控制

**写表 1：** `media_contents`（UPDATE）

| 字段 | 值 | 说明 |
|------|-----|------|
| t1_annotation | T1 完整响应体（JSONB） | 整段存，不逐字段拆解——标注维度以后再调整，不需要改表结构 |
| t1_annotated_at | NOW() | |

**写表 2：** `raw_records`（UPDATE）

| 字段 | 值 |
|------|-----|
| t1_output | T1 Agent 完整响应（JSONB），跟 `media_contents.t1_annotation` 内容一样，这里额外存一份用于追溯 |
| pipeline_status | T1_DONE |

**写表 3：** `pipeline_tasks`（UPDATE）

| 字段 | 值 |
|------|-----|
| t1_status | done |
| t1_started_at / t1_done_at / t1_duration_ms | 调用耗时，监控用 |

**状态变化：** `NORMALIZED → T1_DONE`

**失败处理：** t1_status = failed，retry_count + 1，指数退避重试，超过 `max_retries`（默认3）后 `pipeline_status = FAILED`，推死信队列，不影响其他消息。

**图片/视频晚到的情况：** `medias` 依赖内容摄入时已经关联好的 `media_asset_ids`，但图片经常比文字消息晚到（分开的 Kafka topic）。如果 T1 跑的时候图片还没关联上，只能先按纯文本标注；`ImageEmbeddingJob`（定时任务）会扫描后来才关联上媒体的内容，重新调用一次 `annotate` 补齐图文标注，同时兼做图片向量化（T4的部分工作）。

### Step 6：T2 信息抽取 + 候选召回 + T3 实体消歧（同一步内完成，不是三个独立步骤）

**执行方：** pipeline-service → agent-proxy-service → T2 Agent（`action=extract_entities`），随后在**同一次** `T2ExtractionStep` 执行过程中，后端自己做候选召回，再调 T3 Agent（`action=resolve_batch`）。**这里没有独立的"T3步骤"**，`resolve_batch` 是被 `EntityResolutionService` 当作一个内部子调用同步串起来的，跟 T2 共享同一个 `pipeline_status`（T2_DONE），不会经过旧版本文档里说的 `T3_DONE`。

**输入：** `title`/`text` + `t1Annotation`（T1 `annotate_content` 的完整业务响应体，包含 `aigcDetection`/`annotations`/`evidenceClues`，供参考）+ `context`（含传播链字段：`parentContentId`/`repostOfContentId`/`quotedContentId`/`hashtags`/`mentions`）

**输出：** mention 级实体和关系（不是最终图谱ID，`mentionId` 是这次响应内的临时ID）

#### 6.1 T2 抽取（算法侧）

T2 只负责"这段文字里出现了什么实体、它们之间什么关系"，`entities[]` 每个实体带 `mentionId`/`canonicalName`/`type`(`person`/`organization`/`event`/`location`，没有`narrative`)/`span`/`aliases`/`attributes`；`relations[]` 引用 `mentionId` 表达关系，`predicate` 必须是 `docs/关系词表与头尾实体类型说明.md` 定义的16个类型之一，不在词表内的会被后端过滤掉。

#### 6.2 候选召回（后端，`EntityCandidateRetrievalService`）

针对 T2 输出的每个 mention，用 ES 名称/别名模糊匹配 + Milvus 向量ANN两路召回，合并去重返回 TopK 候选（图谱里可能匹配的已有实体）。

#### 6.3 T3 判断（`EntityResolutionService` 调 `resolve_batch`）

把每个 mention + 它的候选列表 + 上下文一起发给 T3，T3 判断该 `MERGE`（合并进已有实体）/`REVIEW`（转人工审核）/`CREATE`（新建），阈值默认 0.9/0.6（`autoMergeThreshold`/`reviewThreshold`，随请求传入，不写死在算法侧）。

**写表 1：** `persons` / `organizations` / `events`（`CREATE` 时 INSERT，`dedup_status='pending'`；`location` 类型不落PG只写Neo4j）

| 字段 | 值 | 说明 |
|------|-----|------|
| id | gen_random_uuid() | **注意：每次CREATE都是全新随机UUID，不做同名去重检查**，跨批次的同名归一由后台 `EntityDeduplicationJob` 定时批量处理，短期内PG可能有同名多条pending记录，这是设计如此，不是bug |
| canonical_name | 拜登 | |
| importance_score | 85.0 | |
| content_count | 1 | |
| first_seen_at / last_seen_at | NOW() | |

**写表 2：** `entity_fusion_records`（`MERGE`/`REVIEW` 时 INSERT一条记录，追溯这次判断依据）

| 字段 | 值 |
|------|-----|
| match_method / match_score / resolver_model | T3响应体里的 `matchMethod`/`score`/`modelVersion` |
| is_auto_merged | `MERGE`时true（已自动合并），`REVIEW`时false（待前端DedupView人工审核） |

**写表 3：** `Neo4j`（`CREATE`/`MERGE` 都会涉及节点写入或复用，用确定性UUID `stableUuid(type+canonicalName)` 天然去重）

```cypher
-- 新建或复用实体节点
MERGE (p:Person {id: $stableUuid}) SET p.canonicalName = $name, p.type = 'person'

-- 内容与解析出的实体的关系，后端自动补写，不需要T2在relations[]里额外输出
MERGE (c:MediaContent {id: $contentId})-[:MENTIONS]->(p)          -- person/organization/location走MENTIONS
MERGE (c:MediaContent {id: $contentId})-[:DESCRIBES]->(e:Event)   -- event走DESCRIBES

-- T2输出的relations[]（predicate在16个词表内的）
MERGE (subject)-[:EVENT_INVOLVES_ENTITY]->(object)   -- 示例，实际按T2给的predicate

-- 传播链关系（从media_contents的parent_content_id/repost_of_content_id推导，不是T2输出的）
MERGE (c:MediaContent)-[:REPLY_TO]->(parent:MediaContent)
MERGE (c:MediaContent)-[:REPOSTS]->(original:MediaContent)
```

**传播链目标内容晚到怎么办：** 如果 `parent_content_id`/`repost_of_content_id` 指向的内容这时候还没入库，这次先跳过，`media_contents.propagation_synced_to_neo4j` 保持 `false`；`ContentPropagationBackfillJob`（定时任务）扫描这些还没同步的记录重新尝试。

**写表 4：** `social_accounts`（UPDATE，回填内容作者关联；这是内容摄入时`upsertAuthorAccount`已经做的最小信息账号，跟T2抽取无关，这里不重复处理）

**写表 5：** `raw_records`（UPDATE）：`t2_output` 存T2原始响应；`pipeline_status = T2_DONE`

**写表 6：** `pipeline_tasks`（UPDATE）：`t2_status = done`，`t2_duration_ms`

**状态变化：** `T1_DONE → T2_DONE`（**直接到这里，没有 `T3_DONE` 这个中间态**）

### Step 7：索引构建（与 Step 6 这一整块并行触发）

**执行方：** pipeline-service → 后端内部 `EmbeddingService` / `MilvusVectorService` / Elasticsearch
**输入：** `T4IndexingStep.buildContentEmbeddingText()` 拼接出的多字段文本（不依赖 Step 6 的实体抽取结果，可以和 Step 6 同时触发）
**输出：** 原文文本向量、英文 pivot translation 文本向量、ES 文档

**写表 1：** `Milvus · media_content_embeddings`（UPSERT）—— `id`/`content_id`(对应`media_contents.id`)/`platform`/`language`/`content_type`/`published_at`/`title_embedding`/`summary_embedding`/`body_embedding`(4096维)，保存原文向量。

**写表 2：** `Milvus · media_content_pivot_embeddings`（UPSERT）—— 字段结构同 `media_content_embeddings`，但 `title_embedding`/`summary_embedding`/`body_embedding` 来自英文 pivot translation，用于跨语种语义召回。翻译失败时本条 pivot 向量跳过。

**写表 3：** `media_contents`（UPDATE）—— T4 会把标题、正文和 T1 摘要的英文 pivot translation 回写到 `translated_title`/`translated_body_text`/`translated_summary`，用于后续重建 ES 或 Milvus pivot embedding。

**写表 4：** `Elasticsearch · media_contents_index`（INDEX）—— 字段列表和来源见 `docs/数据库设计.md` 三、ES索引设计。`topic_category`/`sentiment_label`/`stance_label`/`aigc_score`/`aigc_type` 这几个字段是索引时用 `T1AnnotationView` 实时解析 `media_contents.t1_annotation` 抽出来的，不是从 PG 专门列取的（那几个 PG 列已经废弃删除了）。跨语种检索字段包括 `translated_title`/`translated_body_text`/`translated_summary`。

**写表 5：** `raw_records`（UPDATE）：`t4_output` 存 `{"textVectorId":"...","pivotTextVectorId":"...","textEmbeddingSkipped":false,"pivotTextEmbeddingSkipped":false}`；`pipeline_status = T4_INDEXED`

**写表 6：** `pipeline_tasks`（UPDATE）：`t4_status = done`，`t4_done_at`，`status = DONE`

图片/视频向量不在这一步里同步生成，`media_asset` 摄入和定时 `ImageEmbeddingJob` 会调用后端内部 `EmbeddingService` 写入 `media_asset_embeddings`；其中 OCR/ASR/caption 还会生成英文 pivot translation，先回写 PG `media_assets.translated_ocr_text`/`translated_asr_text`/`translated_caption_text`，再写入 `media_asset_pivot_embeddings`，并同步写入 ES 字段 `translated_ocr_text`/`translated_asr_text`/`translated_caption_text`。

**状态变化：** `T2_DONE → T4_INDEXED`（最终态）

### Step 4：为什么 from_account_id / to_account_id 会是空的

**根本原因：Kafka 消息到达顺序不保证。**

`account_relation` 消息完全可能先于对应账号的 `social_account` 消息到达，或者目标账号课题三还没采集到（比如是个私密账号）。

如果入库时强行要求这两个字段必须是合法的 `social_accounts.id` 外键，会出现两种问题：要么入库直接失败，要么必须等账号数据到齐才能写关系，两种都会阻塞数据流。

**所以设计上选择：关系先存下来，外键留空，账号数据到位后用独立批处理回填。**

---

### Step 5：UUID 回填批处理（独立任务，不在入库主流程内）

**触发方式：** 两种方式任选其一
- **方式A：** 跟随 T3 节奏。T3 本来就在批量处理实体，每轮 T3 顺带跑一遍回填 SQL
- **方式B：** 独立定时任务，比如每 5 分钟跑一次，专门负责 UUID 对齐，和 T3 解耦

**写表 1：** `account_relations`（UPDATE，增量执行，只处理还没回填的记录）

```sql
-- 回填 from_account_id：按 (platform, platform_user_id) 匹配
UPDATE account_relations ar
SET from_account_id = sa.id
FROM social_accounts sa
WHERE ar.from_account_id IS NULL
  AND sa.platform = ar.platform
  AND sa.platform_user_id = ar.source_platform_user_id;

-- 回填 to_account_id
UPDATE account_relations ar
SET to_account_id = sa.id
FROM social_accounts sa
WHERE ar.to_account_id IS NULL
  AND sa.platform = ar.platform
  AND sa.platform_user_id = ar.target_platform_user_id;
```

**写表 2：** `Neo4j`（写入关系边，仅处理刚回填完成、两端UUID都已就绪的记录）

```sql
-- 找出回填完成、可以写图谱的记录
SELECT id, from_account_id, to_account_id, relation_type, observed_at, source
FROM account_relations
WHERE from_account_id IS NOT NULL
  AND to_account_id IS NOT NULL;
  -- 实际实现建议加一个 synced_to_neo4j BOOLEAN 标记字段，避免重复写入
```

```cypher
-- following 单独映射成 FOLLOWS（不在16个关系词表内，是账号间的结构性关系，不受词表约束）
MATCH (a:SocialAccount {id: $from_id}), (b:SocialAccount {id: $to_id})
MERGE (a)-[:FOLLOWS {
    observedAt: datetime($observed_at),
    source: $source,
    confidence: 1.0
}]->(b)

-- subscribe / member_of / owner_of / creator_of / admin_of / moderator_of
-- 这6种全部统一收敛到 BELONGS_TO（对齐 docs/关系词表与头尾实体类型说明.md，
-- 不再使用 MEMBER_OF/ADMIN_OF/OWNS 这些已经删除的旧关系类型）
MERGE (a)-[:BELONGS_TO {
    observedAt: datetime($observed_at),
    source: $source,
    confidence: 1.0
}]->(b)
```

> **注意：** 这些来自课题三的事实关系（`FOLLOWS`/`BELONGS_TO`）和 T2/T3 从内容里抽取判断出来的关系是两类不同来源，`FOLLOWS`/`BELONGS_TO` 由 `AccountRelationBackfillJob` 按 `RelationType.java` 的映射直接写入，不经过 T2/T3 的 16 词表校验（`FOLLOWS` 本身也不在这16个词表内，是账号关系专属的结构性关系）。

---

### Step 6：无法回填的"孤儿关系"如何处理

如果某条关系的 `target_platform_user_id` 对应的账号课题三一直没有交付（比如私密群组、平台限制无法采集），这条关系的 `to_account_id` 会**永久为 NULL**。

这不影响入库状态——这条记录在 `NORMALIZED` 阶段已经算完成，只是暂时无法在 Neo4j 里建立完整的关系边。

**建议监控指标：**
```sql
-- 统计回填超过7天仍未完成的关系数量，超过阈值需要人工核实
SELECT relation_type, COUNT(*) AS orphan_count
FROM account_relations
WHERE (from_account_id IS NULL OR to_account_id IS NULL)
  AND created_at < NOW() - INTERVAL '7 days'
GROUP BY relation_type;
```

数量过多通常说明：要么课题三某些账号类型的数据确实没交付全，要么 `platform_user_id` 的命名规则在两类消息（social_account 和 account_relation）里不一致，需要联调时核对。

## 八、画像生成流程（定时任务，不属于入库流程）

**触发方式：** 定时任务（每日凌晨或按配置周期），不是用户触发，不做差量补全。

### Step 1：按重要性排序，选取待生成画像的实体

```sql
SELECT id, canonical_name, importance_score
FROM persons
WHERE id NOT IN (
    -- 排除7天内已生成过 active 画像的实体
    SELECT person_id FROM person_profiles
    WHERE status = 'active' AND generated_at > NOW() - INTERVAL '7 days'
)
ORDER BY importance_score DESC
LIMIT 100;  -- 每轮处理数量，避免一次性压垮 T5
```

**输入：** 该人物在 Neo4j 中的完整属性、关联的内容、关系网络
**输出：** 15 维度全量画像数据（一次性生成，非差量）

## 九、事件热度计算流程（定时任务，不属于入库流程）

**触发方式：** `EventHeatJob` 定时任务（30分钟一次），扫描 `dedup_status='canonical'` 的 `Event` 实体，按 `event_heat_computed_at` 升序取（没算过的排最前面优先处理，算过的里最早算的下次优先刷新，靠这个排序自然轮转，不需要额外判断"最近有没有变化"）。

**处理过程：**
1. 用 Neo4j 一跳查询（`findHopGraph`）取出这个 `Event` 关联的人物/组织/地点/账号/内容
2. `MediaContent` 类型的关联节点直接从 Neo4j 属性里拿互动数据（点赞/评论/转发等，已经在内容处理 Step 6 同步写入了），不用再回查 PG
3. 内容数量多时按互动量取 Top 20 采样，连同后端算好的聚合统计（关联内容总数、总互动量、平台覆盖数、时间跨度）一起发给 T1 `annotate_event_heat`
4. T1 判断结果写回 `events.event_heat_level`/`event_heat_score`/`event_heat_confidence`/`event_related_content_count`/`event_heat_computed_at`，以及 Neo4j `Event` 节点的 `eventHeatLevel`/`eventHeatScore` 属性

**为什么不在内容摄入时直接判断事件热度：** 单条内容的热度感知信息量有限，真正的事件热度需要图谱聚合了足够多关联内容才能判断准确，所以做成独立的事件级批量任务，不在 T1 `annotate` 接口里判断（`annotate` 现在已经不输出事件热度这个维度了）。

## 十、文件批量导入流程（不经过 Kafka）

**适用场景：** 课题三还没来得及通过 Kafka 推送，或者需要重放某一批历史数据时，前端"文件导入"直接上传 `zip`/`jsonl`/`json` 文件（`csv`/`excel` 暂不支持，格式不明确的列到字段映射容易做错，直接判失败不猜）。

**处理过程：**
1. 前端上传后，文件原样存入 MinIO（`batch-imports` bucket），创建一条 `status=PENDING` 的 `batch_import_tasks` 记录
2. `BatchImportJob`（定时任务，1分钟一次）扫描 PENDING 任务，下载文件，按格式解析：
    - `zip`：解压，`.jsonl`/`.json` 文件按行/按元素解析成一条条消息；如果 zip 里还带了媒体文件（`media/`目录），重新上传到我们自己的 MinIO（`media-assets` bucket），并把 `minio_bucket`/`minio_key` 注入对应 `media_asset` 消息里再转发
    - `jsonl`/`json`：直接按行/按数组元素解析
3. 每条解析出来的消息，按 `record_type` 转发到对应的 Kafka topic（`kt3.social_content`/`kt3.media_asset`等），走跟正常消息完全一样的消费者处理逻辑——**这个任务的职责到"正确转发到Kafka"为止**，不负责确保后续真的处理成功（那是消费者自己的去重/校验/落库逻辑，不重复做一套端到端追踪）
4. 处理完更新 `total_records`/`success_records`/`failed_records`，`status` 视情况变成 `DONE`/`PARTIAL`/`FAILED`

## 十一、失败重试机制

所有类型的处理失败，统一按以下策略处理：

| 场景 | 处理方式 |
|------|--------- |
| Schema 校验失败 | 写错误日志，消息 ack，不重试 |
| raw_records 写入失败 | 消息不 ack，Kafka 重新投递（最多3次） |
| L1 标准化写入失败 | pipeline_status = FAILED，写 error_message |
| T1 调用失败 | t1_status = failed，retry_count +1，指数退避重试，超3次推死信队列 |
| T2 调用失败（含内部的候选召回+T3判断） | t2_status = failed，retry_count +1，指数退避重试，超3次推死信队列。**没有单独的"T3步骤"可以重跑**，T3是T2这一步内部的子调用，失败了就整个T2这一步重来 |
| 索引构建/embedding 生成失败 | 标记对应索引步骤失败或跳过本次向量写入；支持从索引步骤或图片向量化任务重跑 |
| account_relations 回填长期未完成 | 不算失败，定期监控孤儿关系数量，超阈值人工核实 |
| T5 画像生成失败 | 记录失败实体，下一轮定时任务重新尝试，不影响其他实体 |

**重跑方式：**

```sql
-- 查找失败记录
SELECT id, source_record_id, error_step, error_message
FROM pipeline_tasks
WHERE status = 'FAILED'
ORDER BY created_at DESC;

-- 从指定步骤重跑（以 T2 为例，T3是T2内部子调用，没有单独的t3_status）
UPDATE pipeline_tasks
SET t2_status = 'pending', status = 'PENDING', retry_count = 0
WHERE id = ?;
-- pipeline-service 定时扫描 PENDING 状态的任务，自动重新触发
```

## 十二、数据追溯方式

### 12.1 查一条数据的完整处理过程

```sql
-- 单行查全部处理步骤（不需要跨表JOIN）
SELECT
    id,
    source_record_id,
    pipeline_status,
    t1_output,    -- T1 标注的完整输出
    t2_output,    -- T2 抽取的实体/关系（T3判断结果也在这一步内完成，没有单独的t3_output）
    t4_output     -- T4 写入的向量ID、ES文档ID
FROM raw_records
WHERE source_record_id = 'x_post_123456';
```

### 12.2 查一个采集批次的数据

```sql
-- 查某个采集任务的所有数据
SELECT r.source_record_id, r.pipeline_status, r.platform
FROM raw_records r
WHERE r.crawl_task_id = 'task_20260624_001'
ORDER BY r.created_at;

-- 查采集任务的覆盖情况
SELECT ct.platform, ct.target_languages, ct.records_collected,
       COUNT(r.id) AS actually_ingested
FROM collection_tasks ct
LEFT JOIN raw_records r ON r.crawl_task_id = ct.crawl_task_id
GROUP BY ct.id;
```

### 12.3 查流水线各步骤性能

```sql
SELECT
    AVG(t1_duration_ms) AS avg_t1_ms,
    AVG(t2_duration_ms) AS avg_t2_ms,
    AVG(t4_duration_ms) AS avg_t4_ms,
    COUNT(*) FILTER (WHERE status = 'DONE')   AS done,
    COUNT(*) FILTER (WHERE status = 'FAILED') AS failed
FROM pipeline_tasks
WHERE created_at > NOW() - INTERVAL '24 hours';
```

### 12.4 查 account_relations 的回填进度

```sql
SELECT
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE from_account_id IS NOT NULL AND to_account_id IS NOT NULL) AS fully_linked,
    COUNT(*) FILTER (WHERE from_account_id IS NULL OR to_account_id IS NULL) AS pending_or_orphan
FROM account_relations;
```
