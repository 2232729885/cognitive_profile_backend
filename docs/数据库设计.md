# 课题四 · 全息画像系统数据库设计

> 版本：v3.4 · 2026-07-15
> 基于课题三数据交付需求 v2.0
> 覆盖：PostgreSQL · Elasticsearch · Milvus · Neo4j

---

## 一、设计总览

### 1.1 核心设计决策

| 决策 | 说明 |
|------|------|
| raw_records 必须第一个写 | 任何步骤失败都可从原始记录重跑，不需要重消费 Kafka |
| 每步 T 的完整输出存 JSONB | t1/t2/t3/t4_output 四个字段，单行即可追溯全部处理过程 |
| 实体后台定时融合 | T2 写入时允许同名实体重复存在，EntityDeduplicationJob 后台批量归一 |
| 平台原始 ID 先存，UUID 后回填 | account_relations 等表先存字符串，独立回填任务补齐 UUID，入库不被 FK 阻塞 |
| PG 精简实体表 + Neo4j 完整属性 | PG 实体表只存统计/索引/画像关联需要的 6-8 个字段，详细属性存 Neo4j |
| 画像定时任务批量生成 | 不做差量补全，T5 一次性生成全量 15 维度画像，旧版本自动归档 |
| 账号快照追加不覆盖 | social_account_snapshots 表保留历史快照，支持时序分析 |
| Neo4j 区分事实关系和推断关系 | FOLLOWS 等来自课题三（置信度 1.0），COORDINATES_WITH 等是 T6 推断 |

### 1.2 数据分层

| 层次 | 存什么 | 存在哪里 |
|------|--------|----------|
| L0 原始层 | Kafka 消息原始体，含 T1-T4 完整输出 | PG `raw_records` |
| L1 标准化层 | 从原始消息映射出的业务对象 | PG 各业务表 |
| L2 实体层（精简） | T2 抽取的实体，只存核心字段 | PG 精简实体表 |
| L3 关系层 | 实体间关系图谱 | Neo4j |
| L4 画像层 | T5 生成的全息画像，15+ 维度 | PG `person_profiles` |
| 索引层 | 全文 / 向量 / 图谱检索 | ES + Milvus + Neo4j |

### 1.3 课题三数据类型与写表对应

| record_type | Kafka Topic | 写入表 | 是否走 T1-T4 |
|---|---|---|---|
| `social_content` | kt3.social_content | raw_records + media_contents | ✅ 完整 T1→T2→T3→T4 |
| `news_article` | kt3.news_article | raw_records + media_contents | ✅ 完整 T1→T2→T3→T4 |
| `social_account` | kt3.social_account | raw_records + social_accounts + social_account_snapshots | ❌ NORMALIZED 即完成 |
| `account_relation` | kt3.account_relation | raw_records + account_relations | ❌ NORMALIZED 即完成，T3 批处理回填 UUID |
| `media_asset` | kt3.media_asset | raw_records + media_assets | ❌ NORMALIZED 即完成，T4 异步补图像向量 |
| `collection_task` | kt3.collection_task | raw_records + collection_tasks | ❌ NORMALIZED 即完成 |

### 1.4 pipeline_status 状态机

```
RECEIVED → NORMALIZED → T1_DONE → T2_DONE → T3_DONE → T4_INDEXED   ← social_content / news_article 终态
RECEIVED → NORMALIZED                                                  ← 其余四种类型终态
任意步骤失败 → FAILED（重试超 3 次后）
```

#### `raw_records` — 原始数据记录

**存什么：** 每条从 Kafka 或批量文件进来的原始消息，必须第一个写，一字不改全量保存。T1/T2/T3/T4 的完整输出也存在这张表的 JSONB 字段里，实现单行可追溯。

**为什么这样设计：** 原始记录落库后，任何一步 T 失败都可以从这里重跑，不需要重消费 Kafka。想看一条数据的完整处理过程，一条 SELECT 即可。

```sql
SELECT t1_output, t2_output, t3_output, t4_output
FROM raw_records WHERE source_record_id = 'x_post_123456';
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PK | 主键 |
| schema_version | VARCHAR(32) | 课题三数据格式版本，如实存下发过来的值。**不做强校验**——早期版本卡死过一次真实数据导入（课题三从v1升到v2后被硬编码校验拒收），现在只要信封结构和必填字段对得上就正常处理，不限制具体版本号字符串 |
| record_type | VARCHAR(32) NOT NULL | `social_content` \| `social_account` \| `account_relation` \| `media_asset` \| `news_article` \| `collection_task` |
| source_record_id | VARCHAR(256) NOT NULL | 课题三全局唯一原始记录 ID，去重主键 |
| crawl_task_id | VARCHAR(256) | 关联的课题三采集任务 ID，用于追溯数据来源批次 |
| collected_at | TIMESTAMPTZ | 课题三采集时间，区别于内容发布时间 |
| source_url | TEXT | 原始内容/账号/新闻的 URL |
| payload_hash | VARCHAR(64) NOT NULL | SHA256(raw_payload)，内容级去重 |
| source_type | VARCHAR(32) | `kafka` \| `file_upload` |
| source_topic | VARCHAR(128) | Kafka topic 名称，如 `kt3.social_content` |
| platform | VARCHAR(64) | `x` \| `youtube` \| `telegram` \| `tiktok` \| `news` 等 |
| language | VARCHAR(16) | `zh` \| `en` \| `ja` \| `ko` \| `vi` \| `tl` \| `ar` \| `fa` 等 |
| content_type | VARCHAR(32) | 内容形态，仅 social_content / news_article 有值 |
| raw_payload | JSONB NOT NULL | 完整原始消息体，一字不改 |
| raw_published_at | TIMESTAMPTZ | 内容在平台的原始发布时间 |
| normalized_at | TIMESTAMPTZ | 完成 L1 标准化映射的时间 |
| pipeline_status | VARCHAR(32) | 状态机，默认 `RECEIVED` |
| pipeline_task_id | UUID | 关联 pipeline_tasks.id，Step 4 创建任务后回填 |
| error_message | TEXT | 流水线失败时的错误详情 |
| retry_count | SMALLINT | 当前已重试次数，超过 max_retries(3) 后改为 FAILED |
| t1_output | JSONB | T1 `annotate_content` 接口的完整响应体，用于处理过程追溯 |
| t2_output | JSONB | T2 `extract_entities` 接口的完整响应体，含实体/关系抽取结果（mention级，不含叙事——T2不再抽取narrative类型） |
| t3_output | JSONB | T3 `resolve_batch` 接口的完整响应体，含每个mention的消歧判断（MERGE/REVIEW/CREATE），不是图谱节点/关系本身；只有当条内容存在至少一个有候选可比对的mention时才会调用T3、才有值，否则为 null |
| t4_output | JSONB | T4 索引构建结果，含 Milvus 文本向量ID、是否跳过文本向量化等 |
| batch_import_id | UUID | 批量文件导入任务 ID，非 Kafka 来源时填 |
| created_at / updated_at | TIMESTAMPTZ | 创建/更新时间 |

**索引：**
- `UNIQUE (source_record_id)` — 记录级去重
- `UNIQUE (payload_hash)` — 内容级去重
- `(record_type)` `(pipeline_status)` `(platform)` `(created_at DESC)`
- `(crawl_task_id) WHERE NOT NULL`

#### `social_accounts` — 账号/频道/群组（最新状态）

**存什么：** `social_account` 类型记录标准化后的账号主体最新状态。历史快照存 `social_account_snapshots`。按 `(platform, platform_user_id)` UPSERT。

**账号身份识别（独立于内容流水线）：** 账号摄入时同步调用 T1 `annotate_account_type` 得到 `account_type`；之后由定时任务 `SocialAccountIdentityJob` 按账号类别把账号本身也解析成图谱实体——个人类账号（`ordinary_user`/`political_actor`/`academic_or_expert`/`influencer_kol`）匹配/新建 `Person`，机构类账号（`news_media`/`government_agency`等10类）匹配/新建 `Organization`，写 `HAS_ACCOUNT` 关系并回填 `entity_person_id`/`entity_org_id`；匿名/疑似机器人/未知类账号跳过不处理。这条路径复用跟 T2 内容抽取一样的"候选召回+T3判断"逻辑。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PK | 与 Neo4j SocialAccount 节点 ID 保持一致 |
| platform | VARCHAR(64) NOT NULL | |
| platform_user_id | VARCHAR(256) NOT NULL | 平台原始用户/频道/群组 ID，联合唯一 |
| account_entity_type | VARCHAR(32) | v2.0 新增：`user` \| `channel` \| `page` \| `group` \| `community` \| `forum_board` \| `news_source` |
| platform_native_type | VARCHAR(64) | v2.0 新增：`youtube_channel` \| `telegram_group` \| `facebook_page` 等 |
| handle | VARCHAR(256) | @用户名 |
| display_name | VARCHAR(512) | |
| bio | TEXT | 个人简介/频道描述 |
| avatar_url / profile_url | TEXT | |
| self_declared_location | TEXT | 用户自填位置，不做解析 |
| verified | BOOLEAN | 是否已认证 |
| verified_type | VARCHAR(32) | `none` \| `blue` \| `org` \| `government` \| `media` |
| is_suspended | BOOLEAN | 是否已被平台封禁 |
| account_created_at | TIMESTAMPTZ | 账号在平台的注册时间 |
| followers_count / following_count / subscriber_count / member_count / post_count / view_count | BIGINT | 各项统计（最新快照值） |
| account_type | VARCHAR(32) | T1 `annotate_account_type` 接口判断的账号主类别，18类枚举：`ordinary_user`/`news_media`/`state_affiliated_media`/`government_agency`/`political_actor`/`political_party_or_campaign`/`military_security_agency`/`international_organization`/`ngo_or_civil_society`/`academic_or_expert`/`commercial_brand`/`platform_official`/`influencer_kol`/`community_group`/`anonymous_account`/`suspected_bot_or_automated`/`unknown`/`other` |
| account_type_confidence | NUMERIC(4,3) | |
| entity_person_id | UUID | → persons.id，SocialAccountIdentityJob 身份识别后回填（个人类账号） |
| entity_org_id | UUID | → organizations.id，SocialAccountIdentityJob 身份识别后回填（机构类账号） |
| identity_resolved_at | TIMESTAMPTZ | 身份识别（匹配/新建Person或Organization、或判定跳过）完成时间，NULL表示还未处理 |
| latest_snapshot_at | TIMESTAMPTZ | 最新快照的采集时间 |
| first_seen_at / last_active_at | TIMESTAMPTZ | |
| created_at / updated_at | TIMESTAMPTZ | |

**索引：** `UNIQUE(platform, platform_user_id)` `(entity_person_id) WHERE NOT NULL` `(account_entity_type)` `(handle)`

## 三、Elasticsearch 索引设计（4 个）

> ES 不是主存储，数据由 PG 同步（CDC 或定期同步），只用于全文检索。

### `media_contents_index` — 内容全文检索

> 显式定义了 mapping 的字段（`MediaContentEsService.ensureIndex()` 里逐个声明，中文用 IK 分词器）：`body_text`/`title`/`summary` 是分词全文字段，`platform`/`language`/`hashtags`/`topic_category`/`topic_type`/`sentiment_label`/`stance_label`/`aigc_type`/`ideology_label`/`language_style_labels`/`manipulation_methods`/`risk_label`/`risk_types` 是 `keyword` 精确匹配字段，`published_at` 是 `date`。其余字段走 ES 动态映射。下面列的是 `T4IndexingStep` 实际往 ES 文档里塞的完整字段集合。
>
> **这份 mapping 只在索引不存在、首次创建时生效**——已经存在的索引不会自动应用新增的显式字段声明，那些字段会先落到动态映射里，等索引下次重建才会变成真正的 `keyword`/`text` 类型。

```json
{
  "id":                      "keyword",
  "raw_record_id":           "keyword",
  "platform":                "keyword",
  "content_type":            "keyword",
  "platform_content_id":     "keyword",
  "author_platform_user_id": "keyword",
  "title":                   "text (ik_max_word)",
  "body_text":               "text (ik_max_word)",
  "summary":                 "text (ik_max_word)",
  "language":                "keyword",
  "published_at":            "date",
  "url":                     "keyword",
  "hashtags":                "keyword",
  "mentions":                "keyword",
  "external_urls":           "keyword",
  "like_count / comment_count / share_count / repost_count / quote_count / view_count / reaction_count": "long",
  "topic_category":          "keyword",
  "topic_type":              "keyword",
  "sentiment_label":         "keyword",
  "stance_label":            "keyword",
  "aigc_score":               "float",
  "aigc_type":                "keyword",
  "ideology_label":           "keyword",
  "language_style_labels":    "keyword",
  "manipulation_methods":     "keyword",
  "risk_label":               "keyword",
  "risk_types":               "keyword",
  "entities_hint":            "text",
  "updated_at":               "date"
}
```

**`topic_category`/`topic_type`/`sentiment_label`/`stance_label`/`aigc_score`/`aigc_type`/`summary`/`ideology_label`/`language_style_labels`/`manipulation_methods`/`risk_label`/`risk_types`/`entities_hint` 这几个字段不是从 PG 专门列取的**（`media_contents` 表本身没有这些列），是索引时用 `T1AnnotationView`（`common.util` 包下的共享工具类）实时解析 `media_contents.t1_annotation` 这个 JSONB 字段抽出来的，`T4IndexingStep`/`SearchContentTool`/`ContentPropagationBackfillJob` 都复用这同一个解析工具类，保证取值口径一致。`entities_hint` 之前叫 `narrative_hint`，是历史遗留命名（存的内容一直是实体线索，不是叙事），已改名对齐实际含义；旧的 `topic_subcategory` 字段读取的是 v0.6 已废弃的 `topicTags.subtopicTags` 路径，从未取到过值，已删除不再写入。

### `entities_index` — 图节点全文检索

> `EntityEsService.indexEntity()` 统一服务图节点关键词检索。`Person` / `Organization` / `Event` / `Location`、`SocialAccount`、`MediaContent` 都写入这个索引；标准实体写名称/别名，账号写 display_name/handle/bio 等可检索文本，内容节点写 title/platformContentId/url 等节点级文本。正文全文检索仍由 `media_contents_index` 负责。

```json
{
  "mappings": {
    "properties": {
      "entity_id":        { "type": "keyword" },
      "canonical_name":   { "type": "text", "analyzer": "ik_max_word",
        "fields": { "keyword": { "type": "keyword" } } },
      "normalized_name":  { "type": "text", "analyzer": "ik_max_word" },
      "aliases":          { "type": "text", "analyzer": "ik_max_word" },
      "entity_type":      { "type": "keyword" },
      "source_id":        { "type": "keyword" },
      "platform":         { "type": "keyword" },
      "account_entity_type": { "type": "keyword" },
      "account_type":     { "type": "keyword" },
      "content_type":     { "type": "keyword" },
      "importance_score": { "type": "float" }
    }
  }
}
```

**`normalized_name` 不是 `canonical_name` 的原样复制**——`EntityEsService.indexEntity()` 内部会对 `canonicalName` 做真正的归一化处理（NFD分解去重音符号、转小写、标点替换成空格、合并多余空白），比如 `"U.S. Central Command"` 会归一化成 `"u s central command"`，用来兼容候选召回时"同一个名字，大小写/标点/重音符号写法不同"这种情况。两个字段类型声明看起来一样（都是 `ik_max_word` 分词的 `text`），是因为两者都需要支持全文检索，但实际存的内容不同。

### `social_accounts_index` — 已废弃

> 2026-07-16 起，账号关键词检索统一写入并查询 `entities_index`（`entity_type=SocialAccount`），不再创建或使用 `social_accounts_index`。历史环境中的旧索引可通过一键清理脚本删除。

## 四、Milvus 向量索引设计（3 个 Collection）

> `MilvusVectorService` 用 Milvus V2 SDK 建 Collection，并显式声明标量字段和多个向量字段。向量维度固定为 4096，距离度量统一使用 COSINE。写入统一使用 `upsert`，同一业务对象重复写入会更新已有记录，不产生重复行。
>
> 2026-07-16 起，Milvus 从“一个对象一个拼接文本向量”调整为“一个对象一条记录、多个字段向量”。历史 collection `text_embeddings`、`image_embeddings`、`account_embeddings` 已废弃；重导数据前需要清理旧 collection 和旧版 `entity_embeddings`。

### `media_content_embeddings` — 内容文本向量

> 用于贴文/新闻等内容检索。`T4IndexingStep` 分别生成标题向量和正文语义向量；同一条内容只写一条 Milvus 记录。

| 字段 | 说明 |
|---|---|
| id | 主键，`"media_content_" + contentId` |
| content_id | 对应 `media_contents.id` |
| platform | 平台 |
| language | 语言 |
| content_type | 内容类型 |
| published_at | Unix 时间戳 |
| title_embedding | 标题向量 |
| text_embedding | 正文/T1摘要/标签/结构化标注等文本向量 |

### `media_asset_embeddings` — 媒体资源向量

> 用于以图搜图、文搜图和 OCR 文本语义召回。图片视觉向量与 OCR 文本向量写在同一条媒体资源记录上。

| 字段 | 说明 |
|---|---|
| id | 主键，`"media_asset_" + assetId` |
| asset_id | 对应 `media_assets.id` |
| source_asset_id | 采集侧资源 ID |
| content_id | 所属 `media_contents.id` |
| platform | 平台 |
| asset_type | 媒体类型 |
| mime_type | MIME 类型 |
| image_embedding | 图片视觉向量 |
| ocr_text_embedding | OCR 文本向量 |

### `entity_embeddings` — 图节点语义向量

> 用于实体/账号/内容节点的全域图节点候选召回。标准实体、`SocialAccount`、`MediaContent` 都写入这一张 collection；不再单独维护 `account_embeddings`。

| 字段 | 说明 |
|---|---|
| id | 主键，`"entity_" + entity_type + "_" + entity_id` |
| entity_id | Neo4j/业务节点 ID。标准实体、账号、内容分别对应各自节点 ID |
| entity_type | 节点类型：`Person`、`Organization`、`Event`、`Location`、`SocialAccount`、`MediaContent` |
| canonical_name | 展示/排查用标准名 |
| aliases | 别名文本，多个别名用分隔符拼接 |
| source_id | 外部来源 ID，例如账号平台用户 ID、内容平台 ID |
| platform | 平台 |
| name_embedding | 名称向量 |
| alias_embedding | 别名向量 |
| description_embedding | 描述/属性/正文语义向量 |

### `entity_hybrid_poc_embeddings` — Milvus 单栈实体检索 POC

> 用于验证实体候选召回是否可以由 Milvus 单独承担。该 collection 和 `entity_embeddings` 写入同一批图节点，但额外保存 `keyword_text` 文本字段和 `keyword_sparse` BM25 稀疏向量字段；查询时使用 Milvus `hybridSearch` 同时检索 BM25 sparse 与 dense vector 字段，并由 Milvus 原生 RRF Ranker 输出融合候选。当前仅在 `/api/search/entities/candidates` 传 `retrieval_mode=hybrid` 时使用，不替代默认 ES+Milvus 方案。

| 字段 | 说明 |
|---|---|
| id | 主键，和 `entity_embeddings.id` 一致 |
| entity_id | Neo4j/业务节点 ID |
| entity_type | 节点类型 |
| canonical_name | 展示/排查用标准名 |
| aliases | 别名文本 |
| source_id | 外部来源 ID |
| platform | 平台 |
| keyword_text | BM25 原始文本字段，启用 Milvus analyzer |
| keyword_sparse | 由 BM25 Function 从 `keyword_text` 生成的稀疏向量 |
| name_embedding | 名称向量 |
| alias_embedding | 别名向量 |
| description_embedding | 描述/属性/正文语义向量 |

## 五、Neo4j 图谱设计

> Neo4j 是实体详细属性和关系的主存储，PG 精简实体表只存索引和统计字段，两库通过 UUID 关联（PG 主键 = Neo4j 节点 `id` 属性）。**关系类型词表以 `docs/关系词表与头尾实体类型说明.md` 为唯一权威来源**，当前统一保留 **16 个关系类型**，如果本节内容与那份文档冲突，以那份文档为准。

### 5.1 节点类型

当前统一使用 **6 个基础实体类**：`Person`、`Organization`、`SocialAccount`、`MediaContent`、`Event`、`Location`。

**`Narrative`（叙事）不再作为原子实体**——T2 抽取的实体类型只有 `person`/`organization`/`event`/`location` 四种，不会再输出 `narrative` 类型；PG `narratives` 表和相关代码目前还保留着，但这条抽取路径事实上已经不再产生新数据，后续如果要清理需要专门评估。

**`MediaAsset`（图片/视频等媒体文件）不写入 Neo4j**——图片/视频文件本身只存 PG（`media_assets` 表）和 MinIO，Neo4j 不为每个媒体文件单独建节点；关联关系通过 `MediaContent` 节点自身的 `mediaAssetIds` 属性（PG `media_contents.media_asset_ids` 数组的同步副本）维护，不建独立的图谱边/节点。

```cypher
(:Person {
    id: "uuid",              -- 与 PG persons.id 相同
    canonicalName: "...",
    type: "person"
    -- 具体还会带哪些属性取决于 T2/T3 解析时 mention.attributes 里有什么，
    -- Neo4j 无强制schema，不是每个节点都保证有完整属性集
})

(:Organization {
    id: "uuid",
    canonicalName: "...",
    type: "organization"
})

(:Event {
    id: "uuid",
    canonicalName: "...",
    type: "event",
    eventType: "military | diplomatic | ...",   -- 来自T2抽取时的attributes
    eventTimeStart: "...",
    eventHeatLevel: "low | medium | high | explosive | unclear",  -- EventHeatJob写入
    eventHeatScore: 0.0
})

(:Location {
    id: "uuid",
    canonicalName: "...",
    type: "location"
})

(:SocialAccount {
    id: "uuid",              -- 与 PG social_accounts.id 相同
    platform: "...",
    platformUserId: "...",
    handle: "...",           -- 账号身份识别/关系回填时按实际拿到的字段写入，不保证每个节点都有
    displayName: "..."
})

(:MediaContent {
    id: "uuid",              -- 与 PG media_contents.id 相同
    title: "...",            -- 有标题的内容才有这个属性（评论/回复通常没有标题）
    platform: "...",
    contentType: "...",
    language: "...",
    platformContentId: "...",
    publishedAt: "...",
    url: "...",
    likeCount: 0, commentCount: 0, shareCount: 0, repostCount: 0, quoteCount: 0, viewCount: 0, reactionCount: 0,
    mediaAssetIds: ["uuid", "..."],  -- 关联的 media_assets.id 字符串数组，没有关联媒体资源时不带这个属性
    t1Annotation: "..."      -- T1 annotate_content 接口的完整响应体JSON字符串，整段存成一个属性，不拆字段
})
```

### 5.2 关系类型

当前保留 **16 个关系类型**，分 5 组（详细的中文释义、方向约定、使用边界见 `docs/关系词表与头尾实体类型说明.md`，这里只列头尾实体类型速查）：

| 分组 | 关系 | 头实体 | 尾实体 |
|---|---|---|---|
| 账号关联类 | `HAS_ACCOUNT` | Person、Organization | SocialAccount |
| 归属组织类 | `BELONGS_TO` | Person、Organization、SocialAccount、MediaContent | Organization、SocialAccount |
| 归属组织类 | `PART_OF` | Organization | Organization |
| 内容链路类 | `PUBLISHED_BY` | MediaContent | Organization、SocialAccount |
| 内容链路类 | `REPLY_TO` | MediaContent | MediaContent |
| 内容链路类 | `REPOSTS` | MediaContent | MediaContent |
| 内容链路类 | `MENTIONS` | MediaContent | Person、Organization、SocialAccount、Event、Location |
| 事件地点类 | `DESCRIBES` | MediaContent | Event |
| 事件地点类 | `EVENT_OCCURRED_AT` | Event | Location |
| 事件地点类 | `EVENT_INVOLVES_ENTITY` | Event | Person、Organization、SocialAccount |
| 事件地点类 | `LOCATED_IN` | Organization、SocialAccount | Location |
| 立场认知类 | `SUPPORTS` | Person、Organization、SocialAccount、MediaContent | Person、Organization、SocialAccount、MediaContent、Event |
| 立场认知类 | `OPPOSES` | 同上 | 同上 |
| 立场认知类 | `QUESTIONS` | 同上 | 同上 |
| 立场认知类 | `INCITES` | 同上 | 同上 |
| 立场认知类 | `DE_ESCALATES` | 同上 | 同上 |

**这16个是"T2/T3 抽取输出关系时能用的词表"**，`T2ExtractionStep.writeRelations()` 落图前会校验 `predicate` 必须在这个词表内，不在词表内的会被过滤掉不写入图谱。

**除此之外，还有几类关系是后端结构性写入的，不受这16个词表约束**（跟 T2/T3 算法抽取无关，后端确定性生成）：

```cypher
-- 内容与它成功解析出的实体之间的关系，T2处理每条内容后自动补写，不需要T2在relations[]里额外输出
(:MediaContent)-[:DESCRIBES]->(:Event)                                    -- event类型走DESCRIBES
(:MediaContent)-[:MENTIONS]->(:Person|:Organization|:Location)            -- 其余类型走MENTIONS

-- 账号身份识别（SocialAccountIdentityJob）
(:Person|:Organization)-[:HAS_ACCOUNT]->(:SocialAccount)

-- 内容传播链（T2ExtractionStep实时 + ContentPropagationBackfillJob兜底回填）
(:MediaContent)-[:REPLY_TO]->(:MediaContent)
(:MediaContent)-[:REPOSTS]->(:MediaContent)

-- 账号间关系（AccountRelationBackfillJob，来自课题三account_relation消息，置信度可信）
(:SocialAccount)-[:FOLLOWS { confidence: 1.0, observedAt: "..." }]->(:SocialAccount)   -- following，注意FOLLOWS不在16个词表内，是历史遗留的结构性关系，不要在算法抽取里再产出这个类型
(:SocialAccount)-[:BELONGS_TO { confidence: 1.0 }]->(:SocialAccount)                    -- subscribe/member_of/owner_of/creator_of/admin_of/moderator_of 统一收敛到这个
```

### 5.3 常用查询示例

```cypher
-- 查某人物的 2 跳关系图（前端知识图谱可视化）
MATCH (p:Person {id: $person_id})-[r1]-(n1)-[r2]-(n2)
RETURN p, r1, n1, r2, n2 LIMIT 50

-- 查某个人物关联的账号（身份识别产出）
MATCH (p:Person {id: $person_id})-[:HAS_ACCOUNT]->(a:SocialAccount)
RETURN a.id, a.handle, a.platform

-- 查某个事件被哪些内容描述过、涉及哪些实体
MATCH (c:MediaContent)-[:DESCRIBES]->(e:Event {id: $event_id})
OPTIONAL MATCH (e)-[:EVENT_INVOLVES_ENTITY]->(entity)
RETURN e, collect(DISTINCT c) AS relatedContents, collect(DISTINCT entity) AS involvedEntities

-- 查某条内容的传播链（父帖/转发链）
MATCH path = (c:MediaContent {id: $content_id})-[:REPLY_TO|REPOSTS*1..10]->(root:MediaContent)
RETURN path

-- 查两个账号之间是否存在关注/隶属关系路径
MATCH path = shortestPath(
    (a1:SocialAccount {id: $id1})-[:FOLLOWS|BELONGS_TO*..5]-(a2:SocialAccount {id: $id2})
)
RETURN path
```
