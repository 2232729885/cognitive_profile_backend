# 课题四 · 全息画像系统数据库设计

> 版本：v3.0 · 2026-06-25
> 基于课题三数据交付需求 v2.0
> 覆盖：PostgreSQL · Elasticsearch · Milvus · Neo4j

---

## 一、设计总览

### 1.1 核心设计决策

| 决策 | 说明 |
|------|------|
| raw_records 必须第一个写 | 任何步骤失败都可从原始记录重跑，不需要重消费 Kafka |
| 每步 T 的完整输出存 JSONB | t1/t2/t3/t4_output 四个字段，单行即可追溯全部处理过程 |
| T3 冲突直接覆盖 | 融合时遇到字段冲突，新值直接覆盖旧值，不推审核队列 |
| 平台原始 ID 先存，UUID 后回填 | account_relations 等表先存字符串，T3 归一后回填 UUID，入库不被 FK 阻塞 |
| PG 精简实体表 + Neo4j 完整属性 | PG 实体表只存统计/索引/画像关联需要的 6-8 个字段，详细属性存 Neo4j |
| 画像定时任务批量生成 | 不做差量补全，T5 一次性生成全量 15 维度画像，旧版本自动归档 |
| 账号快照追加不覆盖 | social_account_snapshots 表保留历史快照，支持时序分析 |
| Neo4j 区分事实关系和推断关系 | FOLLOWS 等来自课题三（置信度 1.0），COORDINATES_WITH 等是 T6 推断 |

### 1.2 数据分层

| 层次 | 存什么 | 存在哪里 |
|------|--------|----------|
| L0 原始层 | Kafka 消息原始体，含 T1-T4 完整输出 | PG `raw_records` |
| L1 标准化层 | 从原始消息映射出的业务对象 | PG 各业务表 |
| L2 实体层（精简） | T2 抽取的实体，只存核心字段 | PG 精简实体表 |
| L3 关系层 | 实体间关系图谱 | Neo4j |
| L4 画像层 | T5 生成的全息画像，15+ 维度 | PG `person_profiles` |
| 索引层 | 全文 / 向量 / 图谱检索 | ES + Milvus + Neo4j |

### 1.3 课题三数据类型与写表对应

| record_type | Kafka Topic | 写入表 | 是否走 T1-T4 |
|---|---|---|---|
| `social_content` | kt3.social_content | raw_records + media_contents | ✅ 完整 T1→T2→T3→T4 |
| `news_article` | kt3.news_article | raw_records + media_contents | ✅ 完整 T1→T2→T3→T4 |
| `social_account` | kt3.social_account | raw_records + social_accounts + social_account_snapshots | ❌ NORMALIZED 即完成 |
| `account_relation` | kt3.account_relation | raw_records + account_relations | ❌ NORMALIZED 即完成，T3 批处理回填 UUID |
| `media_asset` | kt3.media_asset | raw_records + media_assets | ❌ NORMALIZED 即完成，T4 异步补图像向量 |
| `collection_task` | kt3.collection_task | raw_records + collection_tasks | ❌ NORMALIZED 即完成 |

### 1.4 pipeline_status 状态机

```
RECEIVED → NORMALIZED → T1_DONE → T2_DONE → T3_DONE → T4_INDEXED   ← social_content / news_article 终态
RECEIVED → NORMALIZED                                                  ← 其余四种类型终态
任意步骤失败 → FAILED（重试超 3 次后）
```

### L0：原始采集层（1 张）

---

#### `raw_records` — 原始数据记录

**存什么：** 每条从 Kafka 或批量文件进来的原始消息，必须第一个写，一字不改全量保存。T1/T2/T3/T4 的完整输出也存在这张表的 JSONB 字段里，实现单行可追溯。

**为什么这样设计：** 原始记录落库后，任何一步 T 失败都可以从这里重跑，不需要重消费 Kafka。想看一条数据的完整处理过程，一条 SELECT 即可。

```sql
SELECT t1_output, t2_output, t3_output, t4_output
FROM raw_records WHERE source_record_id = 'x_post_123456';
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PK | 主键 |
| schema_version | VARCHAR(32) | 数据格式版本，默认 `kt3_to_kt4_v1`，接入适配器据此选择字段映射逻辑 |
| record_type | VARCHAR(32) NOT NULL | `social_content` \| `social_account` \| `account_relation` \| `media_asset` \| `news_article` \| `collection_task` |
| source_record_id | VARCHAR(256) NOT NULL | 课题三全局唯一原始记录 ID，去重主键 |
| crawl_task_id | VARCHAR(256) | 关联的课题三采集任务 ID，用于追溯数据来源批次 |
| collected_at | TIMESTAMPTZ | 课题三采集时间，区别于内容发布时间 |
| source_url | TEXT | 原始内容/账号/新闻的 URL |
| payload_hash | VARCHAR(64) NOT NULL | SHA256(raw_payload)，内容级去重 |
| source_type | VARCHAR(32) | `kafka` \| `file_upload` |
| source_topic | VARCHAR(128) | Kafka topic 名称，如 `kt3.social_content` |
| platform | VARCHAR(64) | `x` \| `youtube` \| `telegram` \| `tiktok` \| `news` 等 |
| language | VARCHAR(16) | `zh` \| `en` \| `ja` \| `ko` \| `vi` \| `tl` \| `ar` \| `fa` 等 |
| content_type | VARCHAR(32) | 内容形态，仅 social_content / news_article 有值 |
| raw_payload | JSONB NOT NULL | 完整原始消息体，一字不改 |
| raw_published_at | TIMESTAMPTZ | 内容在平台的原始发布时间 |
| normalized_at | TIMESTAMPTZ | 完成 L1 标准化映射的时间 |
| pipeline_status | VARCHAR(32) | 状态机，默认 `RECEIVED` |
| pipeline_task_id | UUID | 关联 pipeline_tasks.id，Step 4 创建任务后回填 |
| error_message | TEXT | 流水线失败时的错误详情 |
| retry_count | SMALLINT | 当前已重试次数，超过 max_retries(3) 后改为 FAILED |
| t1_output | JSONB | T1 自动标注 Agent 的完整响应体，用于处理过程追溯 |
| t2_output | JSONB | T2 信息抽取 Agent 的完整响应体，含实体/关系/叙事抽取结果 |
| t3_output | JSONB | T3 信息融合 Agent 的完整响应体，含实体归一 ID 映射表 |
| t4_output | JSONB | T4 索引构建结果，含 Milvus 向量 ID、ES 文档 ID 等 |
| batch_import_id | UUID | 批量文件导入任务 ID，非 Kafka 来源时填 |
| created_at / updated_at | TIMESTAMPTZ | 创建/更新时间 |

**索引：**
- `UNIQUE (source_record_id)` — 记录级去重
- `UNIQUE (payload_hash)` — 内容级去重
- `(record_type)` `(pipeline_status)` `(platform)` `(created_at DESC)`
- `(crawl_task_id) WHERE NOT NULL`

-- ───── 账号-账号事实关系（来自课题三 account_relation，置信度 1.0）─────
(:SocialAccount)-[:FOLLOWS {

### 5.3 常用查询示例

```cypher
-- 查某人物的 2 跳关系图（前端知识图谱可视化）
MATCH (p:Person {id: $person_id})-[r1]-(n1)-[r2]-(n2)
WHERE type(r1) IN ['AFFILIATED_WITH','COOPERATES_WITH','PROMOTES_NARRATIVE']
RETURN p, r1, n1, r2, n2 LIMIT 50

-- 查叙事涉及的所有账号（T6 识别入口）
MATCH (a:SocialAccount)-[:PARTICIPATES_IN_NARRATIVE]->(n:Narrative {id: $narrative_id})
RETURN a.id, a.handle, a.platform, a.follower_count
ORDER BY a.follower_count DESC LIMIT 2000

-- 发现潜在协同操控（账号间最短路径）
MATCH path = shortestPath(
    (a1:SocialAccount {id: $id1})-[:COORDINATES_WITH*..5]-(a2:SocialAccount {id: $id2})
)
RETURN path

-- 查某人物关联的叙事（按传播频次排序）
MATCH (p:Person {id: $person_id})-[r:PROMOTES_NARRATIVE]->(n:Narrative)
RETURN n.id, n.canonical_label, r.frequency, r.confidence
ORDER BY r.frequency * r.confidence DESC LIMIT 10
```
