# 实体候选召回接口规约

面向算法组调用，用于根据用户 query 返回候选实体节点 ID。接口只做召回，不做融合重排；后端分别返回关键词召回候选和语义召回候选，算法侧可基于两路候选自行融合、重排、过滤。

## 1. 接口信息

```text
POST /api/search/entities/candidates
Content-Type: application/json
```

返回体统一使用：

```json
{
  "code": "200",
  "msg": "success",
  "data": {}
}
```

## 2. 请求体

```json
{
  "query": "美伊冲突",
  "entity_types": ["Event", "Person", "Organization", "SocialAccount"],
  "top_k": 10,
  "retrieval_mode": "both",
  "trace_id": "trace-001",
  "context": {
    "slot_id": "object_1",
    "slot_name": "object",
    "user_query": "美伊战争中的重点个体和群体"
  }
}
```

字段说明：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---:|---:|---:|---|
| `query` | string | 是 | - | 用户查询文本，用于关键词召回和向量语义召回 |
| `entity_types` | string[] | 否 | 空数组 | 限定召回类型；为空或不传表示全域召回 |
| `top_k` | number | 否 | `10` | 每一路最多返回的候选数量；小于等于 0 时按 10 处理 |
| `retrieval_mode` | string | 否 | `both` | 召回模式：`keyword`、`semantic`、`both` |
| `trace_id` | string | 否 | - | 调用链追踪 ID，后端原样放到响应 `data.trace_id` |
| `context` | object | 否 | - | 算法侧上下文，后端当前不参与检索计算，仅透传接收，便于后续扩展 |

## 3. entity_types 取值

当前支持：

```text
Person
Organization
Event
Location
SocialAccount
MediaContent
```

说明：

- `Person` / `Organization` / `Event` / `Location` 使用标准实体索引：ES `entities_index` 和 Milvus `entity_embeddings`。
- `SocialAccount` 使用账号索引：ES `social_accounts_index` 和 Milvus `account_embeddings`。
- `MediaContent` 使用内容索引：ES 内容索引、Milvus `text_embeddings`，语义召回时也会查询图片语义关联到内容的结果。
- `entity_types` 为空或不传时，默认召回以上全部类型。
- 传入不支持的类型不会报错，但不会产生对应类型候选。

## 4. retrieval_mode 取值

| 值 | 含义 |
|---|---|
| `keyword` | 只返回关键词候选，即 `data.keyword_candidates` 有值，`data.semantic_candidates` 为空数组 |
| `semantic` | 只返回语义候选，即 `data.semantic_candidates` 有值，`data.keyword_candidates` 为空数组 |
| `both` | 同时返回关键词候选和语义候选 |

传入其他值时返回：

```json
{
  "code": "400",
  "msg": "retrieval_mode只支持keyword、semantic、both",
  "data": {
    "trace_id": null,
    "keyword_candidates": [],
    "semantic_candidates": []
  }
}
```

## 5. 响应体

成功示例：

```json
{
  "code": "200",
  "msg": "success",
  "data": {
    "trace_id": "trace-001",
    "keyword_candidates": [
      {
        "entity_id": "event-001",
        "name": "美伊冲突",
        "entity_type": "Event",
        "score": 12.37
      }
    ],
    "semantic_candidates": [
      {
        "entity_id": "event-009",
        "name": "伊朗与美国军事对峙",
        "entity_type": "Event",
        "score": 0.82
      }
    ]
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---:|---|
| `code` | string | `"200"` 表示成功；`"400"` 表示请求参数错误 |
| `msg` | string | 响应消息 |
| `data.trace_id` | string | 请求传入的 `trace_id`，原样返回 |
| `data.keyword_candidates` | Candidate[] | 关键词召回候选列表 |
| `data.semantic_candidates` | Candidate[] | 语义召回候选列表 |

Candidate 字段：

| 字段 | 类型 | 说明 |
|---|---:|---|
| `entity_id` | string | 候选节点 ID。后续算法输出或回传应使用该字段，不使用 `node_id` |
| `name` | string | 候选展示名，按 canonicalName/displayName/handle/title 等字段兜底生成 |
| `entity_type` | string | 候选类型，如 `Event`、`Person`、`SocialAccount`、`MediaContent` |
| `score` | number | 当前召回通道的原始分数 |

## 6. 分数说明

`keyword_candidates[].score` 和 `semantic_candidates[].score` 不是同一量纲，不能直接比较数值大小。

- 关键词候选分数来自 ES 相关性得分，适合同一路关键词召回内部排序参考。
- 语义候选分数来自 Milvus 向量相似度，适合同一路语义召回内部排序参考。
- 接口不会把两路结果做融合，也不会输出最终综合分。融合重排由算法侧完成。

## 7. 错误响应

`query` 为空：

```json
{
  "code": "400",
  "msg": "query不能为空",
  "data": {
    "trace_id": null,
    "keyword_candidates": [],
    "semantic_candidates": []
  }
}
```

`retrieval_mode` 非法：

```json
{
  "code": "400",
  "msg": "retrieval_mode只支持keyword、semantic、both",
  "data": {
    "trace_id": null,
    "keyword_candidates": [],
    "semantic_candidates": []
  }
}
```

## 8. 对接建议

1. 算法侧优先使用 `entity_id` 作为候选唯一标识。
2. 两路候选可能有重复实体，算法侧融合时需要按 `entity_id` 去重。
3. `keyword_candidates` 和 `semantic_candidates` 的分数只在各自列表内有排序意义。
4. 如果只需要某一路召回，建议明确传 `retrieval_mode=keyword` 或 `retrieval_mode=semantic`，减少无用计算。
5. 如果需要全域实体候选，不传 `entity_types` 或传空数组即可。

## 9. curl 示例

```bash
curl -X POST 'http://localhost:8080/api/search/entities/candidates' \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "美伊冲突",
    "entity_types": ["Event", "Person", "Organization", "SocialAccount"],
    "top_k": 10,
    "retrieval_mode": "both",
    "trace_id": "trace-001",
    "context": {
      "slot_id": "object_1",
      "slot_name": "object",
      "user_query": "美伊战争中的重点个体和群体"
    }
  }'
```
