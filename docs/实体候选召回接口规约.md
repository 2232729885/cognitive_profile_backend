# 实体候选召回接口规约

面向算法组调用，用于根据用户 query 返回候选实体节点 ID。默认模式下接口只做召回，不做融合重排；后端分别返回关键词召回候选和语义召回候选，算法侧可基于两路候选自行融合、重排、过滤。

2026-07-16 起新增 `retrieval_mode=hybrid` POC：后端使用 Milvus 原生 BM25 sparse + dense vector + RRF 做实体候选融合，结果放在 `data.hybrid_candidates`。这个模式用于验证“实体检索是否可以从 ES+Milvus 迁到 Milvus 单栈”，暂不替代默认 `both` 模式。

## 1. 接口信息

```text
POST /api/search/entities/candidates
Content-Type: application/json
```

返回体统一使用：

```json
{
  "code": "200",
  "msg": "success",
  "data": {}
}
```

## 2. 请求体

```json
{
  "query": "美伊冲突",
  "entity_types": ["Event", "Person", "Organization", "Location", "SocialAccount", "MediaContent"],
  "top_k": 10,
  "retrieval_mode": "both",
  "semantic_min_score": 0.7,
  "trace_id": "trace-001",
  "context": {
    "slot_id": "object_1",
    "slot_name": "object",
    "user_query": "美伊战争中的重点个体和群体"
  }
}
```

字段说明：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---:|---:|---:|---|
| `query` | string | 是 | - | 用户查询文本，用于关键词召回和向量语义召回 |
| `entity_types` | string[] | 否 | 空数组 | 限定召回类型；为空或不传表示全域召回 |
| `top_k` | number | 否 | `10` | 每一路最多返回的候选数量；小于等于 0 时按 10 处理 |
| `retrieval_mode` | string | 否 | `both` | 召回模式：`keyword`、`semantic`、`both`、`hybrid` |
| `semantic_min_score` | number | 否 | - | 仅 `retrieval_mode=hybrid` 使用；不传则不按语义分过滤，传入时过滤 `semantic_score` 低于该值的候选；有效范围 `[0,1]`，越界值会被后端截断 |
| `trace_id` | string | 否 | - | 调用链追踪 ID，后端原样放到响应 `data.trace_id` |
| `context` | object | 否 | - | 算法侧上下文，后端当前不参与检索计算，仅透传接收，便于后续扩展 |

## 3. entity_types 取值

当前支持：

```text
Person
Organization
Event
Location
SocialAccount
MediaContent
```

说明：

- `Person` / `Organization` / `Event` / `Location` 默认关键词召回使用 ES `entities_index`，语义召回使用 Milvus `entity_embeddings`。
- `SocialAccount` 默认关键词召回同样使用 ES `entities_index`（`entity_type=SocialAccount`），语义召回使用 Milvus `entity_embeddings`（`entity_type=SocialAccount`）。
- `MediaContent` 默认关键词召回使用 ES `entities_index`（`entity_type=MediaContent`），语义召回同样使用 Milvus `entity_embeddings`（`entity_type=MediaContent`）。这里召回的是图节点候选，不走图片/OCR 专用检索链路。
- `retrieval_mode=hybrid` 时，以上类型统一使用 Milvus POC collection `entity_hybrid_poc_embeddings`，由 Milvus 原生 RRF 输出融合后的候选。
- `entity_types` 为空或不传时，默认召回以上全部类型。
- 传入不支持的类型不会报错，但不会产生对应类型候选。

## 4. retrieval_mode 取值

| 值 | 含义 |
|---|---|
| `keyword` | 只返回关键词候选，即 `data.keyword_candidates` 有值，`data.semantic_candidates` 为空数组 |
| `semantic` | 只返回语义候选，即 `data.semantic_candidates` 有值，`data.keyword_candidates` 为空数组 |
| `both` | 同时返回关键词候选和语义候选 |
| `hybrid` | POC模式：只返回 Milvus 原生融合候选，即 `data.hybrid_candidates` 有值，`keyword_candidates` / `semantic_candidates` 为空数组 |

传入其他值时返回：

```json
{
  "code": "400",
  "msg": "retrieval_mode只支持keyword、semantic、both、hybrid",
  "data": {
    "trace_id": null,
    "keyword_candidates": [],
    "semantic_candidates": [],
    "hybrid_candidates": []
  }
}
```

## 5. 响应体

成功示例：

```json
{
  "code": "200",
  "msg": "success",
  "data": {
    "trace_id": "trace-001",
    "keyword_candidates": [
      {
        "entity_id": "event-001",
        "name": "美伊冲突",
        "entity_type": "Event",
        "score": 12.37
      }
    ],
    "semantic_candidates": [
      {
        "entity_id": "event-009",
        "name": "伊朗与美国军事对峙",
        "entity_type": "Event",
        "score": 0.82
      }
    ],
    "hybrid_candidates": []
  }
}
```

`retrieval_mode=hybrid` 成功示例：

```json
{
  "code": "200",
  "msg": "success",
  "data": {
    "trace_id": "trace-001",
    "keyword_candidates": [],
    "semantic_candidates": [],
    "hybrid_candidates": [
      {
        "entity_id": "e2367b4c-8ebd-4692-a846-450655e0a2b7",
        "name": "The Map Guy",
        "entity_type": "SocialAccount",
        "score": 0.049180325120687485,
        "keyword_score": 13.405767440795898,
        "semantic_score": 1.0,
        "semantic_field": "name_embedding",
        "fusion_score": 0.049180325120687485,
        "matched_channels": ["keyword", "semantic"],
        "metadata": null
      }
    ]
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---:|---|
| `code` | string | `"200"` 表示成功；`"400"` 表示请求参数错误 |
| `msg` | string | 响应消息 |
| `data.trace_id` | string | 请求传入的 `trace_id`，原样返回 |
| `data.keyword_candidates` | Candidate[] | 关键词召回候选列表 |
| `data.semantic_candidates` | Candidate[] | 语义召回候选列表 |
| `data.hybrid_candidates` | Candidate[] | Milvus 原生 hybrid + RRF 融合候选列表，仅 `retrieval_mode=hybrid` 时有值 |

Candidate 字段：

| 字段 | 类型 | 说明 |
|---|---:|---|
| `entity_id` | string | 候选节点 ID。后续算法输出或回传应使用该字段，不使用 `node_id` |
| `name` | string | 候选展示名，按 canonicalName/displayName/handle/title 等字段兜底生成 |
| `entity_type` | string | 候选类型，如 `Event`、`Person`、`SocialAccount`、`MediaContent` |
| `score` | number | 兼容字段。`keyword` 模式为 ES 得分，`semantic` 模式为 Milvus 相似度，`hybrid` 模式为 RRF 融合分 |
| `keyword_score` | number/null | hybrid 模式下的关键词 BM25 分数；未被关键词通道召回时为 `null` |
| `semantic_score` | number/null | hybrid 模式下的 Milvus 向量相似度分数；未被语义通道召回时为 `null` |
| `semantic_field` | string/null | hybrid 模式下贡献 `semantic_score` 的向量字段，如 `name_embedding`、`alias_embedding`、`description_embedding` |
| `fusion_score` | number/null | hybrid 模式下 Milvus RRF 融合后的排序分数 |
| `matched_channels` | string[] | hybrid 模式下命中的通道列表，可能为 `["keyword"]`、`["semantic"]` 或 `["keyword","semantic"]` |
| `metadata` | object/null | 预留扩展字段，当前通常为 `null` |

## 6. 分数说明

`keyword_candidates[].score`、`semantic_candidates[].score`、`hybrid_candidates[].score` 不是同一量纲，不能直接比较数值大小。

- 关键词候选分数来自 ES 相关性得分，适合同一路关键词召回内部排序参考。
- 语义候选分数来自 Milvus 向量相似度，适合同一路语义召回内部排序参考。
- `hybrid_candidates[].score` 等同于 `hybrid_candidates[].fusion_score`，来自 Milvus 原生 RRF 融合排序，适合在该列表内部解释排序，不适合跟 ES/Milvus 单路分数比较。
- `hybrid_candidates[].keyword_score` 和 `hybrid_candidates[].semantic_score` 只是解释该候选分别在关键词/语义通道中的命中情况，二者不是同一量纲，也不要直接相加。
- `hybrid_candidates[].semantic_field` 用于说明语义分来自哪个向量字段，便于判断是名称、别名还是描述命中。
- `keyword` / `semantic` / `both` 模式不会把两路结果做融合，也不会输出最终综合分。融合重排由算法侧完成。

## 7. 错误响应

`query` 为空：

```json
{
  "code": "400",
  "msg": "query不能为空",
  "data": {
    "trace_id": null,
    "keyword_candidates": [],
    "semantic_candidates": [],
    "hybrid_candidates": []
  }
}
```

`retrieval_mode` 非法：

```json
{
  "code": "400",
  "msg": "retrieval_mode只支持keyword、semantic、both、hybrid",
  "data": {
    "trace_id": null,
    "keyword_candidates": [],
    "semantic_candidates": [],
    "hybrid_candidates": []
  }
}
```

## 8. 对接建议

1. 算法侧优先使用 `entity_id` 作为候选唯一标识。
2. 两路候选可能有重复实体，算法侧融合时需要按 `entity_id` 去重。
3. `keyword_candidates` 和 `semantic_candidates` 的分数只在各自列表内有排序意义。
4. 如果只需要某一路召回，建议明确传 `retrieval_mode=keyword` 或 `retrieval_mode=semantic`，减少无用计算。
5. 如果需要全域实体候选，不传 `entity_types` 或传空数组即可。
6. 若算法侧希望自己融合重排，继续使用默认 `both`；若希望评估 Milvus 单栈召回效果，使用 `hybrid`，只消费 `hybrid_candidates`。

```bash
  -H 'Content-Type: application/json' \
  -d '{
    "query": "美伊冲突",
    "entity_types": ["Event", "Person", "Organization", "SocialAccount"],
    "top_k": 10,
    "retrieval_mode": "both",
    "semantic_min_score": 0.7,
    "trace_id": "trace-001",
    "context": {
      "slot_id": "object_1",
      "slot_name": "object",
      "user_query": "美伊战争中的重点个体和群体"
    }
  }'
```
