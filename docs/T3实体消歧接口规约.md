# T3 实体消歧判断接口规约
# 课题四 × T3 算法组接口对接文档

> 本文档专门给 T3 算法组用。T2 怎么抽取实体、后端怎么做候选召回——这些不是 T3 要关心的，T3 拿到的输入已经是"一个 mention + 一批候选实体"，只需要判断这个 mention 应该合并到候选里的哪一个、还是新建、还是转人工审核。如果想了解完整链路，参考 `docs/Kg融合接口规约 v1.md`。

---

## 一、T3 在整个流水线里的位置

```
T2 抽取mention → 后端候选召回（ES+Milvus） → T3 判断 → 后端按判断结果合并/新建/转人工审核
```

T3 只做**判断**，不读写任何数据库、不接触 Neo4j/PG/Milvus/ES，候选实体是后端已经查询好传进来的，T3 要做的是：给定一个 mention（这次抽取到的实体）和它的候选列表（图谱里可能已经存在的同一个实体），判断这个 mention 到底对应候选里的哪一个（或者都不对应，需要新建）。

**这个接口同时服务两种场景，输入结构完全一样，T3 不需要区分调用方是谁**：
- T2 抽取内容之后的实体消歧（`context.contentId` 装的是内容ID）
- 账号身份识别（判断一个社交账号的 `displayName` 对应哪个 Person/Organization，`context.contentId` 装的是账号ID）

---

## 二、接口

### POST（action=`resolve_batch`）

一次请求可以带多个 mention（`items` 数组），每个 mention 独立判断，互不影响。

#### 统一响应包裹

T3 接口的 HTTP 响应体统一使用 `code/msg/data` 包裹，和 T1/T2 保持一致：

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "...": "原业务响应体"
  }
}
```

约定：

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | number | `0` 表示业务成功；非 `0` 表示业务失败 |
| `msg` | string | 成功时建议固定为 `success`；失败时返回简短错误原因 |
| `data` | object\|null | 成功时为原来的 T3 消歧判断结果；失败时可以为 `null` |

HTTP 4xx/5xx 仍表示协议级或服务级失败。`code != 0` 表示 HTTP 请求成功到达算法服务，但业务处理失败，例如输入不合法、模型不可用、schema 不支持等。

#### 请求体

```json
{
  "items": [
    {
      "mention": {
        "mentionId": "m1",
        "name": "美国中央司令部",
        "canonicalName": "U.S. Central Command",
        "type": "organization",
        "span": {"start": 0, "end": 8},
        "aliases": ["CENTCOM", "美国中央指挥部"],
        "attributes": {}
      },
      "candidates": [
        {
          "entityId": "3f8a2b1c-...",
          "canonicalName": "U.S. Central Command",
          "type": "organization",
          "aliases": ["CENTCOM", "美国中央司令部"],
          "importanceScore": 92.0,
          "attributes": {},
          "score": 0.94,
          "retrievalChannels": ["NAME_INDEX", "VECTOR_INDEX"]
        },
        {
          "entityId": "7c4d9e2f-...",
          "canonicalName": "United States Army",
          "type": "organization",
          "aliases": ["美国陆军"],
          "importanceScore": 85.0,
          "attributes": {},
          "score": 0.61,
          "retrievalChannels": ["VECTOR_INDEX"]
        }
      ],
      "context": {
        "contentId": "内容ID（账号身份识别场景下装的是账号ID）",
        "platform": "x",
        "textWindow": "美国中央司令部在霍尔木兹海峡附近展开军事演习",
        "language": "zh"
      }
    }
  ],
  "strategy": {
    "autoMergeThreshold": 0.9,
    "reviewThreshold": 0.6
  }
}
```

**`items[].mention` 字段说明：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `mentionId` | string | 对应 T2 输出的 `mentionId`，或者账号身份识别场景下后端自己生成的临时ID，原样回传即可 |
| `name` | string | 原文出现的名称 |
| `canonicalName` | string | 标准名，判断相似度时以这个字段为准 |
| `type` | string | `person`/`organization`/`event`/`location` |
| `span` | object\|null | 字符位置 `{start, end}`，账号身份识别场景下这个字段通常是 `null`（账号资料不是从某段文本里抽取的） |
| `aliases` | array | 别名 |
| `attributes` | object | 扩展属性 |

**`items[].candidates` 字段说明**（每个候选实体是图谱里已经存在的一个实体）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `entityId` | string | 图谱里的正式实体ID（UUID） |
| `canonicalName` | string | 该候选实体在图谱里的标准名 |
| `aliases` | array | 该候选实体已知的别名 |
| `importanceScore` | number | 该候选实体当前的重要度评分 |
| `score` | number | **后端**召回时算出的初步相似度分数（ES/Milvus 检索分数），仅供参考，T3 应该结合 `mention`/`candidates` 的实际语义相似度重新判断，不能只看这个分数 |
| `retrievalChannels` | array | 这个候选是通过什么渠道召回的：`NAME_INDEX`（ES名称匹配）/`VECTOR_INDEX`（向量召回） |

**`items[].context`**：`textWindow` 是这个 mention 出现的上下文文本片段，账号身份识别场景下是账号的 `bio`；`contentId` 是内容ID或者账号ID，只用于日志追踪，不影响判断逻辑。

**`strategy`**：`autoMergeThreshold`/`reviewThreshold` 是后端这次请求指定的阈值，**T3 的判断逻辑要按这次请求传的阈值来，不要自己写死一套不同的阈值**（这两个值目前后端默认是 0.9/0.6，但以后可能调整，每次请求都会带上当次生效的值）。

#### 响应体

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "results": [
      {
        "mentionId": "m1",
        "action": "MERGE",
        "matchedEntityId": "3f8a2b1c-...",
        "score": 0.96,
        "confidence": 0.96,
        "matchMethod": "exact_name_alias",
        "reason": "canonicalName与候选实体canonicalName完全一致，aliases高度重叠"
      }
    ],
    "modelVersion": "t3-resolve-v2.0"
  }
}
```

以下字段均位于成功响应的 `data` 内。

**`results[]` 字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `mentionId` | string | 是 | 对应请求里的 `mention.mentionId`，原样回传 |
| `action` | string | 是 | `MERGE`/`REVIEW`/`CREATE` 三选一，见下方枚举说明 |
| `matchedEntityId` | string | `action=MERGE`或`REVIEW`时必填 | 判断匹配到的候选实体ID（对应 `candidates[].entityId`）；`action=CREATE`时不填或为`null` |
| `score` | number | 是 | 0-1，T3 自己判断出的匹配分数（不是候选召回时后端给的那个初步分数，是 T3 重新判断之后的最终分数） |
| `confidence` | number | 是 | 0-1，本次判断的置信度 |
| `matchMethod` | string | 否 | 简要说明用什么方法判断的，比如 `exact_name_alias`（名称/别名完全匹配）、`semantic_similarity`（语义相似度）、`context_disambiguation`（结合上下文消歧） |
| `reason` | string | 否 | 简要判断理由，供人工审核时参考 |

**action 枚举说明：**

| action | 触发条件 | 后端处理 |
|---|---|---|
| `MERGE` | `score` ≥ `autoMergeThreshold`（请求里传的，默认0.9） | 直接合并进 `matchedEntityId` 对应的已有实体节点 |
| `REVIEW` | `reviewThreshold` ≤ `score` < `autoMergeThreshold`（默认0.6~0.9之间） | 写入待审核队列，人工确认后再合并 |
| `CREATE` | 无候选，或者 `score` < `reviewThreshold`（默认0.6） | 新建一个实体，不合并到任何候选 |

**`modelVersion`**：位于 `data.modelVersion`，标识这次调用用的是哪个版本的模型，会被记录进 `entity_fusion_records.resolver_model`，方便后续追溯是哪个模型版本做的判断。

## 三、判断建议

1. **`candidates` 为空数组时，直接返回 `action=CREATE`**，不要因为候选是空的就报错或者不返回结果。
2. **不要只看后端给的初步 `score`**，那只是 ES/Milvus 召回时的粗筛分数（比如纯字符串相似度、向量距离），T3 应该结合 `mention` 和 `candidates` 的完整信息（别名、类型、上下文）重新计算一个更准确的匹配分数。
3. **一个 mention 可能有多个候选**，T3 应该在所有候选里选出最匹配的那个（如果有的话）来判断 MERGE/REVIEW，而不是只看第一个候选。
4. **`type` 不同的候选不应该匹配**——请求里传进来的候选理论上后端已经按 `mention.type` 过滤过了，但如果发现候选里有类型不匹配的（不应该出现，出现了说明后端有 bug），不要勉强匹配，按无候选处理（`action=CREATE`）。

## 四、不再使用的旧接口/概念（提醒，避免误实现）

- ~~`/kg/entity/resolve/batch`~~ → 现在统一叫 `resolve_batch`（action名）。
- ~~`/kg/offline-resolution/jobs`~~（离线全局融合任务）→ 已移除，不需要实现，T3 只有这一个接口。
- ~~T3 只服务内容抽取场景~~ → 现在账号身份识别也走这个接口，输入结构完全一样，不用为账号场景单独实现。
