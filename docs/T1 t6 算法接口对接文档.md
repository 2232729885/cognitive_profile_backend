# 课题四 T1-T6 算法接口对接文档 v2.0

> 版本：v2.0 · 2026-07-03
> 相较 v1.0 的主要变更：
> - T3 已从实时流水线移除，后端直接处理 T2 输出写 Neo4j，T3 改为后台融合任务占位
> - T2 输出新增 `relationships` 和 `events` 字段
> - T1 输出结构与 RZDK 对齐（嵌套 `annotations` 对象）
> - T6 输入结构与 RZDK 对齐（传完整账号和内容列表）
> - 明确各处字段名差异，标注后端接入真实算法时需要适配的地方

---

## 0. 当前架构说明

### 流水线（已简化）

```
Kafka消息
  → T1（文本标注）
  → T2（实体+关系抽取）→ 后端直接写 PG + Neo4j
  → T4（向量化）→ 写 Milvus + ES
```

**T3 Agent 已从实时流水线中移除。** 跨批次实体归一由 `EntityDeduplicationJob` 后台定时任务处理（每小时），Neo4j 节点使用 `stableUuid(type:canonicalName)` 保证幂等。

### Neo4j 节点 id 体系

| 节点类型 | id 生成方式 |
|---|---|
| Person/Organization/Event/Narrative/Location | `UUID.nameUUIDFromBytes("type:canonicalName".UTF8)` |
| SocialAccount | `social_accounts.id`（PG 真实 UUID）|
| MediaContent | `media_contents.id`（PG 真实 UUID）|

### 调用约定（全部 Agent 共用）

- 请求方式：`POST`，`Content-Type: application/json`
- URL：`{base_url}/{action}`
- 后端等待超时：30 秒（可在 `sub_agent_registry.timeout_seconds` 配置）
- 切换 mock/real：`PUT /api/system/agents/{code}/url-type`

## 1. T1 自动标注

**职责**：对文本进行多维度自动标注，输出供 T2 使用的结构化提示。  
**action**：`annotate_text`  
**触发时机**：每条 `social_content` / `news_article` 入库后流水线第一步。

### 请求体

```json
{
  "text": "外部势力被指干预选举进程，多名分析人士对此发表质疑。",
  "language": "zh",
  "annotation_types": [
    "topics", "keywords", "summary", "language_style",
    "sentiment", "event_type", "content_purpose",
    "aigc_suspicion", "entities_hint"
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| text | String | 是 | 待标注文本 |
| language | String | 否 | 语言代码，默认 zh |
| annotation_types | String[] | 否 | 标注类型，默认全部 9 项 |

> ⚠️ **后端适配说明**：后端 `T1AnnotateRequest` 目前字段名为 `bodyText`（而非 `text`），接入真实 T1 时需修改为 `text`。同时需补充 `annotation_types` 字段传递。

## 4. T4 向量化

**职责**：将文本或图像转换为向量，写入 Milvus。后端负责写入，T4 只负责生成向量。  
**使用模型**：`Qwen3-VL-Embedding-8B`（`http://172.16.40.232:18013`）

> ⚠️ **定位差异**：RZDK 的 T4 定位为检索服务（`search_entities`/`find_related`），后端定位为向量化服务。当前按后端定位实现，如需对齐 RZDK 定位请提前告知。

### action 一：文本向量化

**action**：`generate_text_embedding`

```json
// 请求
{"text": "外部势力被指干预选举进程。", "imageUrl": null}

// 响应
{
  "embedding": [0.123, -0.456, ...],  // 必须 4096 维
  "modelVersion": "qwen3-vl-embedding-8b-v1.0"
}
```

### action 二：图像向量化

**action**：`generate_image_embedding`

```json
// 请求
{"text": null, "imageUrl": "http://172.16.40.232:9000/media-assets/test/image1.jpg"}

// 响应（结构同文本向量化）
{"embedding": [...], "modelVersion": "qwen3-vl-embedding-8b-v1.0"}
```

**关键约束**：`embedding` 必须是 **4096 维**，Milvus Collection 的 `dim` 固定为 4096，维度不匹配会写入失败。

**数据库连接信息（T5 自行查询）**：
- PostgreSQL：`172.16.40.232:5432`，`postgres/postgres`，数据库 `postgres`


## 7. 后端切换 Mock → 真实模型

无需重启后端，无需改代码，三步完成：

```bash
# 1. 配置真实地址
curl -X PUT http://localhost:8080/api/system/agents/T1/base-url \
  -H "Authorization: Bearer <admin-token>" \
  -H "Content-Type: application/json" \
  -d '{"baseUrl": "http://algo-server:18001"}'

# 2. 切换为真实模式
curl -X PUT http://localhost:8080/api/system/agents/T1/url-type \
  -H "Authorization: Bearer <admin-token>" \
  -H "Content-Type: application/json" \
  -d '{"activeUrlType": "real"}'

# 3. 健康检查
curl -X POST http://localhost:8080/api/system/agents/T1/health-check \
  -H "Authorization: Bearer <admin-token>"

# 回滚（如需）
curl -X PUT http://localhost:8080/api/system/agents/T1/url-type \
  -H "Authorization: Bearer <admin-token>" \
  -H "Content-Type: application/json" \
  -d '{"activeUrlType": "mock"}'
```

## 3.1 T3 `resolve_entities` 接口规约（补充）

**用途**：后端 `EntityDeduplicationJob` 在精确名称匹配完成后，调用此接口做跨语言模糊归一。T3 只负责判断哪些实体应该合并，不执行写库操作（后端执行）。

**action**：`resolve_entities`  
**完整路径**：`POST {base_url}/resolve_entities`

### 请求体

```json
{
  "entities": [
    {
      "id": "8f54f8d6-21d8-4945-9cd8-ca8daffcfa83",
      "entityType": "person",
      "canonicalName": "Leila Farzan",
      "aliases": ["L. Farzan", "莱拉·法尔赞"],
      "importanceScore": 88.0,
      "sourceIdentifiers": {
        "x": "rw_presstv_b56ecf2b",
        "telegram": "12345678"
      }
    },
    {
      "id": "cb14f38d-48b2-4826-9159-f8f355c84ef4",
      "entityType": "person",
      "canonicalName": "ليلى فرزان",
      "aliases": [],
      "importanceScore": 76.0,
      "sourceIdentifiers": {}
    }
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| entities | Array | 是 | 待消歧的实体候选列表，同一批次应为同一类型（person/organization/event/narrative）|
| entities[].id | String | 是 | PG 精简实体表的 UUID |
| entities[].entityType | String | 是 | person \| organization \| event \| narrative |
| entities[].canonicalName | String | 是 | 标准化名称 |
| entities[].aliases | String[] | 否 | 别名列表，供跨语言匹配参考 |
| entities[].importanceScore | Number | 否 | 重要性评分 0-100，用于确定 survivor（保留分数最高的）|
| entities[].sourceIdentifiers | Object | 否 | 平台原生标识符，key=平台名，value=platform_user_id。person 类型会填充，其他类型为空 |

### 响应体（选项B：直接输出聚合好的分组）

```json
{
  "mergeGroups": [
    {
      "survivorId": "8f54f8d6-21d8-4945-9cd8-ca8daffcfa83",
      "mergedIds": ["cb14f38d-48b2-4826-9159-f8f355c84ef4"],
      "confidence": 0.92,
      "matchMethod": "cross_language"
    }
  ],
  "disjointPairs": [],
  "uncertain": []
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| mergeGroups | Array | 应该合并的实体分组，每组第一个是 survivor |
| mergeGroups[].survivorId | String | 归一后保留的实体 ID（建议选 importanceScore 最高的）|
| mergeGroups[].mergedIds | String[] | 被合并的实体 ID 列表（后端会置为 deduplicated）|
| mergeGroups[].confidence | Number | 归一置信度 0-1，**后端只执行 confidence ≥ 0.8 的归一建议**，低于阈值的跳过 |
| mergeGroups[].matchMethod | String | 归一依据（见下表）|
| disjointPairs | Array | 明确不是同一实体的 ID 对（否定证据，后端暂不处理，预留字段）|
| uncertain | String[] | 置信度低或证据不足、无法判断的实体 ID 列表 |

**matchMethod 枚举值：**

| 值 | 含义 |
|---|---|
| `exact_name` | 精确名称匹配（后端精确匹配已覆盖，T3 通常不需要返回这类）|
| `alias_match` | 别名匹配（如 "Biden" 匹配到 aliases 里有 "Biden" 的实体）|
| `cross_language` | 跨语言归一（如 "拜登" 和 "Biden"，"霍尔木兹" 和 "Hormuz"）|
| `semantic_similarity` | 语义相似度归一（向量余弦，用于叙事/事件等长文本实体）|
| `identifier_match` | 平台标识符匹配（sourceIdentifiers 里 platform_user_id 相同，置信度最高）|

**重要约束：**
- 后端只执行 `confidence ≥ 0.8` 的归一建议，低于阈值视为不确定，不执行合并
- `survivorId` 必须是请求体 `entities` 数组里存在的某个 id，不能返回请求里没有的 id
- 不同类型的实体不应该出现在同一个 mergeGroup 里（T3 按类型分批调用）
