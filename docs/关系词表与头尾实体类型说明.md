# 关系词表与头尾实体类型说明

本文档用于同步 RZDK 实体关系设计的最新统一口径，面向所有后续模块工程师。

本版本已经吸收与甲方讨论后的定论。若本文与当前历史文档、样例数据、前端展示稿或现有代码枚举不一致，一律以本文为准，后续代码再按本文同步。

## 最新定论总览

当前统一保留 16 个关系类型，共分为 5 组：

1. 账号关联类：1 个
2. 归属组织类：2 个
3. 内容链路类：4 个
4. 事件地点类：4 个
5. 立场认知类：5 个

当前保留关系如下：

1. 账号关联类：`HAS_ACCOUNT`
2. 归属组织类：`BELONGS_TO`、`PART_OF`
3. 内容链路类：`PUBLISHED_BY`、`REPLY_TO`、`REPOSTS`、`MENTIONS`
4. 事件地点类：`DESCRIBES`、`EVENT_OCCURRED_AT`、`EVENT_INVOLVES_ENTITY`、`LOCATED_IN`
5. 立场认知类：`SUPPORTS`、`OPPOSES`、`QUESTIONS`、`INCITES`、`DE_ESCALATES`

## 已删除或转属性的历史关系

以下历史关系不再进入统一关系词表，后续模块不要再新增、抽取、展示或落库使用：

- 删除 `SAME_AS`
- 删除 `MERGED_INTO`
- 删除 `AFFILIATED_WITH`
- 删除 `OWNS`
- 删除 `MEMBER_OF`
- 删除 `ADMIN_OF`
- 删除 `CONTROLS`
- 删除 `AUTHORED`
- 删除 `REPORTS`
- 删除 `POSTS_FROM`
- 删除 `COMMENT_ON`
- 删除 `QUOTES`
- 删除 `SHARES`
- 删除 `CONTENT_EXPRESSES_NARRATIVE`
- 删除 `NARRATIVE_TARGETS_ENTITY`
- 删除 `NARRATIVE_ABOUT_EVENT`
- 删除 `HAS_EMOTION`
- 整体删除传播协同类全部关系：`AMPLIFIES`、`BRIDGES_COMMUNITY`、`COORDINATES_WITH`、`POTENTIAL_SUBORDINATE_TO`、`INFLUENCES`
- 整体删除证据治理类全部关系：`ASSERTED_BY`、`DERIVED_FROM`、`CONFLICTS_WITH`、`REVIEWED_BY`

补充说明：

- `ALIAS` 不再作为关系存在，而是作为 `Person`、`Organization`、`Location` 的属性存在
- 证据来源、派生链路、冲突、复核等治理语义，后续应通过证据表、血缘表、版本表、审计字段等机制表达，不再通过关系词表表达
- 传播协同、桥接、影响力等分析型结果，后续如需保留，应作为分析标签、特征或专题输出，不再作为基础关系类型

## 当前 6 个基础实体类

本文档中的实体类型统一使用当前主口径的 6 个基础实体类：

- `Person`
- `Organization`
- `SocialAccount`
- `MediaContent`
- `Event`
- `Location`

补充说明：

- 当前 `Narrative` 不作为原子实体
- `aliases` 作为属性存在于 `Person`、`Organization`、`Location`
- 关系抽取与下游建模应尽量围绕上述 6 个基础实体类收敛

---

## 1. 账号关联类

### `HAS_ACCOUNT`

- 中文关系名：拥有账号 / 关联账号
- 解释：表示现实主体拥有、使用或稳定关联某个线上账号节点
- 推荐头实体类型：`Person`、`Organization`
- 推荐尾实体类型：`SocialAccount`

---

## 2. 归属组织类

### `BELONGS_TO`

- 中文关系名：隶属 / 归属 / 属于
- 解释：统一表示稳定归属关系，覆盖历史上的任职、成员、挂靠、拥有、账号归口、内容归属等语义
- 方向约定：统一使用“被归属对象 -> 所属主体 / 上级主体 / 归口主体”的方向
- 推荐头实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`
- 推荐尾实体类型：`Organization`、`SocialAccount`

使用示例：

- 人物属于某机构：`Person -> Organization`
- 组织属于上级组织：`Organization -> Organization`
- 账号归属于某人物或机构时，如需明确主体账号关系，优先使用 `HAS_ACCOUNT`；如强调账号归口，也可使用 `SocialAccount -> Organization`
- 内容归属于某发布主体时，优先使用 `PUBLISHED_BY`，不要与 `BELONGS_TO` 重复造边

### `PART_OF`

- 中文关系名：组成部分 / 上下级组成关系
- 解释：表示一个组织是另一个组织的组成部分、内部结构单元或下级单位
- 推荐头实体类型：`Organization`
- 推荐尾实体类型：`Organization`

补充说明：

- `PART_OF` 用于强调组织结构组成关系
- `BELONGS_TO` 用于表达更一般化的归属关系
- 若仅需保留一种最宽泛的归属语义，可优先使用 `BELONGS_TO`

---

## 3. 内容链路类

### `PUBLISHED_BY`

- 中文关系名：由…发布 / 由…发表
- 解释：统一表示内容与其发布者之间的关系，覆盖历史上的 `PUBLISHED_IN`、`AUTHORED`、`REPORTS`、`POSTS_FROM` 中“内容和来源”的表达语义，更适用于“账号发表贴文”或“媒体发表文章”这两类主场景
- 推荐头实体类型：`MediaContent`
- 推荐尾实体类型：`Organization`、`SocialAccount`

### `REPLY_TO`

- 中文关系名：回复
- 解释：表示一条内容对另一条内容形成回复关系
- 推荐头实体类型：`MediaContent`
- 推荐尾实体类型：`MediaContent`

### `REPOSTS`

- 中文关系名：转发
- 解释：表示一条内容对另一条内容进行再次传播
- 推荐头实体类型：`MediaContent`
- 推荐尾实体类型：`MediaContent`

### `MENTIONS`

- 中文关系名：提及
- 解释：表示一条内容中提及某个实体
- 推荐头实体类型：`MediaContent`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`、`Event`、`Location`

---

## 4. 事件地点类

### `DESCRIBES`

- 中文关系名：描述
- 解释：表示某条内容描述某个事件
- 推荐头实体类型：`MediaContent`
- 推荐尾实体类型：`Event`

### `EVENT_OCCURRED_AT`

- 中文关系名：事件发生于
- 解释：表示某个事件发生在某个地点
- 推荐头实体类型：`Event`
- 推荐尾实体类型：`Location`

### `EVENT_INVOLVES_ENTITY`

- 中文关系名：事件涉及实体
- 解释：表示某个事件涉及某个人物、组织或账号作为参与方、对象或受影响方
- 推荐头实体类型：`Event`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`

### `LOCATED_IN`

- 中文关系名：位于
- 解释：表示某个组织、账号或其他节点位于某个地点或法域范围内
- 推荐头实体类型：`Organization`、`SocialAccount`
- 推荐尾实体类型：`Location`

---

## 5. 立场认知类

### `SUPPORTS`

- 中文关系名：支持
- 解释：表示主体 A 对主体 B 或事件 B 表达正向立场、认可、声援、背书、帮助或维护
- 推荐头实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`、`Event`

### `OPPOSES`

- 中文关系名：反对
- 解释：表示主体 A 对主体 B 或事件 B 表达负向立场、批评、质疑、否定、谴责、抵制或驳斥
- 推荐头实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`、`Event`

### `QUESTIONS`

- 中文关系名：质疑
- 解释：表示主体 A 对主体 B、事件 B 或相关说法表达怀疑、追问、挑战、求证、削弱可信度或提出不确定性判断
- 推荐头实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`、`Event`

### `INCITES`

- 中文关系名：煽动
- 解释：表示主体 A 对事件 B、群体 B 或传播过程 B 进行动员、鼓动、扩散、放大、激化或推动传播
- 推荐头实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`、`Event`

### `DE_ESCALATES`

- 中文关系名：平息
- 解释：表示主体 A 对冲突 B、争议 B、恐慌 B 或升级风险 B 进行降温、澄清、调停、缓和或阻断扩散
- 推荐头实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`
- 推荐尾实体类型：`Person`、`Organization`、`SocialAccount`、`MediaContent`、`Event`

---

## 开发注意事项

1. 后续新增、抽取、展示、入库时，只能使用本文保留的 17 个关系类型。

2. 已删除关系即使仍然出现在历史代码枚举、样例 JSON、前端图示或旧测试中，也应视为历史遗留，不得继续沿用。

3. `aliases` 是实体属性，不是关系：
   - `Person.aliases`
   - `Organization.aliases`
   - `Location.aliases`

4. 组织归属、成员关系、拥有关系等统一向 `BELONGS_TO` 收敛；不要再并行使用 `AFFILIATED_WITH`、`OWNS`、`MEMBER_OF`、`ADMIN_OF`、`CONTROLS`。

5. 内容来源统一向 `PUBLISHED_BY` 收敛；不要再并行使用 `AUTHORED`、`REPORTS`、`POSTS_FROM` 表达“内容和来源”的关系。

6. 叙事、情绪、传播协同、证据治理等语义，不再作为基础关系词表的一部分；如业务需要，应通过属性、标签、特征、专题分析结果或独立治理模型承载。

7. 对“任务书要求不少于 6 种核心关系”的当前解释口径为：
   - `SUPPORTS`
   - `OPPOSES`
   - `QUESTIONS`
   - `INCITES`
   - `DE_ESCALATES`
   - `EVENT_INVOLVES_ENTITY`
     其中 `EVENT_INVOLVES_ENTITY` 在当前验收口径中承担“影响”关系的表达角色，但其建模名称仍保持事件参与关系，不单独恢复 `INFLUENCES`。
